# Question

The project schedule shows 20% contingency buffer at the end. A stakeholder asks why the project can't just deliver 20% earlier without the buffer. How should the PM explain?

## Options

- **A**: Remove the buffer to please the stakeholder
- **B**: Explain that schedule buffer protects against normal variation - not all tasks will complete exactly as estimated. Without buffer, any delay becomes a project delay. The buffer provides realistic delivery confidence.
- **C**: Agree that buffers are unnecessary
- **D**: Add more buffer to be safe

## Answer

B

## Explanation

The correct answer is B. Schedule buffers aren't slack - they're protection against normal uncertainty. Explain: estimates have uncertainty; some tasks will take longer than planned. Buffer absorbs variation without impacting the delivery date. Without buffer, the first delay becomes a project delay. Buffer provides realistic confidence in the date.

Option A removes protection without understanding.

Option C is factually incorrect.

Option D doesn't address the stakeholder's question.
