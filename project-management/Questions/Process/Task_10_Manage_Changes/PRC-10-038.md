# Question

A PM inherits a mid-flight project where prior changes were poorly documented and nobody is sure which version of the requirements is current. What should the PM do FIRST to re-establish change control?

## Options

- **A**: Reconstruct the change history from records and interviews, agree the current state with stakeholders, and re-baseline from there.
- **B**: Discard the prior history, document the new requirements as discovered, and operate without a baseline going forward.
- **C**: Halt project work until full historical documentation has been recreated and reviewed.
- **D**: Continue execution and let the change log capture only changes from this point forward without re-establishing a baseline.

## Answer

A

## Explanation

The PM cannot control changes against an unknown current state. Reconstructing what is known, agreeing the current state with stakeholders, and re-baselining establishes the reference point that change control needs. (B) abandons useful history and leaves the project ungoverned. (C) halts the project unnecessarily; reconstruction can run in parallel. (D) leaves the baseline undefined, which is the underlying problem.
