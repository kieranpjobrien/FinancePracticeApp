# Question

When should an issue be closed?

## Options

- **A**: When the team is tired of working on it
- **B**: When the underlying problem has been resolved and verified, or when a decision has been made to accept the impact
- **C**: After 30 days regardless of status
- **D**: Only at the end of the project

## Answer

B

## Explanation

The correct answer is B. Issues should be closed when actually resolved (the problem no longer exists and this has been verified) or when stakeholders have consciously decided to accept the impact (the issue won't be fixed but this is a documented decision). Closure should reflect reality, not arbitrary rules.

Option A isn't a valid closure criterion.

Option C uses arbitrary time limits rather than resolution status.

Option D doesn't reflect that issues should be resolved throughout the project.
