# Question

A team using Scrum has defined their Definition of Done to include "all tests pass" and "code reviewed." A story is marked complete but the reviewer approved it without running the code locally. A bug is found in production. What should the PM address?

## Options

- **A**: Update the Definition of Done to explicitly require reviewers to run code locally
- **B**: Add a separate QA review step after code review
- **C**: Discuss with the team whether the Definition of Done was truly followed in spirit
- **D**: Implement automated deployment gates that prevent merging without certain checks

## Answer

C

## Explanation

The correct answer is C. The Definition of Done was technically followed (code was reviewed) but arguably not in spirit if the review wasn't thorough. Before adding process, the PM should facilitate a discussion about what "code reviewed" really means and whether team members share that understanding.

Option A adds specificity but may not address the real issue if the reviewer would have approved regardless. Understanding the gap comes first.

Option B adds process overhead. The existing process should work if followed properly.

Option D automates checks but doesn't address the human judgment component of reviews. Automation complements but doesn't replace thoughtful review.
