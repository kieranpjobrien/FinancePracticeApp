# Question

Your team practices continuous integration with a comprehensive automated test suite. A developer's change passes every automated test, but during a casual conversation another developer points out an unhandled edge case in the change that could corrupt customer data. Which quality practice should be strengthened?

## Options

- **A**: Make peer code review a mandatory step in the Definition of Done so a second pair of eyes inspects every change for issues automated tests cannot catch.
- **B**: Raise the minimum automated code-coverage threshold from its current level to 95% so more code paths are exercised by tests.
- **C**: Add a manual smoke-test phase before each deployment to production to catch issues that automated tests miss.
- **D**: Require developers to write a documented edge-case analysis for every story before they begin coding it.

## Answer

A

## Explanation

The defect was caught by another human, not by tests, and the lesson is that human review catches design and logic issues automation does not. (A) institutionalises that practice. (B) increases coverage but coverage measures lines exercised, not whether tests assert the right behaviour. (C) inserts late inspection rather than upstream prevention. (D) front-loads paperwork and assumes developers can predict every edge case alone — exactly the assumption that just failed.
