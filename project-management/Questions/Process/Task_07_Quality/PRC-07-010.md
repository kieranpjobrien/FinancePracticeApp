# Question

A software project has a "Definition of Done" that includes code review, unit tests, and integration tests. Under pressure to meet a sprint deadline, a developer asks to skip code review for one story to save time. What should the Scrum Master do?

## Options

- **A**: Allow the exception since the deadline is important
- **B**: Refuse and explain that the Definition of Done is non-negotiable for quality assurance
- **C**: Escalate to the product owner to decide
- **D**: Allow it but document the exception

## Answer

B

## Explanation

The correct answer is B. The Definition of Done exists to ensure quality and prevent technical debt. Skipping quality gates under pressure leads to accumulated problems - bugs found later, harder to trace, more expensive to fix. The Scrum Master should protect the team's quality standards and help find other ways to meet the deadline.

Option A starts a pattern of compromising quality under pressure. Each exception makes the next easier to justify.

Option C is inappropriate - the DoD is a team commitment, not a PO decision. Quality standards shouldn't be traded for features.

Option D formalises quality shortcuts. Documentation doesn't prevent the problems that code review would have caught.
