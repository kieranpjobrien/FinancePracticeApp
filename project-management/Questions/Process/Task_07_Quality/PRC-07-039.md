# Question

A data analytics platform launches on schedule. Users confirm the data is accurate, but report queries take 40 seconds when they expected sub-second response. The build team points to the requirements document, which only specified accuracy. What is the best way for the PM to characterise this situation?

## Options

- **A**: The team met the contract; performance is a new requirement and should follow change control.
- **B**: Users should adjust expectations to the documented quality criteria.
- **C**: This is a quality planning gap — performance is a quality attribute that should have been elicited and made explicit.
- **D**: Add a performance test suite to future releases so similar surprises do not recur.

## Answer

C

## Explanation

Quality is fitness for purpose, which spans accuracy, performance, usability, security, and more. If performance was never elicited, the failure is in the Plan Quality / requirements work, not in the team's execution. (A) is technically defensible but treats quality as contract conformance rather than user value. (B) blames users for a planning omission. (D) is a sensible forward control but doesn't acknowledge or remediate the current platform.
