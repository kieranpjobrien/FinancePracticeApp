# Question

The project team proposes using test-driven development (TDD) where tests are written before code. Some stakeholders question whether this is efficient. How should the PM frame the value of TDD?

## Options

- **A**: TDD is an industry best practice that all modern teams should follow
- **B**: TDD builds quality in by ensuring code is designed to be testable and requirements are clear upfront
- **C**: TDD eliminates the need for other forms of testing
- **D**: TDD primarily benefits developers by making debugging easier

## Answer

B

## Explanation

The correct answer is B. TDD is a prevention-focused practice. Writing tests first forces clarity about requirements and ensures code is designed to be testable. It builds quality in rather than testing it in afterwards, aligning with quality management principles.

Option A appeals to authority without explaining value. Stakeholders deserve to understand why a practice helps, not just that it's popular.

Option C overstates TDD's role. TDD provides unit-level verification but doesn't replace integration testing, user acceptance testing, or other quality activities.

Option D understates the value. While TDD does help debugging, its primary value is ensuring requirements are clear and code meets them.
