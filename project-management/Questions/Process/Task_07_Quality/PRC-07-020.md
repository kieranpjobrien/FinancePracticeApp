# Question

A project manager is establishing quality metrics for a new software product. The sponsor wants to track "code quality" but hasn't defined what that means. What should the PM do FIRST?

## Options

- **A**: Use industry-standard metrics like cyclomatic complexity and code coverage
- **B**: Work with stakeholders to define measurable quality attributes that matter for this project
- **C**: Ask the development team to recommend appropriate metrics
- **D**: Review similar past projects to see what metrics they used

## Answer

B

## Explanation

The correct answer is B. Quality is defined by stakeholder expectations. Before selecting metrics, the PM must understand what quality attributes matter for this specific project (reliability, performance, maintainability, security, etc.). Metrics should measure what stakeholders value.

Option A applies generic metrics that may not reflect what matters for this project. Code coverage might be irrelevant if stakeholders care most about performance.

Option C gets technical input but doesn't start with understanding stakeholder needs. The team might optimise for metrics that don't align with business value.

Option D provides useful reference but past projects had different stakeholders with different priorities. Each project needs its own quality definition.
