# Question

An agile team is delivering working software every two weeks, but stakeholders complain that while features work individually, the overall user experience is inconsistent and confusing. What quality practice should the team adopt?

## Options

- **A**: Extend sprints to allow more time for polish
- **B**: Add a hardening sprint at the end to fix UX issues
- **C**: Include UX review as part of the Definition of Done for each user story
- **D**: Hire a dedicated UX designer to review releases quarterly

## Answer

C

## Explanation

The correct answer is C. Including UX review in the Definition of Done ensures that quality (including user experience) is built into every increment. This is consistent with agile principles of delivering potentially shippable product each sprint and not accumulating technical or UX debt.

Option A (extend sprints) doesn't address the root cause. More time doesn't guarantee better UX if UX isn't part of the process.

Option B (hardening sprint) accumulates problems and fixes them in batches, which is anti-agile. It also means the team isn't delivering truly done increments during regular sprints.

Option D (quarterly UX review) is too infrequent. By the time issues are identified, the team has built months of inconsistent work that's expensive to fix.
