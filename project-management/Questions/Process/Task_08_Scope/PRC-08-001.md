# Question

During sprint planning, the product owner introduces a new feature as urgent, expecting it to be added to the current sprint that was already fully committed. What should the Scrum Master do?

## Options

- **A**: Accept the new feature to satisfy the product owner
- **B**: Reject the new feature since the sprint is already committed
- **C**: Facilitate a conversation about what current commitment would need to be removed to accommodate the new feature
- **D**: Add the feature but extend the sprint duration to accommodate

## Answer

C

## Explanation

The correct answer is C. Sprint scope is protected, but urgent business needs are real. The solution isn't rigid rejection or unlimited acceptance - it's trade-off discussion. If the new feature is truly more valuable than something already committed, the team and product owner can agree to a swap. This respects both sprint commitment and business needs.

Option A accepts scope creep without boundaries, undermining sprint predictability.

Option B rigidly protects commitment without considering genuine business urgency. Sometimes priorities do change.

Option D violates the sprint timebox, which is a foundational agile practice. If we can just extend sprints when scope grows, the timebox provides no discipline.
