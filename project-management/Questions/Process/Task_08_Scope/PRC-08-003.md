# Question

A PM discovers that several useful features delivered in the last release came from informal verbal requests to developers and were never in the approved scope baseline. Stakeholders are happy with them. What should the PM do?

## Options

- **A**: Quietly remove the features in the next release to restore the original baseline.
- **B**: Document the additions, assess impact on cost/schedule, and submit a retrospective change request through the CCB.
- **C**: Leave the baseline as-is and continue, since the customer is satisfied.
- **D**: Issue a written warning to the developers who built the features without authorisation.

## Answer

B

## Explanation

The work is done; ripping it out wastes effort and may upset users who now rely on it. The right move is to bring documentation back in line with reality through formal change control so the baseline reflects what was actually delivered. (A) destroys realised value to satisfy a procedural concern. (C) leaves baselines and reality misaligned, which corrodes future change control. (D) addresses people instead of the system gap that allowed informal requests to bypass scope.
