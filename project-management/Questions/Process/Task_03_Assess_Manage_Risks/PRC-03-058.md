# Question

An agile team faces a new external API integration where the team has no prior experience. The PO wants user stories estimated now for release planning. The team is nervous about committing. What should the PM advise?

## Options

- **A**: Commit the stories at the team's best-guess estimate so the release plan is complete and stakeholders have a firm date at this stage.
- **B**: Timebox a short technical spike to reduce uncertainty on the integration, then re-estimate the stories with evidence before committing to any release dates.
- **C**: Defer the integration to a later release entirely so the current release plan can proceed without this particular source of uncertainty.
- **D**: Add a large contingency buffer to the estimates so the team has room to absorb any unknown discovery along the way to delivery.

## Answer

B

## Explanation

PMI-aligned agile practice addresses uncertainty with a spike before commitment; estimates without evidence are false precision. (B) reduces the uncertainty before committing. (A) a guessed commit damages both plan credibility and team trust. (C) deferring avoids the problem but doesn't reduce future uncertainty. (D) padding without investigation hides rather than reduces the real risk exposure.
