# Question

A key developer who built critical system components is leaving the project in two weeks. The developer's knowledge isn't documented. What is the MOST effective way to capture this knowledge?

## Options

- **A**: Ask the developer to write comprehensive documentation before leaving
- **B**: Schedule pair programming sessions where the developer works alongside other team members on their code
- **C**: Record video tutorials of the developer explaining the system
- **D**: Have the developer create detailed comments in the code

## Answer

B

## Explanation

The correct answer is B. Pair programming provides hands-on knowledge transfer - the remaining team members learn by doing, can ask questions in context, and gain tacit knowledge that's hard to document. This is more effective than passive documentation because learning happens through practice.

Option A (comprehensive docs) sounds good but rarely works well with tight timelines. Two weeks isn't enough to document everything, and written docs often miss tacit knowledge.

Option C (video tutorials) captures information but lacks interactivity. Viewers can't ask follow-up questions, and videos become outdated as code changes.

Option D (code comments) helps but is limited. Comments explain what code does, not the reasoning behind design decisions or alternatives considered.
