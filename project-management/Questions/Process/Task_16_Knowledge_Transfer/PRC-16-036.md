# Question

A 12-month software platform build relied on three failed architectural approaches before the team found one that worked. The project manager wants to capture these failures in the lessons learned register, but team members are reluctant because they fear the documentation will be used in performance reviews. How should the project manager proceed?

## Options

- **A**: Frame the lessons around technical insights and decision context rather than individual actions, and confirm the register will not feed performance reviews.
- **B**: Capture the lessons confidentially without naming the team and submit them directly to the PMO without any team review first.
- **C**: Defer the lessons learned exercise until the team has rotated off the project so that participation no longer feels professionally risky.
- **D**: Ask each team member to submit a written account of the failed approaches they were involved in for inclusion in the project archive.

## Answer

A

## Explanation

PMI treats lessons learned as a blameless artifact focused on organisational learning, not individual accountability. (A) reframes the documentation around insights and decisions while addressing the underlying fear directly, which is what enables honest contribution. (B) loses the context the team holds and bypasses their participation, producing thin lessons. (C) delays capture past the point where memory and detail are still fresh, which is when retrospective insight is most accurate. (D) makes the activity feel like a personal report card and reinforces the very fear the PM is trying to dissolve.
