# Question

During a retrospective, the team identifies the same problem for the third time - unclear requirements causing rework. Previous action items haven't been implemented. What should the Scrum Master do?

## Options

- **A**: Add it to the action items again and hope for better results
- **B**: Investigate why previous actions weren't implemented and address barriers, then create a more specific, actionable improvement with clear ownership
- **C**: Stop doing retrospectives since they're not working
- **D**: Blame the product owner for unclear requirements

## Answer

B

## Explanation

The correct answer is B. Recurring problems indicate either wrong solutions or implementation barriers. The Scrum Master should understand why previous actions failed - too vague? no owner? conflicting priorities? Then create a better action item that's specific, has clear ownership, and addresses identified barriers. This is coaching the team to improve their improvement process.

Option A is the definition of insanity - repeating the same thing expecting different results.

Option C abandons a valuable practice instead of improving it.

Option D shifts blame without solving the systemic issue.
