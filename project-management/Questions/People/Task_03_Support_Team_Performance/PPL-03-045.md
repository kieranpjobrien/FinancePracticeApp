# Question

Code review feedback from two senior engineers to junior developers is technically accurate but consistently blunt and personal in tone. Junior developers have started holding pull requests open longer to avoid the feedback. What should the project manager do FIRST?

## Options

- **A**: Add a code review style guide to the working agreement and discuss it at the next retrospective.
- **B**: Have a private conversation with each senior engineer about the impact of their tone on team flow.
- **C**: Pair each junior developer with a different senior engineer so the existing feedback relationships are diluted.
- **D**: Move code review to a synchronous walk-through format so all feedback is delivered face-to-face instead.

## Answer

B

## Explanation

The behaviour is specific to two individuals and is causing measurable team harm, so the right first move is direct, private feedback to those two people, which is what (B) does. (A) is a useful follow-up but treats a personal behaviour issue as a team policy gap. (C) reshuffles the org around the behaviour without addressing it. (D) changes the medium but blunt feedback delivered live is at least as damaging as in writing and may be worse.
