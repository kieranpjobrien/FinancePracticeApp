# Question

A team uses acceptance criteria to define when user stories are complete, but developers and testers frequently disagree about whether criteria are met. What does this suggest and how should it be addressed?

## Options

- **A**: The testers are being too strict
- **B**: The developers aren't reading the criteria carefully
- **C**: The acceptance criteria lack shared understanding; facilitate collaborative criteria writing with examples that make expected behaviour concrete and testable
- **D**: Remove acceptance criteria to reduce conflict

## Answer

C

## Explanation

The correct answer is C. Disputes over acceptance criteria indicate the criteria weren't truly understood the same way by all parties. Collaborative writing (including developers and testers) surfaces different interpretations early. Concrete examples (Given-When-Then, specific scenarios) make criteria unambiguous. If criteria can be interpreted multiple ways, they're not specific enough.

Option A and B blame one party without addressing root cause.

Option D removes valuable quality definition.
