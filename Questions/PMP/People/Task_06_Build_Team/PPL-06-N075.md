# Question

On an agile team building a retail loyalty platform, only one member can write the automated tests, so stories pile up behind her in the last days of every sprint and several spill over. The product owner wants more throughput within the next sprint or two, and a major promotion launches in five weeks that the backlog must be ready for. The other developers are strong but have never used the test framework, and the specialist is already fully loaded. What should the project manager encourage the team to do?

## Options

- **A**: Have the specialist pair with the other developers as they work, so testing capability spreads across the team and the queue dissolves at its source over the coming sprints.
- **B**: Have the specialist write the tests while pulling forward the highest-risk stories first, so the most important work clears the bottleneck soonest before the launch.
- **C**: Bring in an experienced test engineer from another team for the run-up to the launch, adding throughput now while the in-house developers learn over time.
- **D**: Have the team write the tests themselves from the start of next sprint with the specialist reviewing, accepting slower, rougher tests in exchange for removing the single point of failure immediately.

## Answer

A

## Explanation

The bottleneck is that a whole capability sits in one person, and the durable agile fix is to spread it so any member can test; pairing transfers the skill through real work and steadily removes the constraint without sacrificing quality (A). Reordering by risk is sensible queue management, but it still funnels every story through the same single person, so the structural limit and the spillover remain — it manages the symptom, not the cause (B). Borrowing a test engineer adds short-term capacity but recreates a one-deep dependency on an outsider and does little to build the team itself; it buys time rather than resolving the fragility (C). Throwing all testing to untrained developers from day one does break the silo, but doing it cold with only review risks a wave of weak tests right before a major launch — too much quality risk at the worst moment, where guided pairing achieves the same end more safely (D). Pairing is the response that builds lasting capability without gambling quality against the deadline.
