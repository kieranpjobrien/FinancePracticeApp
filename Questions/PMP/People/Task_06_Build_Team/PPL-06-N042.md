# Question

A new agile team delivering a data-platform product has one data engineer, one analytics developer, and one QA automation engineer, each deeply skilled but narrow. The roadmap is aggressive and stakeholders expect a working increment from the very first sprints. By sprint two, completed development is queuing in front of QA and the increment slips. The data engineer and analytics developer have some testing background; the QA engineer has none in development. What should the project manager do?

## Options

- **A**: Have the data engineer and analytics developer take on testing for their own stories this sprint to clear the queue, and begin cross-skilling the team in parallel.
- **B**: Add a second QA automation engineer so the QA stage can match upstream development throughput.
- **C**: Limit work in progress so development cannot start more stories than QA can verify, keeping the increment flowing end to end.
- **D**: Reorder the next sprint's backlog so the heaviest QA items are spread across sprints rather than clustered.

## Answer

C

## Explanation

The increment is slipping because development is pushing more work into the system than QA can pull through; capping work in progress at the constraint immediately restores end-to-end flow and forces the upstream specialists to help at QA rather than pile up more unfinished work (C). (A) is a reasonable same-sprint move and supports cross-skilling, but without a WIP limit the developers will keep starting new stories and the queue re-forms. (B) adds capacity at one station, entrenches the silo, and will not land for several sprints. (D) smooths the QA load but treats the symptom; the bottleneck returns whenever QA-heavy work arrives because nothing constrains upstream starts.
