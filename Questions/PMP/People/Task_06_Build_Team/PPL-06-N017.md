# Question

A Scrum team building a retail e-commerce checkout consistently finishes coding early each sprint, then stalls because only one member can perform the security review every story needs before it is done. "Done" stories keep slipping into the next sprint. The security skill is moderately learnable, the single reviewer is willing to teach, and the work will continue indefinitely. How should the project manager best help the team?

## Options

- **A**: Add a second security reviewer to the team so review capacity matches the team's coding throughput.
- **B**: Have the team limit work in progress so fewer stories pile up at the review step and the single reviewer is never overwhelmed.
- **C**: Make security review the reviewer's sole focus and redistribute their other tasks so review never waits behind their other work.
- **D**: Support several developers in learning enough security review to share the work, with the current reviewer coaching them on real stories.

## Answer

D

## Explanation

A bottleneck at a single, learnable skill inside a team with continuing work is best dissolved by broadening that skill so the team can finish what it starts — the generalising-specialist pattern, made durable here by coaching on real stories (D). (A) is reasonable and would raise capacity, but it grows the team and leaves checkout's security knowledge concentrated in two people rather than the whole team; it treats the symptom more than the structural constraint. (B) limiting WIP genuinely smooths flow and would reduce pile-ups, yet it caps the team to the reviewer's rate instead of lifting that rate — useful alongside D, insufficient alone. (C) maximises one person's review output but entrenches the single-point dependency exactly where the work is permanent.
