# Question

An agile team building a gaming studio's live-operations service has delivered well, but the project manager sees a fragility: each of the four critical skills — backend, real-time networking, analytics, and live-content tooling — sits with exactly one person. Any single absence stalls a whole area, and a heavy release period is approaching when an outage cannot wait. The team is willing and works well together. Which TWO actions should the project manager take to build a more resilient team? (Select TWO.)

## Options

- **A**: Introduce regular pairing and rotation so members deliberately work alongside the specialists and build genuine second-area competence over time.
- **B**: Have each specialist document their area and run knowledge-sharing sessions, leaving durable references the rest of the team can use and extend.
- **C**: Recruit two experienced generalists into the team who can ramp across several of the four areas, broadening coverage without disrupting the specialists during the release run-up.
- **D**: Stand up a clear on-call rota with documented runbooks for each area, so whoever is on duty can follow defined steps to handle an incident outside the specialist's hours.
- **E**: Prioritise reducing each area's bus factor in the backlog, allocating explicit capacity each sprint to whatever raises coverage, and let the team choose how.

## Answer

A, B

## Explanation

The fragility is that critical knowledge is locked in individuals, so resilience comes from transferring that knowledge into the team: pairing and rotation build real second-area competence through doing (A), and specialist-led documentation plus knowledge-sharing leave durable, extensible references others can draw on (B) — together they directly dissolve each single point of failure. Recruiting generalists adds coverage but takes time to ramp, costs headcount, and still leaves the existing four areas dependent on the original specialists in the near term; it supplements rather than fixes the siloed knowledge (C). An on-call rota with runbooks is sound operational practice and worth having, yet runbooks cover known incidents only and a duty member without the underlying skill will still stall on anything novel — it mitigates the symptom during outages rather than building shared capability (D). Allocating backlog capacity to reducing bus factor is a reasonable enabling mechanism, but it is a way to make room for the actual work, not the work itself; on its own it names the goal without specifying the knowledge transfer that achieves it (E). The first-order actions are cross-skilling and knowledge transfer (A, B); the others are supports or partial mitigations.
