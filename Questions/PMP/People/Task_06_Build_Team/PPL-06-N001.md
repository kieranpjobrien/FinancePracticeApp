# Question

A telehealth provider is forming the Scrum team for a patient-scheduling product built in two-week sprints. The candidate pool is deep in front-end and API skills, but only one available engineer knows the hospital's HL7 interface, and roughly a third of the near-term backlog touches that interface. The product owner has fixed an aggressive first release and expects most stories to be picked up in parallel. The project manager has already decided to include the HL7 specialist on the team. As the team is being assembled, what should the project manager do?

## Options

- **A**: Add a second HL7-capable contractor for the first few sprints so two people can take interface stories in parallel from the outset.
- **B**: Work with the product owner to order the backlog so HL7-dependent stories are spread across sprints, and pair a front-end engineer with the specialist on the early interface stories.
- **C**: Bring the specialist and the product owner together to re-forecast the release date against realistic interface throughput before committing the team to it.
- **D**: Have the specialist write up the HL7 integration patterns as reference material the rest of the team can follow when they reach those stories.

## Answer

B

## Explanation

All four are things a competent PM might do, but the question is how to *build* this team given a single-skill constraint while keeping the committed release in view. Pairing grows a second capable contributor from within the team while sequencing keeps work flowing around the constraint, addressing both flow and key-person risk in the team you are actually forming (B). (A) relieves the immediate bottleneck but buys capacity rather than building the team's own capability, and adds onboarding load; it is a fallback, not the first move when a willing internal engineer can be grown. (C) is prudent and may follow, but re-forecasting before you have tried to relieve the constraint treats the date as the only variable; team design comes first. (D) helps knowledge spread but reference docs alone rarely transfer a genuinely scarce integration skill fast enough; hands-on pairing is the stronger capability-building act.
