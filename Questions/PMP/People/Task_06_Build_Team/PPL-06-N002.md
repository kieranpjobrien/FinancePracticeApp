# Question

A predictive water-treatment upgrade has an approved scope statement and a baselined WBS, but no individuals are named yet. The sponsor wants the number and type of people the project needs this week so functional managers can be approached. The project manager has the WBS, the activity list, a prior treatment-plant project's staffing records, and access to organisational process assets. What should the project manager rely on PRIMARILY to size the team the project needs?

## Options

- **A**: The prior treatment-plant project's staffing records, adjusted in proportion to the difference in contract value between the two projects.
- **B**: The resource requirements estimated from this project's own activities, with the prior project's staffing and process assets used to calibrate those estimates.
- **C**: A resource breakdown structure built from the WBS, naming the categories and quantities of resource per work package.
- **D**: The functional managers' advice on what headcount they could realistically supply for work of this kind.

## Answer

B

## Explanation

All four touch real inputs, so the question turns on what *primarily* drives the requirement. Team size follows from this project's own decomposed work, estimated into resource requirements, with analogous data and process assets used to sanity-check rather than replace that estimate (B). (C) is genuinely useful and often produced alongside, but a resource breakdown structure organises and categorises the resources; it does not by itself establish the quantities — those come from estimating the activities, which is why it is a component of the answer rather than the primary basis. (A) leans on analogy scaled by a crude proxy (contract value), which can mislead when scope content differs. (D) lets supply shape the requirement, sizing the team to what is convenient to release rather than to what the work demands.
