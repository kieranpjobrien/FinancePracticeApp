# Question

In a programme with five feature teams feeding one integration team, increments are being rejected because teams interpret "done" differently — some stop at code-complete, others include automated tests and documentation. There is no shared written Definition of Done. The release train engineer asks you to fix it before the next integration. Several approaches are on the table. Which is best?

## Options

- **A**: Have the integration team, which feels the pain of bad handoffs, define the Definition of Done and circulate it to the feature teams to apply.
- **B**: Take the most thorough team's existing Definition of Done and roll it out to the other four as the programme standard.
- **C**: Bring representatives from all five feature teams and the integration team together to agree a single Definition of Done, then have each team adopt it.
- **D**: Have each feature team document its current Definition of Done and reconcile the differences at the next programme-level retrospective.

## Answer

C

## Explanation

A Definition of Done that governs work crossing several teams plus an integrator is a shared ground rule, and it sticks only when all the teams who must meet it help set it. (C) brings every affected team to the table, so the standard fits the programme and carries the buy-in that makes teams self-enforce it. (A) puts the integration team's real pain to good use but a DoD handed down by one downstream group becomes an externally policed rule rather than a shared norm, so feature teams comply grudgingly. (B) is fast and the thorough team's bar may even be right, but adopting one team's standard wholesale skips the others' input and may set requirements that do not fit every team's context, recreating resistance. (D) surfaces the differences but defers the fix to a later retrospective, leaving the next integration to fail with the inconsistency still unresolved — the RTE asked for it fixed before then.
