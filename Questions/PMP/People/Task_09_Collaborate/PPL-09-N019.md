# Question

On an agile government digital-services project, the policy team, the service-delivery team and the accessibility team each have a different view of what "done" and "valuable" mean for the next release, and the product backlog reflects whoever lobbied the product owner most recently. Releases keep satisfying one group and disappointing the others. As the project lead, which actions would most durably build genuine alignment among these stakeholders? (Select TWO.)

## Options

- **A**: Facilitate a joint session where the three teams agree shared, explicit value criteria and a definition of done they all sign up to, then order the backlog against them.
- **B**: Run regular collaborative backlog refinement with all three groups present, so prioritisation decisions are made openly against the agreed criteria.
- **C**: Have the product owner publish a clear written prioritisation policy describing how competing requests will be weighed, and circulate it to all three teams.
- **D**: Set up a joint release-planning session each release where the three groups negotiate which items make the cut, so everyone has a say in the contents.
- **E**: Introduce a shared value-scoring model the product owner applies to each request, so backlog position is determined by a consistent calculation rather than by lobbying.

## Answer

A, B

## Explanation

Durable alignment needs both a shared *basis* for value and a recurring, transparent *forum* to apply it: agreeing common value criteria and a joint definition of done (A) replaces conflicting private views with a collective standard, and ongoing collaborative refinement with all three groups present (B) keeps every prioritisation decision open and criteria-based rather than lobbying-driven. (C) a written prioritisation policy is helpful, but a one-way document the product owner issues does not, on its own, build the shared ownership of *what counts as valuable* — teams can still dispute the inputs it runs on. (D) per-release negotiation gives everyone a say but recreates horse-trading each cycle without a stable shared definition of value beneath it, so alignment is re-fought every release rather than built. (E) a value-scoring model improves consistency, yet if the teams have not jointly agreed the criteria it encodes, it simply moves the disagreement to the model's inputs and can feel like lobbying dressed up as arithmetic.
