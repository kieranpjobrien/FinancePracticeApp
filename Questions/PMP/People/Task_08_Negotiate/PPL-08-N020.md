# Question

A customer wants a new e-commerce platform delivered for a fixed budget and a fixed launch date, but acknowledges the detailed feature set is still uncertain. The project manager is negotiating the form of the agreement so both sides are protected. Which terms make this a sound agreement for an agile delivery? (Select THREE.)

## Options

- **A**: Fix the budget and the timeline, but treat the detailed scope as variable, to be drawn from a prioritised backlog as the work proceeds.
- **B**: Agree that the customer's product owner will continuously re-prioritise the backlog so the most valuable features are built within the fixed envelope.
- **C**: Define a minimum viable release the supplier commits to deliver by launch, with further features added only as capacity allows.
- **D**: Lock the complete detailed feature list now so the supplier is contractually bound to deliver every item by the fixed date for the fixed price.
- **E**: Bind the supplier to a fixed velocity for every sprint and tie payment to hitting that story-point figure each iteration.

## Answer

A, B, C

## Explanation

A fixed-budget, fixed-date, variable-scope agreement is the standard agile contracting pattern: hold cost and time, flex scope from a prioritised backlog (A), give the product owner authority to re-prioritise so value is maximised within the envelope (B), and commit to a defined minimum viable release so the customer is assured of a usable outcome (C). (D) re-imposes a fixed full scope on top of a fixed date and price, which is the over-constrained "iron triangle" trap agile contracting avoids. (E) is the subtler error: velocity is an empirical planning measure that naturally varies, so contracting a guaranteed story-point figure per sprint and paying against it rewards inflating estimates and gaming the metric rather than delivering value, and it punishes the team for the normal variation agile expects — it protects neither party the way a minimum-viable-release commitment does.
