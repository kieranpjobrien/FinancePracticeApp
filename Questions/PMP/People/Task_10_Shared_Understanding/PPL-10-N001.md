# Question

On a logistics tracking platform, a feature is accepted at sprint review, but two days later warehouse end users complain it does not match what they described in refinement. The developers show that the written acceptance criteria were met exactly. The product owner relays warehouse needs second-hand; the warehouse users do not attend refinement, and the team is already part-way through building three similar stories from the same source. As the project manager, what is the most effective way to prevent this recurring across the remaining stories?

## Options

- **A**: Have a warehouse user join refinement alongside the developers so intent is confirmed directly between the people who hold the need and those who build it
- **B**: Introduce example mapping in refinement so each story's rules and concrete examples are agreed before it is pulled into a sprint
- **C**: Ask the product owner to validate each drafted acceptance criterion with the warehouse users before refinement
- **D**: Add a usability check with real warehouse users to the definition of done so mismatches are caught before the story is accepted

## Answer

A

## Explanation

All four are legitimate shared-understanding moves, so the discriminator is which one removes the actual cause rather than a downstream effect. The defect is structural: requirement intent passes through the product owner as a single relay and is never confirmed between the user and the developer, so the same distortion will recur on every story. Bringing a warehouse user into refinement with the developers (A) closes the gap at its source for all remaining stories at once. Example mapping (B) is a strong technique, but run with only the product owner and team present it still pins down examples based on the relayed (possibly wrong) intent — it sharpens precision without verifying the source. Product-owner pre-validation (C) keeps the same intermediary who has already proven lossy, just adding a round trip. A definition-of-done usability check (D) detects the mismatch only after build, accepting the waste that direct alignment would prevent. A is earliest and addresses the root relay problem.
