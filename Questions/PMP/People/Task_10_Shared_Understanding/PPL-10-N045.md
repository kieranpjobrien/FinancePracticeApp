# Question

At sprint reviews for a healthcare scheduling product, business stakeholders nod along and approve the increment, yet weeks later they raise the same concerns as if the features were never discussed, claiming the team "built the wrong thing". The product owner is confident the right things were demonstrated. Which actions would most effectively build genuine shared understanding at the reviews? (Select all that apply.)

## Options

- **A**: Have stakeholders interact directly with the working increment during the review rather than watching a scripted demo
- **B**: Send stakeholders a written feature summary before the review so the session can focus on confirming sign-off
- **C**: Capture decisions and feedback visibly during the review and confirm them back to the room before closing
- **D**: Demonstrate the increment strictly against the original acceptance criteria so stakeholders can verify each was met
- **E**: Ask open questions that prompt stakeholders to describe how the feature meets their need, exposing any mismatch

## Answer

A, C, E

## Explanation

Passive nodding at a scripted demo is a classic illusion of agreement; understanding is genuinely built when stakeholders use the increment themselves (A), articulate in their own words how it meets their need so mismatches surface (E), and see decisions captured and confirmed back so consensus is explicit (C). Sending a written summary in advance to streamline sign-off (B) optimises for getting approval faster, but it is reading-then-approving — the same passive confirmation that already produced false agreement. Demonstrating strictly against the original acceptance criteria (D) only proves the team built what was *written*; the stakeholders' complaint is that the written intent diverged from their real need, so a criteria-by-criteria check confirms conformance to the wrong target rather than rebuilding shared understanding of the need itself.
