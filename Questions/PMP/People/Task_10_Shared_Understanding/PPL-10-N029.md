# Question

An agile travel-booking team already runs strong collaborative refinement: developers restate stories, co-author acceptance criteria, and stories leave refinement well understood. Yet for one recurring class of work, integrations with third-party airline APIs, the same stories still spawn a stream of mid-sprint questions, because the answers hinge on quirks of each external API that nobody in the room actually knows at refinement time. The product owner is engaged but is not an integration specialist. As the project manager, what is the best way to give the team genuine shared understanding of these stories before the sprint?

## Options

- **A**: Add a timeboxed spike before each integration story so the team investigates the specific API and brings concrete findings into refinement to ground the acceptance criteria
- **B**: Invite an engineer from the airline's integration team to the refinement session where their API is being discussed
- **C**: Have the team capture each mid-sprint question and its answer in a shared integration reference so future stories can draw on it
- **D**: Split each integration story into a thin end-to-end slice the team builds first to expose the unknowns, then refine the remaining scope with what they learn

## Answer

A

## Explanation

The gap here is not about dialogue or intent, which this team's refinement already handles well, but about missing external knowledge that no participant holds, so the team must acquire that knowledge and feed it into refinement before committing the story. A timeboxed spike does exactly that: it produces concrete findings about the specific API that turn speculative acceptance criteria into grounded ones the whole team shares. Option B can work when the external party is available, but it depends on a third party's goodwill and calendar and often cannot be arranged per story, making it a less reliable fit. Option C builds a useful asset over time but is reactive, helping only after a question has already cost a sprint, and each brand-new API starts from an empty reference. Option D surfaces the unknowns by building, which is legitimate, but it pulls discovery into the sprint as committed work rather than equipping the team to understand the story beforehand, which is what was asked. All four are genuine agile practices; A most directly closes this specific knowledge gap before the sprint.
