# Question

A scrum team's retrospectives have become routine: members list a few minor process tweaks, agree them, and little changes between sprints. The same impediment — slow code review turnaround — has appeared in the last four retrospectives, each time generating an action item that no one carries through. The PM, as scrum master, wants the team's own improvement to start producing real change. What is the best action?

## Options

- **A**: Help the team focus the next retrospective on this one recurring impediment, dig into its root cause together, and commit to a single experiment with a named owner and a check-in next retrospective.
- **B**: Bring the slow code-review data to the retrospective — cycle times, where reviews queue — so the team diagnoses from evidence rather than impressions, and let them decide the action from there.
- **C**: Change the retrospective format each sprint using different facilitation techniques, so stale routine is broken and the team engages more deeply with the issues it raises.
- **D**: Have the team track its retrospective action items on the board alongside delivery work, so the commitments stay visible and accountability for follow-through improves.

## Answer

A

## Explanation

Two things are failing: attention is spread across minor tweaks instead of the impediment that keeps recurring, and the actions that are agreed have no ownership so nothing follows through. (A) fixes both — it concentrates the retrospective on the persistent problem, drives to root cause, and secures a single owned experiment with a follow-up check, which is the pattern that converts retrospective talk into change. (B) is a strong move and evidence sharpens diagnosis, but data addresses only the diagnosis half; without focusing the session and pinning ownership and follow-through, a well-diagnosed action can still die exactly as the last four did. (C) refreshes engagement but treats boredom as the cause when the real gaps are focus and accountability; a livelier format that still produces unowned actions changes nothing. (D) directly targets the follow-through gap and is a sound complementary habit, but visibility alone does not focus the team on root cause, and an item with no owner stays unactioned on the board just as it did in the notes.
