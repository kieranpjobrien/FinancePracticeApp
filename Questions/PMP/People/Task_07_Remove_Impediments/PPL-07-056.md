# Question

A third-party API that your product integrates with has just announced deprecation of its current version in six months, with a replacement API that requires non-trivial rework. The team is mid-build on the current version. What should the project manager do?

## Options

- **A**: Finish the current build against the existing API, ship it on schedule, and plan the migration to the replacement API as a separate follow-up project after go-live.
- **B**: Reprioritise the backlog: assess migration effort, phase current work so it can adopt the new API early where viable, and treat the deprecation window as a hard date in planning.
- **C**: Escalate to the sponsor to seek an extension of the deprecation window through commercial negotiation with the third party, and pause the migration until that outcome is known.
- **D**: Abandon the integration with this third-party entirely and source an alternative provider whose API is more stable, even if this means rebuilding the integration from scratch.

## Answer

B

## Explanation

PMI treats a known hard external deadline as a planning input that reshapes the backlog, not something to be finished around, negotiated away, or replaced wholesale. (B) does that. (A) guarantees a migration project immediately after go-live on a hard deprecation clock. (C) relies on negotiation with a vendor that has already published a schedule. (D) over-reacts by replacing a working third-party choice.
