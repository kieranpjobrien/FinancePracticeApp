# Question

On a predictive hospital-IT project, a developer tells the project manager that a test server the team needs has not been provisioned, and configuration work will run out of tasks in about two days without it. Provisioning runs through the IT operations team, whose intake queue is typically a few days; the team also has a backlog of unrelated configuration items it could pull forward. What should the project manager do?

## Options

- **A**: Submit the provisioning request to IT operations through the normal intake and monitor it against the two-day horizon.
- **B**: Reorder the work to bring the unrelated configuration backlog forward so the team keeps moving while provisioning is sorted in parallel.
- **C**: Request the server and, given the intake lead time, flag the two-day risk to the IT operations lead so the request is treated as time-critical.
- **D**: Escalate the provisioning need to the IT operations manager and the project sponsor so it is prioritised over other teams' requests.

## Answer

C

## Explanation

The server request itself is routine, but the standard intake lead time is close to the two-day horizon, so simply lodging it normally (A) risks the queue outrunning the deadline — correct mechanics, wrong sense of urgency. The strongest move is to submit it and, because of the timing, surface the two-day risk to the IT operations lead so it is handled as time-critical (C): same channel, but with the impact made visible to the person who controls the queue. (B) pulling unrelated backlog forward keeps the team busy but is a coping move that leaves the actual blocker unaddressed and is best done alongside, not instead of, chasing the server. (D) escalating to the operations manager and sponsor to leapfrog other teams is disproportionate before the owning lead has even been given the chance to expedite. The margin is between lodging-and-watching (A), working around (B), and lodging-with-urgency through the right owner (C).
