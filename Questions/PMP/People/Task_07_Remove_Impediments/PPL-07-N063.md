# Question

Over the last three sprints, an agile team developing telematics software has seen stories pile up in a "ready for testing" column because the two testers are consistently overloaded while four developers keep pulling new work. Velocity looks healthy but little actually reaches "done", and morale is dipping. As the Scrum Master, which TWO actions most directly address this flow impediment? (Choose two.)

## Options

- **A**: Introduce a work-in-progress limit on the testing column so developers stop starting new work and instead help clear the testing queue.
- **B**: Use the next retrospective to surface the bottleneck data and have the team agree to swarm on testing tasks or build in more shared testing skill.
- **C**: Ask the product owner to reduce the number of stories committed each sprint until the testing backlog clears.
- **D**: Track testing cycle time and report it to leadership so additional testers can be justified.
- **E**: Reassign the two testers to report to a central QA function so their workload is balanced across the organisation.

## Answer

A, B

## Explanation

The root cause is a flow constraint — developers outpacing testing capacity — so the two actions that most directly relieve it are imposing a WIP limit that forces the team to finish before starting (A) and using the retrospective to agree swarming and skill-sharing so the bottleneck is collectively owned (B). (C) cutting commitment lowers throughput but treats the symptom rather than balancing the workflow and can mask the real constraint. (D) gathering cycle-time evidence is useful groundwork for a longer-term capacity case, but it does not unblock the current flow. (E) moving testers to a central function removes them from the team and is likely to worsen, not improve, responsiveness. The strongest pair rebalances flow within the team's own control.
