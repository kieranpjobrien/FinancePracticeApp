# Question

Your project is mid-sprint, but the enterprise change-freeze for the year-end peak period begins in two days and will last six weeks. The freeze policy blocks any non-emergency deployment. Your team needs to release a routine change next week for integration testing. What should the project manager do?

## Options

- **A**: Push the routine change to production quickly today, before the freeze formally begins, even though the change is not fully ready for release into the production environment.
- **B**: Engage the change authority with a documented risk assessment, request an authorised pre-freeze slot or an exception for a pre-production environment, and re-plan the sprint if needed.
- **C**: Submit the change as an emergency deployment during the freeze so that it can be deployed anyway, on the assumption that integration testing will count as sufficient business justification.
- **D**: Skip integration testing entirely for the next six weeks, continue writing code blind against unverified integration, and plan to fix any problems once the change freeze lifts.

## Answer

B

## Explanation

PMI works with the change authority to find a compliant path, balancing risk and delivery, rather than rushing an unready release, misusing emergency categories, or skipping integration testing. (B) does that. (A) pushes unready code to production. (C) falsifies an emergency category, which is both a policy and an ethics breach. (D) creates six weeks of unverified code that is almost certain to surface issues post-freeze.
