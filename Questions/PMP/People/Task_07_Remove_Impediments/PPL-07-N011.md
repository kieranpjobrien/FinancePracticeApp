# Question

A distributed agile team raises in the retrospective that the shared build pipeline fails most nights, and developers lose roughly the first hour each morning working out whether a red build is their code or the flaky environment. The infrastructure is owned and operated by a separate platform team. The team has quietly learned to live with it. What is the best response from the project manager?

## Options

- **A**: Set up an isolated build agent dedicated to this team, decoupling them from the shared infrastructure that keeps failing so their mornings are protected.
- **B**: Add automated build-health notifications that post the environment's status at start of day, so developers can instantly tell a real failure from an environment one and stop losing the diagnosis hour.
- **C**: Quantify the lost hours and recurring impact and take it to the platform team that owns the pipeline as a prioritised impediment, working with them to stabilise or isolate the build environment and tracking it to resolution.
- **D**: Have the team add a quick automated smoke test that runs first and distinguishes environment failures from code failures, surfacing the cause immediately each morning.

## Answer

C

## Explanation

Unusually, all four options genuinely reduce the pain and several are good engineering — the margin is whether the action removes the impediment at its source or merely insulates the team from it. The root cause is unstable infrastructure owned by another team, and a recurring obstacle of this size warrants raising it as a prioritised impediment to the owning team and driving it to resolution, which is the only path that actually fixes the failing pipeline (C). (B) is a smart, cheap mitigation that recovers much of the lost hour, but it diagnoses the failure faster rather than stopping it — the environment still breaks nightly. (A) is the strongest of the workarounds and could be part of the fix, but standing up a separate agent sidesteps the shared instability for one team while leaving the underlying system broken for everyone, and may not even be within this team's authority. (D) similarly speeds up diagnosis without preventing the failures. The defining factor is that only C addresses the cause with the team that owns it, rather than helping the team tolerate a removable obstacle.
