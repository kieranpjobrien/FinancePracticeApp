# Question

A Kanban team supporting an e-commerce platform has a WIP limit of three on the "In Test" column, which has sat full and stalled for a week because their single shared test environment keeps being overwritten by another team. Rather than sit idle, the developers have pulled new development work, and the untested backlog is now growing. Securing a dedicated environment requires a budget approval that will take a fortnight. As the team facilitator, what should you do first?

## Options

- **A**: Have the team stop pulling new development work and swarm on getting the three cards in "In Test" through, using the environment in protected windows the two teams agree between themselves.
- **B**: Open the budget request for a dedicated test environment now, since the recurring contention is the root cause and only a separate environment removes it for good.
- **C**: Raise the "In Test" WIP limit so the developers can keep cards flowing into test and development without stalling while the environment situation is worked.
- **D**: Convert the idle developer time into writing automated tests for the stalled cards, so when environment access returns the cards clear faster.

## Answer

A

## Explanation

The permanent fix (a dedicated environment) is two weeks out, so the question is what restores flow now and stops the damage already happening — work piling up untested. (A) does both: stopping new pulls halts the growing untested backlog, swarming clears the existing three, and a negotiated sharing window gets the team usable access immediately while the budget request runs in parallel. (B) is the correct longer-term root-cause action and should be started, but as the *first* move it leaves the team idle and the untested backlog growing for a fortnight — it does not restore flow now. (C) raising the WIP limit is the classic anti-move: it lets even more unfinished work accumulate behind the same bottleneck, hiding the constraint that WIP limits exist to expose. (D) keeps people busy and has some value, but it neither clears the stalled cards nor stops the backlog growth, and it presumes automation is what is missing. The margin is between restoring flow with the access you can get today while pursuing the permanent fix (A) and either waiting on the slow fix (B), masking the constraint (C), or staying busy without unblocking (D).
