# Question

An agile team developing a fraud-detection service finds that most of its defects originate in a few intricate rules-engine modules and are only caught days later in integration testing, where they are costly to trace back. The team already has solid automated unit tests, yet defects in the complex branching logic still get past them. The project manager wants detection to happen closer to where these defects are introduced. Which practice best achieves that for these modules?

## Options

- **A**: Introduce peer review or pair programming on the complex rules-engine modules so a second engineer examines the logic as it is written, surfacing defects where they originate.
- **B**: Raise unit-test coverage targets across the whole codebase so more of the code, including the rules engine, is exercised by automated tests.
- **C**: Schedule integration testing earlier and more frequently in the sprint so the same defects are caught some days sooner than they are now.
- **D**: Add property-based or fuzz tests that generate many input combinations against the rules engine to probe edge cases the current unit tests miss.

## Answer

A

## Explanation

The defects sit in intricate logic, already evade good unit tests, and are found downstream at integration — so the goal is detection at the moment the logic is created. Peer review or pairing puts a second mind on the complex code as it is written, catching design and logic flaws at their source, the earliest possible point. (A) is correct. (B) adds more of the same kind of automated test that is already missing these subtle logic defects; broad coverage does not guarantee catching hard branching errors, and it is still machine checking, not examination at authoring time. (C) and (D) are both genuine improvements but neither catches the defect at creation: earlier integration testing still finds it after assembly, only sooner, and generative tests — though strong at edge cases — still execute after the code is written rather than scrutinising the logic as it is built. The discriminator is "at the point the logic is authored" versus "valuable but still after it exists."
