# Question

On a logistics platform, developers report that accumulated technical debt is driving a rising defect rate and slowing each release. They have quantified the impact and shared it. The product owner, pressured by a competitor, has loaded the next three sprints entirely with customer-facing features and, when asked, declines to schedule any refactoring. The defect trend keeps climbing. As the project manager, what is the most appropriate next step?

## Options

- **A**: Bring the quantified defect-and-velocity data to the product owner and walk through how the rising debt is now reducing the throughput of the very features being prioritised, so the trade-off is decided with that consequence explicit.
- **B**: Facilitate the team to agree a Definition of Done and quality criteria for new work so further debt stops accumulating, and leave the existing backlog ordering as the product owner has set it.
- **C**: Propose the team reserve a fixed slice of each sprint for the highest-impact debt while still delivering features, for the product owner to accept.
- **D**: Continue surfacing the quality data each sprint and respect the ordering, since prioritising the backlog is the product owner's accountability.

## Answer

A

## Explanation

The product owner owns ordering, so the project manager cannot insert refactoring or override the ranking — but the project manager can make sure the decision is being made on the real consequences. The data already exists; what has not happened is connecting it to the product owner's own goal, namely that mounting debt is throttling feature delivery, which reframes refactoring from "competing with features" to "protecting feature throughput". (A) does this and keeps the decision with its accountable owner. (B) is a strong move and may well follow, but a Definition of Done only stops *new* debt; the existing debt is what is driving the climbing defect trend, so on its own it leaves the actual problem growing. (C) is a fair negotiation, yet it proposes consuming sprint capacity the product owner controls and has effectively already been refused; pressing the same ask without first reframing the trade-off is unlikely to move a product owner who has just declined. (D) takes no new action while the defects worsen, mistaking "respect the role" for passivity when the project manager has a legitimate, non-overriding lever left to use.
