# Question

Over many sprints a product's codebase has accumulated significant technical debt; new features now take noticeably longer and defect rates are climbing. The product owner wants to keep all capacity on customer-visible features and views refactoring as wasted effort. The team is frustrated. What is the BEST way for the project manager to handle this?

## Options

- **A**: Defer to the product owner's prioritisation, since maximising customer-visible value is the team's purpose.
- **B**: Let the developers quietly refactor within feature stories so the work happens without needing the product owner's agreement.
- **C**: Help the team make the cost of the technical debt visible in terms of slowing delivery and rising defects, so the product owner can weigh refactoring as a value-protecting investment.
- **D**: Schedule a separate technical-debt project after the current release to address the accumulated issues in one effort.

## Answer

C

## Explanation

Technical debt is a quality and value issue, but the product owner is treating it as pure cost because its impact is invisible to them. The PM's role is to make the trade-off transparent, quantifying how the debt slows delivery and raises defects, so the product owner prioritises it as protecting future value (C). (A) ignores a worsening quality trend the PM is obliged to surface. (B) hides work from the person accountable for the backlog and erodes trust. (D) batches the debt into a big-bang effort, which is riskier and contrary to addressing quality incrementally.
