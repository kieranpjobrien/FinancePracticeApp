# Question

An agile team delivering a mobile ticketing app keeps producing the same kinds of defects sprint after sprint. Stakeholders still accept each increment, and the product owner is already adding the specific defects to the backlog to be fixed. The project manager judges that fixing the individual defects will not stop the pattern, because something in how the team works keeps regenerating them. Within the team's existing events, where should the project manager steer the conversation about changing that?

## Options

- **A**: The sprint retrospective, where the team examines how it worked this sprint and commits to a process change aimed at the recurring defects.
- **B**: Backlog refinement, where the recurring defect types can be written up and prioritised by the product owner alongside the fixes already added.
- **C**: The sprint review, extended so stakeholders help analyse the defect pattern when they accept the increment.
- **D**: The daily standup, by adding a standing item so the team tracks the defect trend together every morning.

## Answer

A

## Explanation

The product fixes are already being handled on the backlog; the open need is to change the team's way of working so the defects stop recurring, and each event is a genuine venue for some agile activity, which is what makes the choice require discrimination rather than elimination. The retrospective is the event explicitly for the team to inspect its own process and agree improvements, so that is where a change to how the work is done belongs. (A) is correct. (B) is the right place for the defect items as work to prioritise, but refinement shapes the backlog, it does not change the team's process. (C) misuses the review, which is for inspecting the increment and gathering stakeholder feedback on the product, not for reworking internal practice. (D) overloads the standup, a short event for coordinating the day's work; trend-watching there surfaces the symptom without being the place to redesign the process.
