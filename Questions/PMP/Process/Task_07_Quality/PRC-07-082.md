# Question

On a hybrid laboratory-information-system project, an automated sample-tracking module is demonstrated at a sprint review. It behaves exactly as its written acceptance criteria specify, and the development team built it faithfully to those criteria. The lab manager who will use it daily refuses to accept it, saying it does not match how the lab actually logs samples. Checking the records, the project manager sees the acceptance criteria were drafted months ago by a business analyst and were never put in front of the lab manager. What should the project manager do first?

## Options

- **A**: Sit with the lab manager now to establish the criteria that reflect how samples are really logged, then route the corrected requirement through the backlog before the module is reworked.
- **B**: Record the module as meeting its agreed acceptance criteria and accept it, then handle the lab manager's concerns as a new change request for a later iteration.
- **C**: Rework the module immediately to match what the lab manager described at the review, so the deliverable becomes usable as quickly as possible.
- **D**: Escalate to the sponsor to confirm whether the documented, met criteria or the lab manager's objection should govern acceptance.

## Answer

A

## Explanation

The module passed verification — built to its stated criteria — but those criteria were never validated against the real user need, so the defect is in the requirement, not the build, and every option must reckon with a deliverable that is technically conforming yet unusable. The first step is to close the validation gap at its source: confirm the correct criteria with the process owner, then feed them back through the backlog before reworking. (A) is correct. (B) accepts a conforming-but-wrong deliverable and defers the real need, banking value the lab cannot use. (C) jumps straight to rework from an informal verbal description given at a review, before the true criteria are properly captured and agreed — risking a second wrong build. (D) escalates a requirements-validation problem as if it were an authority dispute; the answer is not for the sponsor to pick which criteria win but to validate what the lab actually needs.
