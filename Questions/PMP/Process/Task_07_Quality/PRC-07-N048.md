# Question

A hybrid programme migrating a banking application to the cloud has agile squads releasing every two weeks, but each release is gated by a two-day manual regression test run by a separate QA group. Under deadline pressure the QA group is increasingly waved through, and escaped defects are rising. The programme has limited engineering capacity this quarter and cannot do everything at once. What should the project manager advocate as the first improvement?

## Options

- **A**: Begin automating the regression suite for the most defect-prone, frequently changed areas and run it in the pipeline, expanding coverage as capacity allows.
- **B**: Embed the QA testers inside the squads so regression testing happens within each sprint instead of as a separate downstream gate.
- **C**: Make the two-day manual regression a mandatory release gate that cannot be waived, to stop defects escaping under deadline pressure.
- **D**: Stagger the squads' release dates so the single QA group has enough time to fully regression-test each release in turn.

## Answer

A

## Explanation

The release cadence has outgrown a slow, manual, hand-off-based check, which is why it gets skipped and defects escape. With limited capacity, the highest-value first move is to start automating regression where the risk and change rate are greatest and run it continuously in the pipeline, building quality into the flow and removing the bottleneck incrementally. (A) does this. (B) is a genuine improvement — moving testing into the squads — but with one QA group and frequent releases it redistributes manual effort rather than making the checks keep pace, and it does not scale as releases multiply. (C) enforces the existing slow gate; it stops the waving-through but reinstates the bottleneck the cadence cannot absorb, throttling delivery. (D) slows the flow of value to fit a manual process, treating the symptom by reducing release frequency.
