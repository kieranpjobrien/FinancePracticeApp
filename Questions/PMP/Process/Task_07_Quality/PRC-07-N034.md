# Question

An agile team is tightening its Definition of Done after several stories were marked complete but later needed fixes before release. The team wants the Definition of Done to hold each individual story to the quality bar that makes it releasable. Which of the following belong in that Definition of Done? (Choose three.)

## Options

- **A**: Code for the story has been peer reviewed and merged, with automated tests for the new behaviour passing in the build.
- **B**: The story meets the agreed non-functional criteria, such as accessibility and performance, that apply to a releasable increment.
- **C**: User-facing documentation and release notes affected by the story have been updated.
- **D**: The product owner has accepted the story's outcome as meeting its specific acceptance criteria.
- **E**: All regression testing for the entire release candidate has completed with zero open defects across the product.

## Answer

A, B, C

## Explanation

A Definition of Done is the team's shared, repeatable quality checklist applied to every story so each is genuinely releasable: peer-reviewed merged code with passing automated tests (A), conformance to the agreed non-functional criteria that apply to an increment (B), and updated user-facing documentation and release notes affected by the change (C). (D) describes acceptance criteria — the story-specific functional conditions the product owner confirms — which are distinct from the Definition of Done; a story can pass its acceptance criteria yet still violate the Definition of Done (e.g. no tests), so acceptance is not itself a DoD item. (E) sets a release-level, whole-product gate (full regression of the candidate, zero defects everywhere) that no single story can satisfy on its own; loading product-wide release gates into a per-story Definition of Done makes it impossible to ever call one story done. The discrimination here is between story-level done criteria (A, B, C) and the neighbouring concepts of acceptance criteria (D) and release readiness (E).
