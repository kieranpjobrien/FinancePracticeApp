# Question

A new team is migrating a government agency's website. Every page template must pass the same fixed acceptance standards for accessibility, security headers, and broken links. The standards themselves are already documented and agreed. The project manager's concern is that, across dozens of templates and an unfamiliar team, some checks will be performed inconsistently or skipped. Which approach best addresses that specific concern?

## Options

- **A**: Derive a quality checklist from the standards and have each template signed off against every checklist item before it is accepted.
- **B**: Schedule periodic quality audits during the migration to confirm the team is following the agreed process.
- **C**: Build the accessibility, security-header, and link checks into an automated test suite that runs against each template in the pipeline.
- **D**: Add the standards as explicit acceptance criteria on every template's user story so the team sees them when picking up the work.

## Answer

A

## Explanation

The stated concern is consistency of execution — that a known, repeatable set of checks gets applied to every one of many similar items without anything being skipped. A quality checklist exists precisely for this and gives a per-item, verifiable sign-off record (A). (D) makes the standards visible as acceptance criteria but visibility is not the gap; the team can see the criteria and still apply them unevenly, with no per-item verification trail. (C) is attractive and would help, but several of these standards (notably accessibility conformance) require human judgement and cannot be fully automated, so automation alone would leave gaps and is a heavier build than the concern warrants. (B) audits the process after the fact and samples it, so it detects inconsistency rather than preventing it template by template. The checklist is the lightest tool that directly guarantees each template is verified against each standard.
