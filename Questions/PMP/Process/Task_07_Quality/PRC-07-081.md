# Question

An agile team building an online insurance-quote tool is drafting a Definition of Done that every story must satisfy to be considered releasable. The team wants criteria that genuinely belong in a per-story Definition of Done, not items that sit elsewhere. Which of the following are appropriate to include? (Choose three.)

## Options

- **A**: Automated unit and integration tests for the story's behaviour are written and passing in the pipeline.
- **B**: The code has been peer reviewed and merged, and the static-analysis quality gate passes with no new violations.
- **C**: Any non-functional criteria that apply to a releasable increment — such as the agreed accessibility and security checks — are met for the story.
- **D**: The product owner has confirmed the story meets its own specific acceptance criteria for the requested behaviour.
- **E**: The full release has been signed off by the change advisory board and scheduled into the production deployment window.

## Answer

A, B, C

## Explanation

A Definition of Done is the team's standing, story-level quality checklist: passing automated tests for the story (A), peer-reviewed merged code clearing the static-analysis gate (B), and meeting the non-functional criteria that apply to an increment such as accessibility and security (C) all belong. (D) is acceptance criteria — the product owner's story-specific functional verification — which is a separate concept from the Definition of Done; a story meeting its acceptance criteria can still fail the DoD. (E) is a release-level governance gate (change advisory board sign-off, deployment scheduling) that applies to a whole release, not to whether one story is done; loading it into the DoD would make any single story impossible to complete. The discrimination is between standing per-story done criteria (A, B, C) and the neighbouring ideas of acceptance criteria (D) and release readiness (E).
