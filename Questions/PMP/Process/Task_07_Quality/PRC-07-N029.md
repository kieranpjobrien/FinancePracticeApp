# Question

An agile team delivering a mobile banking feature finds that manual regression testing at the end of each sprint takes two days and frequently slips past the sprint boundary, delaying the release. The regression set has grown organically and no automated coverage exists yet. The product owner is pressing for the cadence to hold and is reluctant to give up story capacity. The project manager wants a sustainable fix, not a one-sprint patch. What is the BEST approach for the project manager to support?

## Options

- **A**: Have the team reserve a slice of each sprint's capacity to automate regression tests, sequencing the build so the highest-risk and most-repeated checks are automated first and manual effort falls sprint over sprint.
- **B**: Charter a dedicated two-sprint automation initiative to build the full regression suite up front, pausing most feature delivery so the team returns to a clean, fully automated cadence as quickly as possible.
- **C**: Keep regression manual but move it to a continuous activity within the sprint — testing each story as it completes rather than batching at the end — so the work no longer collides with the sprint boundary.
- **D**: Add a one-sprint testing buffer after every few delivery sprints to absorb the regression run, keeping feature sprints on cadence while regression is done in the buffer.

## Answer

A

## Explanation

The principle is to build quality in so the increment stays shippable, and the growing manual regression burden clearly signals automation — but every option claims to address that, so the discrimination is *how* to get there under real constraints (a protective product owner, no existing automation, the need for a durable fix). Reserving steady capacity and automating risk-first lets coverage compound while feature delivery continues and manual effort shrinks each sprint — the sustainable path (A). (B) reaches the same end state but stops the value stream for two sprints and bets the whole fix on a big up-front build; in banking, halting delivery to tool up is rarely the proportionate choice and the product owner has signalled they will resist surrendering that much capacity. (C) is a genuine improvement — shifting from batched to continuous manual testing eases the boundary collision — but it leaves the work manual, so the burden keeps growing with the product and the relief is temporary, not structural. (D) protects feature-sprint optics but institutionalises a separate test phase and the very batching that caused the slip, which is the opposite of integrated quality. The fine margin is between incremental, risk-prioritised automation woven into each sprint (A) and three options that either over-invest up front, relieve the symptom without removing it, or re-batch the testing.
