# Question

A government client procured a deliberately basic, low-cost inventory system for a small warehouse, specifying only a handful of functions. Testing shows the system performs every contracted function reliably with no defects. At the acceptance walkthrough, the warehouse supervisor who will use it daily is visibly disappointed, saying it feels "bare-bones" next to tools they have seen elsewhere and questions whether it is good enough. The contract sponsor is satisfied. How should the project manager proceed?

## Options

- **A**: Proceed toward acceptance against the contracted requirements, and capture the supervisor's feature wishes as feedback to inform any future enhancement decision.
- **B**: Raise a change request to add several of the features the supervisor expects, so the end user is satisfied before handover.
- **C**: Hold acceptance and escalate the gap between the supervisor's expectations and the delivered system to the sponsor for a decision.
- **D**: Add a few low-effort enhancements within the team's remaining capacity to close the most visible gaps the supervisor raised.

## Answer

A

## Explanation

The system is low grade by design but high quality: it meets every agreed requirement without defects. The supervisor's disappointment is about grade (feature richness), which was a deliberate, funded procurement choice, not a quality shortfall. (A) completes acceptance on the agreed basis while properly logging the user's input for a future, funded decision. (B) and (D) both add scope the client never contracted — gold-plating — with (D) compounding it by doing so informally outside change control. (C) over-escalates: there is no actual conformance gap to resolve, only a difference between one user's expectations and the scope the client deliberately bought, so halting acceptance and routing it to the sponsor as a problem misrepresents a satisfied deliverable as a deficiency.
