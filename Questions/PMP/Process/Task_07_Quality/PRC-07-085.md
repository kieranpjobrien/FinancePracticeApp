# Question

An agile team building a telehealth platform that now handles regulated health data reports that several stories accepted as done in recent sprints later caused production incidents, each traced to a missing security check or an untested edge case. The product owner wants to protect this sprint's feature commitments. The project manager has to decide what to change so the same class of defect stops slipping through. Which course should the project manager prioritise?

## Options

- **A**: Work with the team to add the missing security and edge-case verification into the standard every story must meet to be called done, applying it to new work from now on.
- **B**: Stand up a dedicated security-and-edge-case review stage, run by a separate team, that every story must clear after the sprint before it can go to production.
- **C**: Keep the current done standard unchanged and raise each production incident as its own defect to be fixed in an upcoming sprint.
- **D**: Lengthen the sprint to give the team more time at the end to test stories more thoroughly before they are declared done.

## Answer

A

## Explanation

The stories were genuinely declared done yet escaped with a recurring class of defect, which means the bar for "done" no longer matches the quality now required for regulated health data — so the durable fix is systemic, not case-by-case. Building the missing verification into the standard every story must meet prevents the whole class at the point each story is completed. (A) is correct. (B) catches the same defects but reverts to a downstream inspection stage run by another team, leaving the team's own definition of done still inadequate and pushing verification later. (C) treats each escape as a one-off and never closes the gap that lets them through, so the pattern persists. (D) buys more end-of-sprint testing time without changing what done requires; the problem is the missing criteria, not the amount of time.
