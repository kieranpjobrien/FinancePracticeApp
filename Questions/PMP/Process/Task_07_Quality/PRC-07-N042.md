# Question

On a predictive firmware project, the same defect category has recurred across the last four builds. The team already runs a fishbone session after each occurrence and has a short list of plausible causes, but the agreed fix never seems to stop the category returning in the next build. Each defect is still closed before release, so management is content. The project manager is told to "just keep closing them." What should the project manager do next?

## Options

- **A**: Run a fresh cause-and-effect workshop on the latest occurrence to generate a wider set of candidate causes for the team to consider.
- **B**: Build a Pareto and trend analysis across the four builds to confirm which specific cause is actually driving the recurrence before committing to another fix.
- **C**: Implement the most likely fix from the existing fishbone list and verify in the next build whether the defect category disappears.
- **D**: Add a dedicated regression check for this defect category to each build so any recurrence is caught and corrected earlier.

## Answer

B

## Explanation

The team is not short of candidate causes; it keeps acting on plausible causes that turn out not to be the real one, so the recurrence continues. The missing step is using the data already accumulated across four builds to verify which cause is genuinely responsible before spending another fix cycle on a guess. (B) grounds the next fix in evidence rather than plausibility. (A) generates yet more hypotheses when the team already has a list it cannot confirm. (C) is the very pattern that has failed three times — picking the "most likely" cause without confirming it. (D) strengthens detection and earlier correction but still does nothing to stop the cause from producing the defect.
