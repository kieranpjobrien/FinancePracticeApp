# Question

On an agile project, stakeholders and team members keep asking for status the team already knows, and on a couple of occasions work has restarted on decisions that had in fact been settled because people did not realise. The project manager wants people to be able to get the current state for themselves, at the moment they need it, without routing every query through someone. What is the most fitting approach?

## Options

- **A**: Maintain always-visible displays of current state — a task board, a burndown, and a running decisions log — that anyone can consult whenever they need to.
- **B**: Send a thorough written status report to stakeholders and the team at the end of each sprint.
- **C**: Add a standing status slot to the daily stand-up where outstanding questions about progress and decisions are answered.
- **D**: Publish a status-and-decisions summary on a set schedule and notify everyone each time it is refreshed.

## Answer

A

## Explanation

The aim is for people to pull the current picture themselves the moment they need it, so the test is which option gives continuous, self-serve visibility rather than periodic or mediated updates. (A) keeps state and decisions on permanent display, so anyone can check at any time and stale assumptions are corrected on sight — directly killing the repeat questions and the reopened decisions. (B) pushes a snapshot that is already ageing the day after it lands and answers nothing in between sprints. (D) is more frequent than (B) and notifies on change, but it still delivers point-in-time summaries on a cadence and goes stale between refreshes rather than reflecting the live state. (C) routes status through a time-boxed event meant for coordination, reaching only those present and turning the stand-up into a reporting session. Continuous, always-available visibility beats every periodic or single-channel alternative.
