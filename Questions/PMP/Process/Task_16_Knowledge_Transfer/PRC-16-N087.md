# Question

On a hybrid insurance-claims modernisation project, team members keep emailing each other for configuration spreadsheets and decision records. Several near-identical versions are now circulating, people sometimes work from a superseded copy, and there is recurring confusion over which is current. The project manager wants to fix this so it does not keep happening. What is the most appropriate action?

## Options

- **A**: Set up a single shared repository as the agreed source of truth, with named ownership per artefact and version control so the current copy is always identifiable.
- **B**: Have the team consolidate the scattered copies into one master version of each artefact and delete the duplicates now circulating.
- **C**: Introduce a naming and versioning convention for the files so the latest version is obvious whenever one is emailed.
- **D**: Run a short weekly sync where the team confirms which version of each key artefact is current before the week's work.

## Answer

A

## Explanation

The discriminator is which action removes the *cause* — the absence of a single managed home — rather than tidying its effects. (A) establishes one authoritative source with ownership and version control, so divergence cannot recur and "which is current" is answered by design. (B) is a reasonable first cleanup and genuinely helps, but consolidating today does nothing to stop the same fragmentation re-forming tomorrow because files still live in inboxes. (C) a versioning convention improves how copies are labelled, yet multiple labelled copies still circulate by email and people can still grab the wrong one; it manages the symptom. (D) a weekly confirmation reduces confusion within the week but is recurring manual effort to reconcile copies that should not be diverging at all. The fine margin is the structural fix (A) against the partial fixes (B, C, D): only (A) gives the artefacts one controlled location.
