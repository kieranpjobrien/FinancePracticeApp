# Question

An agile team building an internal logistics tool has consciously kept documentation light to maximise delivery speed, in line with valuing working software over comprehensive documentation. As the product has grown, onboarding new team members now takes weeks, and two recent production incidents were traced to behaviour that only the original developers understood. The product owner still pushes for maximum feature throughput. What should the project manager do?

## Options

- **A**: Facilitate a conversation with the product owner and team to right-size documentation — capturing the knowledge that is proving costly to lack — and treat that work as part of delivering sustainable value.
- **B**: Add a comprehensive documentation phase to the release plan to bring all product documentation up to a complete standard.
- **C**: Defer to the product owner's prioritisation, since maximising feature throughput reflects the agile value of working software over documentation.
- **D**: Assign a technical writer to document the product so the development team's throughput is not affected.

## Answer

A

## Explanation

"Working software over comprehensive documentation" values documentation less, not at zero, and the missing knowledge is now causing incidents and slow onboarding — real costs to value delivery. (A) reframes targeted documentation as enabling sustainable value and brings the PO into a data-informed trade-off, which is the agile way to handle it. (B) over-corrects to "complete" documentation, the comprehensive extreme the manifesto warns against, and stalls delivery. (C) misreads the value as licence to skip documentation even when its absence is demonstrably harming the product. (D) outsources capture of deep tacit knowledge to a writer who lacks it, producing thin docs while leaving the developers' understanding untransferred.
