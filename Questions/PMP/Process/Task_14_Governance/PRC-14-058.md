# Question

A hybrid project to deploy smart meters is one of six projects in a utilities transformation programme. The project manager needs a decision to change a customer-data interface that this project shares with three of the other projects. The change is well within the project's own budget and schedule tolerances, and the project manager has the authority to make interface changes inside the project. How should the project manager pursue this decision?

## Options

- **A**: Take the decision within the project, since it sits inside the project's delegated cost and schedule tolerances and the project manager holds authority over its interfaces.
- **B**: Raise it through the programme-level governance, because the interface is a shared asset and changing it affects projects beyond this one's remit.
- **C**: Agree the change directly with the three affected project managers and proceed once all three concur, keeping the decision among those actually impacted.
- **D**: Make the change but notify the programme office and the three projects afterwards, so the work is not held up and everyone is kept informed.

## Answer

B

## Explanation

Project governance nests inside programme governance, and authority is bounded by scope as well as by tolerance: this project's interface authority and its cost/schedule tolerances cover its own delivery, not a component shared across several projects. Because the change touches an asset other projects depend on, the decision rises to the programme level, which holds the remit and oversight for shared components. (A) is the headline trap — the change is within tolerance and the project manager does own its interfaces, so it feels in-bounds, but neither fact extends the project's authority over a shared asset. (C) is the strongest distractor: consulting the impacted project managers is exactly the right instinct, yet a cluster of bilateral agreements is not the same as a programme-level decision — it lacks the authority and coordination to keep the shared interface coherent and can yield inconsistent commitments the programme later has to unpick. (D) acts first and informs after, which treats a cross-project change as a project-level call and bypasses the governance that should authorise it.
