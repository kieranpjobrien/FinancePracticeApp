# Question

An agile team releases working software to production every two weeks, but the programme's governance body that must approve any production release meets only once a quarter. As a result, completed increments sit undeployed for weeks, eroding the value of frequent delivery. The project manager raises this with the sponsor. What change to the governance structure best resolves the mismatch?

## Options

- **A**: Allow the team to deploy without approval and have the quarterly body review the releases retrospectively.
- **B**: Ask the quarterly governance body to meet every two weeks so it can approve each release as it is ready.
- **C**: Have the team batch its increments and release them all together once per quarter to align with the governance cadence.
- **D**: Delegate release-approval authority to a role available at the delivery cadence, within defined limits, so that releases inside those limits no longer wait for the quarterly body.

## Answer

D

## Explanation

The mismatch is between a fast delivery cadence and a slow approval cadence, and good governance resolves it by placing the routine release decision with an authority that operates at the delivery rhythm, bounded by clear limits, while genuinely significant releases still escalate (D). (B) forces a senior quarterly body into a fortnightly commitment it is not designed for, an inefficient use of that body. (C) sacrifices the core benefit of frequent delivery to fit an unsuitable cadence, solving the symptom by degrading the project. (A) removes the control entirely and reviews only after the fact, which abandons release governance rather than re-aligning it; retrospective review cannot prevent an unsafe release that has already gone live.
