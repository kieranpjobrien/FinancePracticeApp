# Question

On a predictive software release, an approved defect fix changed a configuration item that three other modules share. A week after deployment, one of those dependent modules starts failing intermittently in production; the fix itself is correct in isolation, was logged, and the failure is now affecting users. The project manager has confirmed the fix is the cause. What should the project manager do first?

## Options

- **A**: Raise a new change request to address the dependent module's failure and route it through the change control board for assessment and approval.
- **B**: Have the team determine which configuration items depend on the changed item and how each is affected, then bring that analysis into the change response.
- **C**: Invoke the change management plan's emergency change procedure so the dependent module can be restored quickly under expedited authority.
- **D**: Update the configuration management records to capture the newly discovered dependency between the changed item and the failing module.

## Answer

B

## Explanation

A correct fix has broken a module that shares the configuration item, which means the dependency relationships were not fully understood when the change was assessed; before the project manager can choose a sound remedy or scope a change request, they need to know the actual blast radius across all items that depend on the changed one. (B) produces that understanding and feeds it into the response, which is why it precedes the others. (A) is a valid and likely next step, but raising and routing a change request before the dependency impact is mapped risks scoping the fix to only the one module that happened to fail first while the other two dependents remain exposed. (C) presumes an emergency is warranted and an emergency procedure applies; an intermittent fault is serious but the scenario does not establish the outage severity or the procedure's existence, and reaching for expedited authority before understanding the dependencies can rush an equally incomplete fix. (D) is genuinely necessary housekeeping and should happen, but updating records documents the gap rather than resolving the live production impact, so it is not the first move. Understanding the dependency impact across the shared configuration item is what makes every subsequent action correctly scoped.
