# Question

On a predictive project, a team member proposes a genuinely useful improvement to a reporting screen. The screen was built and demonstrated, but it sits within a module whose scope baseline has not yet been formally accepted by the customer, who is reviewing it next week. The project manager agrees the improvement has merit and that it is small. What should the project manager do?

## Options

- **A**: Raise it as a change request and run it through impact assessment and the change control process before the improvement is built, because the reporting screen is part of work governed by the scope baseline.
- **B**: Fold the improvement into the module now while it is still being finished and the baseline is not yet customer-accepted, then present the improved screen at next week's review for acceptance.
- **C**: Note the improvement and raise it as a change request only if the customer accepts the module as-is next week, since until then the screen may still be revised under normal completion work.
- **D**: Authorise the improvement directly as a minor refinement within the team's discretion and record it in the change log so there is a trail of what was altered.

## Answer

A

## Explanation

The reporting screen is part of work governed by the approved scope baseline, so a deliberate change to its agreed content goes through change control and impact assessment regardless of whether the customer has formally accepted that module yet; "not yet accepted" is not the same as "not yet baselined". (A) routes it correctly. (B) is the strong trap: it exploits the fact that the module is still open to slip a scope change in as ordinary completion work, but adding new agreed functionality is a change to baselined scope, not finishing what was already specified, and bouncing it straight into next week's acceptance hides the change from assessment. (C) makes the need for change control hinge on the customer's acceptance, when the baseline already governs the work today; it defers a decision that the current baseline already settles. (D) treats the change log as a substitute for assessment and authority, letting the PM self-approve a baseline change so long as it is written down, which records the change without controlling it. The margin is that (B), (C) and (D) all lean on the module's "still in progress" status to avoid the process, but the baseline governs the moment the work is agreed, not the moment the customer signs.
