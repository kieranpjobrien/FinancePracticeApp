# Question

A hybrid programme runs an agile build team alongside a predictive hardware workstream under a shared schedule. A stakeholder change request affects both at once: it reprioritises a software feature and requires a hardware design alteration, and the software feature cannot be demonstrated until the altered hardware is available. The product owner can reprioritise the feature into an upcoming sprint within days; the hardware alteration must clear formal change control, which will take longer and may not be approved as requested. How should the project manager handle the request?

## Options

- **A**: Have the product owner reprioritise and start the software feature now so it is ready the moment the hardware lands, while the hardware alteration proceeds through formal change control in parallel.
- **B**: Route the hardware alteration through change control first and hold the software reprioritisation until the hardware change is approved, so the product owner commits sprint capacity only to work that can actually be realised.
- **C**: Process the whole request through the change control board so both halves are assessed together against the shared schedule and approved or rejected as one decision.
- **D**: Have the product owner reprioritise the software feature and, to keep the workstreams aligned, ask the hardware lead to begin the design alteration ahead of formal approval given the stakeholder backing.

## Answer

B

## Explanation

Each workstream is rightly governed by the mechanism that fits it, the software by the product owner's backlog and the hardware by formal change control, and all four options respect that split in some form, so the discriminator is how the dependency between them is sequenced. The hardware alteration is the gating uncertainty: it takes longer and may not be approved as requested, and the software feature delivers no demonstrable value until that hardware exists. (B) sequences to that reality, settling the constraining decision before committing sprint capacity to work that cannot yet be realised. (A) feels efficient and the parallelism is tempting, but it has the product owner pull a feature forward whose only value depends on a hardware change that may be altered or rejected, risking sprint capacity spent on a feature that then has to wait or be reworked when the hardware decision lands differently. (C) collapses the agile half into the board's cadence, forcing the product owner's fast, reversible reprioritisation to wait on predictive governance it does not need and undermining the product owner's ownership of sprint scope. (D) keeps the timelines aligned only by letting hardware design start before its change is authorised, which is exactly the uncontrolled commitment formal change control exists to prevent on the predictive side. Letting the gating, slower-to-approve hardware change set the pace, while still governing each half by its own mechanism, is the tailored response.
