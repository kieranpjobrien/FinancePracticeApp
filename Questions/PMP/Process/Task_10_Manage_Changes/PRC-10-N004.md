# Question

A stakeholder on an agile mobile-app project is frustrated: "Every feature tweak I ask for goes onto a change request form and waits for a board. I thought agile welcomed change." The team runs Scrum with an engaged product owner and a healthy backlog. The same project also has a fixed delivery budget and a vendor contract governed by a change control board. How should the project manager address the underlying issue?

## Options

- **A**: Route ordinary feature requests to the product owner for backlog prioritisation, and reserve the change control board for matters that affect the budget or the vendor contract.
- **B**: Send every request to the product owner for the backlog and retire the change control board, since the product owner now owns what the project delivers.
- **C**: Keep all requests, including small feature tweaks, going to the change control board, but commit to a faster turnaround so the stakeholder waits less.
- **D**: Let the stakeholder bring feature requests straight to the product owner, and have the product owner forward anything with cost implications to the board on the stakeholder's behalf.

## Answer

A

## Explanation

Agile absorbs evolving feature requirements through the prioritised backlog, while genuine baseline matters such as funding and contractual scope still require formal change control; the fix is to match each kind of change to the right mechanism. (A) does exactly that. (B) correctly frees feature tweaks but disbands the board entirely, removing governance the budget and vendor contract still need. (C) speeds up the board but keeps routing ordinary backlog items through it, leaving the root cause, heavyweight ceremony on team-level work, in place. (D) sends features to the product owner but makes the product owner the conduit for budget and contract changes, blurring the line between product prioritisation and the authority that actually owns those baselines. Each change type should sit with the body that owns it.
