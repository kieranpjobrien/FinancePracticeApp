# Question

A business sponsor on a predictive ERP rollout keeps emailing individual developers small "tweaks" directly, and several have been quietly implemented. The project manager notices the build no longer matches the approved scope baseline and the team is behind. What should the project manager do?

## Options

- **A**: Instruct developers to roll back every undocumented tweak immediately to restore the approved baseline before doing anything else.
- **B**: Accept the implemented tweaks as the new baseline since they came from the sponsor, and update the documentation to match.
- **C**: Meet the sponsor to explain the change control process, then bring the undocumented changes back through assessment and the change control board to reconcile the baseline.
- **D**: Agree a protocol with the sponsor that all future requests come through the project manager, and let the tweaks already built stand so the team can focus on catching up.

## Answer

C

## Explanation

Uncontrolled changes are addressed by reinforcing the process with the sponsor and routing the changes that were already made back through assessment and the board, so the baseline once again reflects approved reality. (C) fixes both the behaviour and the integrity of the artefacts without discarding potentially valuable work. (A) blanket-reverts before anyone understands the impact, destroying work that may well be wanted. (B) rubber-stamps scope creep and abandons change control, however senior the source. (D) is the closest trap: stopping future direct tweaks is right, but leaving the already-built changes unassessed means the live build still silently diverges from the approved baseline, so the configuration is never reconciled.
