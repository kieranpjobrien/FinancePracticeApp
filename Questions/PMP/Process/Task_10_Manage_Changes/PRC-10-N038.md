# Question

A production outage on a live system needs an urgent fix that will alter a baselined component. The change management plan contains a defined emergency change procedure with a designated emergency approver who is currently reachable. A senior engineer is standing by and confident in the fix. What should the project manager do?

## Options

- **A**: Authorise the engineer to apply the fix now to end the outage, then log it and complete the change documentation once service is restored.
- **B**: Obtain the designated emergency approver's authorisation, have the fix applied, then log it and complete the documentation retrospectively.
- **C**: Convene an expedited virtual change control board to approve the fix before the engineer applies it.
- **D**: Apply the fix under the emergency procedure and capture full impact analysis and baseline updates before service is restored.

## Answer

B

## Explanation

The plan already defines an emergency path with a named approver who is available, so the correct move is to use exactly that path: get the designated emergency approval, apply the fix, and complete the standard documentation afterward. (B) honours the expedited governance the organisation pre-authorised for this situation. (A) is tempting under time pressure and gets the documentation timing right, but it skips the emergency authorisation step the plan requires, letting an engineer change a baseline on their own confidence. (C) ignores that an emergency procedure exists precisely so the team need not wait to assemble a board, even a virtual one, while a system is down. (D) misorders the work by insisting on full impact analysis and baseline updates before restoring service, which defeats the purpose of an emergency response; that documentation is completed retrospectively. Following the pre-defined emergency approval balances speed against accountability.
