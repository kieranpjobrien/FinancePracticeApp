# Question

An operations team uses Scrum but routinely fails to meet its sprint commitments because production incidents and stakeholder requests interrupt the team almost daily. What should the team consider?

## Options

- **A**: Add a "buffer" story to every sprint so unplanned work has somewhere to land.
- **B**: Move to Kanban with explicit WIP limits, since flow handles unplanned work better than sprint commitments.
- **C**: Run a more rigorous sprint planning to lock the sprint scope and decline interruptions.
- **D**: Extend sprint length to four weeks so interruptions are absorbed within a single iteration.

## Answer

B

## Explanation

Scrum's commitment model assumes the team's capacity for the iteration is mostly knowable; that assumption breaks for an interrupt-driven team. (B) matches the framework to the work — flow with WIP limits acknowledges variability while controlling overload. (A) hides the problem inside a planning fudge factor. (C) rejects the operational reality and harms stakeholder trust. (D) makes the same assumption Scrum already makes; longer sprints will still be interrupted.
