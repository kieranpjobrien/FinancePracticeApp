# Question

An agritech firm is building a sensor-and-software platform for crop monitoring on an adaptive lifecycle. Growers are willing to pay for partial functionality and want something useful in the field within the first few months. At the same time, the analytics models at the platform's core are giving inconsistent results, and the data scientists say several rounds of trial and rework will be needed before the models can be trusted. The product owner wants the team's first releasable increment to be a genuinely useful one. How should the project manager sequence the early work?

## Options

- **A**: Ship a thin first increment built on the current models now to meet the growers' timeline, and continue refining the models in the increments that follow.
- **B**: Spend the opening iterations refining the analytics models to an acceptable confidence level, then start releasing useful increments to growers from that base.
- **C**: Carve the platform into increments by feature area and release them in priority order, treating the model issues as defects to be fixed within each increment as it ships.
- **D**: Pause grower releases and run a separate research track to perfect the models, rejoining incremental delivery only once they are fully proven.

## Answer

B

## Explanation

On an adaptive lifecycle the team can both iterate (refine poorly understood work through repetition) and release increments (deliver usable value early) — but when the core is unreliable, the order matters. (B) uses the first iterations to lift the models to a usable confidence level, then ships increments growers can actually trust, which is what "a genuinely useful first increment" requires; a few months easily accommodates this. (A) hits the date but releases capability built on models the data scientists say are not yet reliable, risking a first increment that misleads growers and damages credibility — speed bought at the cost of usefulness. (C) reframes fundamental model uncertainty as ordinary defects to be patched per feature, underestimating that the analytics underpin every increment, not one feature area. (D) over-corrects into a fully separate research project, abandoning early delivery the growers value and the lifecycle is meant to provide. The discriminator is sequencing iteration before increment when the shared core is unreliable — not whether to combine the two, but which comes first so the early value is real.
