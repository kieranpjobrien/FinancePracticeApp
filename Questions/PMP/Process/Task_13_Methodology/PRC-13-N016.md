# Question

A bank's digital transformation spans roughly two hundred practitioners across twelve teams and several value streams, with strong dependencies on enterprise architecture and compliance. Most teams already run Scrum well. Leadership wants strategy-to-execution alignment and coordinated cross-team planning, and has accepted that an enterprise scaling framework is needed. The project manager must decide how to introduce it. What is the BEST first step?

## Options

- **A**: Roll the chosen scaling framework out to all twelve teams simultaneously so the whole transformation adopts a single synchronised cadence from the outset.
- **B**: Pilot the framework's cross-team planning and integration cadence with a few interdependent teams in one value stream, then extend it based on what is learned.
- **C**: Adopt only the framework's portfolio and governance layer first so leadership gets strategy-to-execution visibility, leaving team-level practices unchanged for now.
- **D**: Have each team tailor its own variant of the framework locally so adoption fits each team's context before any cross-team mechanisms are introduced.

## Answer

B

## Explanation

Introducing a scaling framework is itself a significant change best approached incrementally: piloting the cross-team cadence with a cluster of genuinely interdependent teams targets the actual pain — coordination and alignment — while generating evidence to guide a wider rollout. (B) does this. (A) is a big-bang transformation across 200 people at once, maximising disruption and risk before anyone has validated how the framework behaves here. (C) installs the governance layer but defers the cross-team planning and integration mechanics that the dependencies actually require, so the alignment problem persists. (D) lets each team build a different local variant, which directly undermines the cross-team synchronisation and shared cadence that scaling is meant to provide.
