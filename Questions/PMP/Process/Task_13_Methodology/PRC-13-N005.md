# Question

A newly formed agile team is delivering a customer analytics dashboard. Across the first two sprints, stories accepted at the review repeatedly broke in the demo, and each time the cause traced back to a different gap — sometimes missing automated tests, sometimes unhandled error cases that nobody had agreed were in scope. The team is fast but inconsistent about what "finished" includes, and the gaps vary story to story. At the retrospective the team agrees their way of working must change. Which adjustment most directly addresses the root cause?

## Options

- **A**: Add a dedicated hardening sprint after every few delivery sprints to find and fix the defects before they reach stakeholders.
- **B**: Have a senior engineer technically review and sign off each story's implementation before it is accepted.
- **C**: Agree and enforce a team definition of done that fixes what every story must include — automated tests and error handling among them — before it can be called complete.
- **D**: Write more detailed acceptance criteria on each individual story so the specific tests and error cases for that story are spelled out before it is built.

## Answer

C

## Explanation

The escapes are not random: each is a piece of work the team never agreed was part of "finished", and the gaps differ story to story precisely because there is no shared, default standard — which is exactly what a definition of done supplies, applied uniformly to every story without re-deciding each time (C). (D) is the strongest distractor and genuinely helps — per-story acceptance criteria can name the tests and error cases — but they must be remembered and written correctly on each story, so the same omissions recur whenever someone forgets, whereas the DoD covers every story by default. (A) batches defects into a later sprint, delaying feedback and contradicting the principle that each increment should already be done. (B) can catch problems but routes every story through one reviewer as a bottleneck and still leaves "done" undefined, so the team keeps disagreeing about what sign-off should require.
