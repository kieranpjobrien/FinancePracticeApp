# Question

A government agency is procuring a new case-management system. Its procurement office must be able to point to defined milestones and demonstrate value for money to auditors, and it wants one contract rather than a string of small awards. The delivery team has told the agency that the detailed requirements will only become clear as the system is built and tested with caseworkers, so much of the specification cannot responsibly be written before work starts. Which contracting structure best reconciles these pressures?

## Options

- **A**: A contract built on fixed high-level outcomes and incremental milestone-based payments, with detailed scope elaborated iteratively within each increment.
- **B**: A firm-fixed-price contract against a complete, detailed requirements specification agreed before any build begins.
- **C**: An uncapped time-and-materials contract with the team free to deliver whatever the backlog surfaces.
- **D**: A framework that defines the full scope up front but releases it as fixed-price work packages the agency approves one at a time.

## Answer

A

## Explanation

A contract that fixes high-level outcomes and ties payment to incremental milestones, while letting detailed scope be elaborated iteratively, gives the procurement office the defined milestones and demonstrable value for money its auditors need and still lets the team discover detail during build, which is the essence of agile procurement in a governed environment (A). A firm-fixed-price contract against a complete up-front specification (B) honours governance but forces exactly the early requirement-fixing the team says is not responsible, inviting heavy change traffic and waste. Uncapped time-and-materials (C) fits the emergent build but offers no defined milestones and no ceiling to evidence value for money, so it would struggle against the agency's accountability rules. A framework of approved fixed-price work packages (D) is closer, and its staged approvals help governance, but defining the full scope up front and pricing each package firm still pins down detail the team cannot yet fix, so requirements churn reappears inside every package. The fine margin is that B fixes scope too early, C abandons milestones and cost control, and D smuggles the same up-front scope-fixing into staged packages, whereas only A keeps milestones and accountability while leaving detailed scope genuinely emergent.
