# Question

A Scrum team has just brought in an external supplier to build a specialised module that must integrate with the team's product each sprint. In the first sprint review, the supplier's work technically functioned but failed the team's automated quality gates and was not documented to the team's standard, so it could not be accepted into the increment. What should the project manager do to prevent this recurring?

## Options

- **A**: Have the internal team rework the supplier's output each sprint to bring it up to standard before integration.
- **B**: Agree a shared definition of done with the supplier so their deliverables meet the same acceptance and quality standards as the team's.
- **C**: Extend the supplier's sprint length so they have more time to finish to the required quality.
- **D**: Move the supplier's work to a separate release track so it no longer has to meet the team's sprint quality gates.

## Answer

B

## Explanation

When an external supplier contributes to a team's increment, aligning on a shared definition of done makes the quality and acceptance standards explicit and binding, which prevents the recurring rejection (B). Having the internal team rework the output each sprint (A) hides the problem, wastes capacity, and never fixes the root cause. Lengthening the supplier's sprint (C) addresses time, but the failure was about standards, not duration. Splitting the supplier onto a separate release track (D) breaks the integration the module is meant to provide. A shared definition of done sets the bar everyone builds to.
