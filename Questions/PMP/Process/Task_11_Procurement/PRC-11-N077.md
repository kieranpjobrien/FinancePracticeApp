# Question

A startup is bringing in an external agency to help build a new web application. The startup releases to users almost weekly and reshapes what it builds next based on how those users behave, so what the agency works on is expected to shift often and cannot be listed out in advance with any confidence. The startup wants the engagement set up so the agency can simply pick up the current top priorities each sprint. Which contract type best supports this way of working?

## Options

- **A**: Firm fixed price against a fixed, agreed scope of features.
- **B**: Fixed-price incentive fee with a target cost and sharing ratio.
- **C**: Time and materials, paid on actual effort as priorities evolve.
- **D**: A firm fixed price per individual feature, agreed before each feature starts.

## Answer

C

## Explanation

When requirements are emergent and the buyer wants the agency to follow changing priorities each sprint, a time-and-materials contract fits because it pays for effort actually expended and does not tie the work to a predetermined deliverable list, so scope can flex freely (C). Firm fixed price against a fixed scope (A) is the opposite of what is needed, since fixing the price requires fixing the features and would penalise the very change the startup expects. Fixed-price incentive fee (B) still rests on a defined target scope and cost to incentivise against, which an emergent backlog cannot supply, so the incentive machinery has nothing stable to measure. Pricing each feature firm-fixed before it starts (D) injects a re-scoping and re-pricing negotiation in front of every feature, adding friction that works against fluid sprint-by-sprint reprioritisation. The fine margin is that A, B, and D all depend on some fixed scope to function, whereas only time and materials accommodates continuously changing priorities without that dependency.
