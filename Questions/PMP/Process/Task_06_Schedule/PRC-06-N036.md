# Question

A project manager has a first-pass critical path method schedule for a predictive office-relocation project: the activity list, dependencies, and three-point duration estimates with reserves are all in, and the network calculation has produced a set of early and late dates. The dates currently assume every named resource is available continuously and that work proceeds on calendar days. Before publishing this as the schedule baseline, which refinements genuinely make the dates more realistic and the critical path more defensible? (Select all that apply.)

## Options

- **A**: Apply the resource calendars and confirmed assignments and level any over-allocations, then recompute, accepting that the critical path may shift onto a resource-constrained chain.
- **B**: Replace the calendar-day assumption with the project working-time calendar so weekends, holidays and shift patterns push out the finish dates accordingly.
- **C**: Enter the charter's imposed completion date and contractual milestone dates as constraints, then check the calculated dates against them and treat any negative float as a recovery problem to solve before baselining.
- **D**: Pull the early-start dates forward to their optimistic three-point values so the published baseline shows the most ambitious achievable finish and motivates the team.
- **E**: Set every activity to an as-late-as-possible constraint so the schedule preserves maximum float and defers commitment of resources until the latest moment.

## Answer

A, B, C

## Explanation

Turning a raw network calculation into a baseline means making the dates resource-feasible, calendar-accurate, and reconciled with the dates the project is actually committed to, all of which can move the critical path and must be settled before publishing. (A) levelling against real resource availability is exactly what exposes a resource-constrained critical path and prevents a baseline that assumes impossible parallel work. (B) the working-time calendar converts duration into real elapsed dates; leaving it on calendar days understates the finish. (C) imposed and milestone dates enter as constraints, and any resulting negative float is a genuine feasibility problem to resolve, not to baseline over. (D) baselining to the optimistic estimates discards the most-likely basis and the reserves, setting a plan the team is expected to miss from day one, the opposite of a realistic baseline. (E) blanket as-late-as-possible constraints strip the schedule's float and create a brittle plan with no slack to absorb variation, which is a scheduling anti-pattern, not a feasibility refinement.
