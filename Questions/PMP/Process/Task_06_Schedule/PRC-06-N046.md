# Question

While sequencing a data-centre migration, the project manager finds the schedule lets "Decommission legacy servers" begin before "Confirm all applications migrated successfully" finishes. The team wants to start decommissioning early to free rack space and is confident the migration will succeed. The migrated environment has no verified rollback once the legacy servers are wiped. How should the project manager sequence these two activities?

## Options

- **A**: Keep a mandatory finish-to-start dependency from migration confirmation to decommissioning, since wiping the only fallback before confirmation is an irreversible risk.
- **B**: Use a finish-to-start relationship with a negative lag so decommissioning can begin a few days before confirmation completes, overlapping the low-risk early servers.
- **C**: Use a start-to-start relationship so decommissioning of already-verified servers proceeds in parallel with confirmation of the remainder.
- **D**: Keep the finish-to-start dependency but split decommissioning into per-application activities, each starting once its own migration is individually confirmed.

## Answer

A

## Explanation

The decisive fact is that there is no verified rollback: once legacy servers are wiped, a migration defect is unrecoverable, which makes confirmation a hard (mandatory) predecessor for the whole decommissioning activity (A). (B) introduces a lead that starts destroying the fallback before confirmation finishes — attractive for schedule, but it reintroduces exactly the irreversible exposure the confirmation step exists to close. (C) overlaps the two via start-to-start, which is sound logic only if "confirmation" were truly per-server complete at start; as written, confirmation covers all applications, so parallel wiping again pre-empts the safety gate. (D) is the most tempting alternative and is good practice in principle, but it silently re-scopes a single confirmation milestone into many independent ones; absent evidence the project can confirm and certify migration application-by-application, it assumes a granularity the scenario does not establish, whereas (A) is correct under the stated single-confirmation, no-rollback conditions.
