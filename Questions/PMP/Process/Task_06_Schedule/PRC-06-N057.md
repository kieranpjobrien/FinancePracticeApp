# Question

A project manager is developing the schedule model for a predictive bridge-strengthening project and wants the resulting baseline to be realistic and defensible. Which inputs and actions genuinely strengthen the schedule before it is baselined? (Choose all that apply.)

## Options

- **A**: Sequencing activities with explicit dependency types and any required leads or lags, rather than a simple ordered list.
- **B**: Incorporating resource availability calendars so the model reflects when crews and equipment can actually work.
- **C**: Running a schedule risk analysis over duration uncertainty to express the finish date as a confidence level.
- **D**: Setting each activity's duration by working back from the contractual finish date so the network resolves to the required end date without negative float.
- **E**: Pooling the duration uncertainty into a single contingency reserve held by the project manager and released through a defined process, rather than buffering each activity privately.

## Answer

A, B, C, E

## Explanation

The discriminator is that one tempting option reverses the planning logic. (A) dependency logic with leads/lags, (B) resource calendars, and (C) schedule risk analysis are core, defensible schedule-development inputs that make the model feasible and its finish date credible. (E) is correct because aggregating uncertainty into one managed, transparently released reserve is exactly how contingency should be handled — it removes the hidden padding that otherwise inflates and corrupts the baseline. (D) is the trap: deriving durations by working backward from the deadline is date-driven (reverse-engineered) planning — it forces activity estimates to fit a predetermined end rather than building the finish up from real effort and logic, so it manufactures a baseline that "meets" the date on paper while hiding the true schedule risk. Comparing the *logically built* schedule against the contractual date and then compressing where needed is legitimate; setting the durations to back-solve to the date is not.
