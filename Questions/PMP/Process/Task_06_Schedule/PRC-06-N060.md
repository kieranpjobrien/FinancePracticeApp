# Question

Two months into a predictive project, a change request to add a significant new deliverable has been approved by the change control board. The new work extends several activities and pushes the planned finish out by three weeks. The project manager has already entered the revised durations and logic into the schedule tool. How should the project manager handle the schedule baseline?

## Options

- **A**: Establish a new approved schedule baseline incorporating the change and retain the previous baseline version as a superseded record, so variance is now measured against the current approved plan.
- **B**: Keep the original baseline as the measurement reference and track the change's three-week impact separately as a known variance, so the project's performance against its initial commitment stays visible.
- **C**: Re-baseline only the activities affected by the change, leaving the unaffected portions of the original baseline in place to preserve the most performance history.
- **D**: Defer re-baselining until the change's added work actually begins, so the new baseline reflects confirmed rather than planned dates for the new deliverable.

## Answer

A

## Explanation

All four are plausible baseline-handling positions a PM might hear argued, and none discards data or stops tracking — the margin is what configuration management prescribes once a change is *approved through the CCB*. (A) is correct: an approved change is the trigger to set a new, fully integrated baseline while archiving the prior version for traceability, so future earned-value variance is measured against the currently approved plan rather than a superseded one. (B) preserves visibility of the original commitment but, by continuing to measure against an obsolete reference, it generates variance that conflates approved scope growth with genuine performance problems — the original target is better kept as historical context, not the live baseline. (C) sounds efficient but a partial re-baseline fractures the schedule into mixed references, breaking the integrity of network calculations and EVM, which require a single coherent baseline. (D) confuses baselining with progress: a baseline is the approved *plan* to measure against, so waiting for work to start leaves the project with no valid reference for the new deliverable in the interim.
