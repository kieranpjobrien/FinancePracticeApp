# Question

Eight sprints into a twelve-sprint release, the product owner has been adding stories to the release scope at roughly the rate the team completes them. Velocity is stable, yet the release feels no closer to done. A stakeholder asks the project manager to show whether the release will actually finish on time and why it seems stuck. Which artefact best reveals what is happening and supports the conversation?

## Options

- **A**: A release burndown chart, plotting remaining work for the release down toward zero.
- **B**: A release burn-up chart, plotting completed work and total scope as two separate lines.
- **C**: A cumulative flow diagram for the release, showing the quantity of work in each workflow state over time.
- **D**: A velocity trend chart for the eight completed sprints, showing points delivered each sprint.

## Answer

B

## Explanation

The defining symptom is scope rising at the same rate as completion, so the artefact must expose the scope line itself. (B) a burn-up plots completed work and total scope separately, making the climbing scope ceiling visible against steady progress, which is exactly the dynamic and explains why the gap will not close. (A) a release burndown nets scope changes into the single remaining-work line, so added scope flattens the line and disguises the very cause the stakeholder is asking about, the classic trap of burndown under changing scope. (C) a cumulative flow diagram reveals bottlenecks and WIP but not release-level scope growth against completion. (D) the velocity trend confirms the team is steady, which the scenario already states, and cannot show the moving scope line. Only the burn-up separates scope from progress.
