# Question

A predictive data-centre fit-out is two weeks behind on the critical path. The sponsor has authorised additional budget and asked the project manager to crash the project. Several critical activities can be accelerated by adding crews or shifts. Which considerations should genuinely drive which activities to crash? (Select all that apply.)

## Options

- **A**: The cost slope of each candidate activity, choosing those that buy the most time saved per additional dollar spent.
- **B**: Whether crashing a given critical activity could raise a parallel near-critical path to become the new constraint, capping the recoverable time.
- **C**: Whether the activity can actually absorb added resources productively, since some work does not go faster with more people.
- **D**: The amount of total float on each candidate activity, preferring to crash those with the most float so the change is lowest-risk.
- **E**: The number of separate activities on the critical path, crashing as many as possible so the saving is spread thinly across them.

## Answer

A, B, C

## Explanation

Effective crashing is driven by economics, network behaviour, and feasibility. (A) the cost slope ranks activities by cost per time saved, the core selection criterion. (B) compressing one critical path can expose a parallel near-critical path as the new limit, so the project manager must re-check the network and not over-spend past that ceiling. (C) some activities cannot be sped up by adding people (divisible-work limits), so crashability is a real constraint. (D) is wrong: by definition critical activities have zero total float, and float is the lever for resource smoothing, not for selecting what to crash. (E) spreading crash effort across many activities for its own sake ignores cost slope and may waste budget on low-yield compression. Cost, network re-analysis, and feasibility govern crashing, not float or activity count.
