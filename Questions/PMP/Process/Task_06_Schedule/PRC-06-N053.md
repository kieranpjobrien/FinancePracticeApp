# Question

A project manager is launching a hybrid programme with two workstreams: a regulatory reporting module whose requirements are fixed and well understood, and a customer-facing portal whose features are still being discovered with users. The portal must consume data the reporting module produces. Leadership wants "one consistent way of scheduling" across the programme. How should the project manager set up scheduling?

## Options

- **A**: Use a baselined predictive schedule for the regulatory module and an iterative, backlog-driven cadence for the portal, integrating them at shared milestones and the data hand-off.
- **B**: Run both workstreams on an iterative cadence but hold the regulatory module's scope and dates fixed within it, so the programme has a single rhythm leadership can follow.
- **C**: Schedule the regulatory module predictively and time-box the portal, but front-load the portal's data-dependent features so the integration risk with the fixed module is resolved early.
- **D**: Tailor each workstream to its nature and manage the dependency through a rolling integration plan rather than fixed milestones, since the portal's needs will keep shifting.

## Answer

A

## Explanation

These are variations on the same correct instinct — fit the approach to the work — so the margin is the cleanest, most defensible structure for stable scope feeding emergent scope. (A) tailors each workstream appropriately and gives leadership the integration points (shared milestones plus the data hand-off) they need to see one coherent programme. (B) keeps a single rhythm by wrapping the regulatory module inside an iterative cadence; this can work, but imposing iteration ceremony on fixed, well-understood scope adds overhead without benefit and is weaker than letting it run predictively. (C) is sound sequencing advice and a good complement, but "front-load the portal's data features" can conflict with discovering portal requirements with users; it optimises integration risk at the possible expense of the portal's emergent nature, so it is a refinement rather than the primary set-up. (D) tailors correctly but discards fixed integration milestones; even with an evolving portal, the data hand-off to a fixed regulatory module is a concrete commitment best anchored to a milestone, so wholly rolling integration is looser than needed.
