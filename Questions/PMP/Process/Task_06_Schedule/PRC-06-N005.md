# Question

A project manager inherits a predictive office-relocation project where, over several months, the schedule was repeatedly adjusted in status meetings to reflect new realities, but none of the changes went through change control and the baseline was never updated. The team now reports progress against this drifted schedule, and the sponsor cannot tell whether the project is ahead or behind. What should the project manager do first?

## Options

- **A**: Re-baseline the current working schedule as it stands through change control, so the team has an approved reference from today onward.
- **B**: Restore the original approved baseline and measure current progress against it to expose the full extent of the slip.
- **C**: Define and agree schedule variance thresholds with the sponsor so future reporting flags meaningful deviations.
- **D**: Reconstruct which of the informal adjustments were legitimate approved changes versus uncontrolled drift, then re-baseline only the justified scope and dates through change control.

## Answer

D

## Explanation

A reliable baseline must be both approved and defensible, so before re-baselining the project manager has to know what the drift actually represents. (D) separates real, justified changes from uncontrolled drift and re-baselines on a sound footing, which is the prerequisite for any meaningful variance. (A) rubber-stamps a schedule that may embed unjustified slippage, baking the problem into the new reference. (B) reverting to the original baseline ignores legitimate changes that genuinely occurred and produces a misleading variance the other way. (C) agreeing thresholds is useful governance but pointless until there is a trustworthy baseline to measure against. Establishing what the baseline should contain comes first.
