# Question

A predictive product-development schedule has two parallel paths, P1 and P2, that both call on the sole specialist test rig during the same three-week window; the network was built assuming the rig was always free. P1's run of activities sets the project's finish date and is the longer chain. P2 reaches the same merge point with its work but finishes its chain well ahead of the merge, then sits idle until the paths join. The completion date is firm, and the project manager has just spotted the rig double-booking. What is the most appropriate way to resolve it?

## Options

- **A**: Level the rig-dependent activities into the rig's single-unit availability and accept whatever extension to the finish date that forces.
- **B**: Reschedule P2's rig activity later within the slack it already has before the merge, so it no longer overlaps P1's rig activity and the finish date is untouched.
- **C**: Submit a change request to lease a second rig so both paths run exactly as originally scheduled.
- **D**: Leave the network as planned and have the two activity leads work out a day-by-day rig-sharing rota between themselves.

## Answer

B

## Explanation

P2 finishing its chain well before the merge means it carries slack that P1 does not, and that slack is what makes a no-cost fix possible. (B) is resource smoothing: shifting P2's rig activity later within the gap it already has removes the contention while leaving the firm finish date intact, the least disruptive valid fix. (A) resource levelling would work but is heavier than needed and lets the schedule extend past a date the slack on P2 could have protected. (C) leasing a second rig solves it but adds cost and procurement lead time for a clash the existing slack already absorbs. (D) an informal rota leaves a known hard constraint to chance and produces no agreed schedule logic. The discriminator is recognising that one path has room to move and the other does not, then smoothing within that room rather than levering the date or spending money.
