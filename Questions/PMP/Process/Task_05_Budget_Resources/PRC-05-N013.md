# Question

An agile team can fund only part of a backlog of four sizeable features this quarter. Each has a credible business-value estimate and a rough size estimate, and the sizes differ widely. One feature unlocks a contractual revenue payment that is forfeited if it is not delivered before quarter end; the other three carry no deadline. The product owner asks the project manager which ordering basis will get the most value out of the constrained quarter. What should the project manager recommend?

## Options

- **A**: Rank the features by weighted shortest job first — cost of delay divided by job size — so the time-critical feature's forfeited payment and each feature's size both drive the order.
- **B**: Rank purely by cost of delay, placing the feature with the expiring contractual payment first and ignoring the differences in size.
- **C**: Rank by business value alone, doing the highest-value features first regardless of how large they are or when value is realised.
- **D**: Rank by return on investment, dividing each feature's business value by its build cost and taking the highest ratios first.

## Answer

A

## Explanation

With a hard capacity limit and widely differing sizes, weighted shortest job first (cost of delay ÷ job size) maximises value delivered per unit of scarce capacity while still surfacing the expiring payment through its high cost of delay — it is the basis that accounts for both urgency and size. (B) cost of delay alone correctly elevates the deadline-bound feature but ignores size, so it can spend the quarter on one large item and forgo several smaller high-value ones. (D) ROI (value ÷ cost) captures efficiency but omits time-criticality, so it can rank the expiring-payment feature below a marginally better ratio and forfeit the payment. (C) value alone ignores both size and timing, the two factors that actually decide throughput here; it is the weakest because it optimises neither the constraint nor the deadline.
