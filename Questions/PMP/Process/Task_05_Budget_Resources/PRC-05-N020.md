# Question

On an agile hardware programme, an iteration stalled because a component with a six-week supplier lead time was needed but had not been ordered — the team had been treating procurement as just-in-time, the way it treats software work. The components vary: some are commodity and quick to source, a few are long-lead and expensive to hold in stock. The project manager wants a practice that stops iterations stalling on parts without tying up cash in inventory. What should the project manager institute?

## Options

- **A**: Run a rolling lookahead each iteration that identifies upcoming long-lead components and triggers their orders far enough ahead of the iteration that will consume them.
- **B**: Bulk-order all components for the whole release at the start so nothing is ever missing when an iteration begins.
- **C**: Insert a buffer iteration after each iteration so any component that arrives late has time to catch up before the work is needed.
- **D**: Add "components may arrive late" to the risk register with a mitigation to expedite shipping whenever a part is found to be missing.

## Answer

A

## Explanation

Long-lead physical resources cannot be sourced just-in-time, but not every part needs ordering early either; a rolling lookahead that flags upcoming long-lead items and triggers their procurement ahead of need matches order timing to lead time without holding unnecessary stock. (B) ordering everything up front guarantees availability but ties up cash, risks obsolescence as the backlog changes, and contradicts the goal of not carrying inventory. (C) buffer iterations burn capacity on every cycle and only absorb lateness after it has already occurred — they manage the symptom, not the procurement timing. (D) a risk-register entry with expedited shipping is a reactive fallback that still depends on noticing the gap late; it does not establish the forward planning that prevents the miss, and expedite fees recur each time.
