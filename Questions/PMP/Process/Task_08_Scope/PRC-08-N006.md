# Question

On a predictive hospital records-migration project, a senior consultant clinician has been emailing individual developers with "small tweaks" to data fields. Each is minor, but collectively the migration logic has drifted from the baseline and two were implemented last week. The PM discovers this during a routine review. The clinician is influential and means well. What should the PM do first?

## Options

- **A**: Raise a change request for the two implemented tweaks for impact assessment, and ask the developers to route any further requests from the clinician to the PM.
- **B**: Submit a single change request bundling all the tweaks already made so the baseline is updated retrospectively.
- **C**: Meet the clinician to explain the change process, log the already-implemented tweaks as changes for assessment, and redirect all future requests through change control.
- **D**: Roll back the two implemented tweaks immediately to restore the baseline, then continue as before.

## Answer

C

## Explanation

The core failure is an uncontrolled channel feeding scope changes around the baseline, so the PM must close the channel, assess the existing drift, and engage the influential stakeholder so the new routing is understood and sustained. (C) addresses behaviour, governance, and the existing drift while preserving a good relationship. (A) handles the mechanics correctly (change request plus re-routing) but skips speaking with the clinician, so a well-meaning, influential stakeholder is redirected by a developer with no explanation and is likely to push back or work around it again; the relationship and the durability of the fix both suffer. (B) retrospectively rubber-stamping the changes skips evaluation of cost, risk, and downstream impact, which is the point of change control. (D) rolling back before assessing whether the tweaks are actually beneficial may discard valuable changes and does nothing to stop the back-channel recurring.
