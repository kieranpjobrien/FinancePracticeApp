# Question

A predictive aerospace test-rig project is at design sign-off. The PM keeps a requirements traceability matrix and it is fully populated. At the review, a reviewer flags a costly self-test feature in the design that nobody present can tie to a stated need, and separately a regulator has just issued a new safety requirement that must be reflected before sign-off. Checking the matrix, the PM finds the self-test feature does not appear against any requirement, and the new safety requirement is not yet in the matrix at all. What should the PM do first?

## Options

- **A**: Add the new safety requirement to the matrix and trace it forward to specific design elements and tests, and raise the unlinked self-test feature for a decision on whether it is needed or should be removed.
- **B**: Treat the self-test feature as an approved part of the design since it is already built, and focus the remaining time on capturing the new safety requirement.
- **C**: Update the requirements documentation and WBS dictionary to describe both the new safety requirement and the self-test feature, then reconcile the matrix afterwards.
- **D**: Proceed to sign-off on the basis that the matrix is fully populated, and handle the safety requirement and the self-test feature as changes in the next baseline cycle.

## Answer

A

## Explanation

The matrix has surfaced two distinct gaps and the PM must act on both before sign-off: a new requirement with no forward links yet, and a design feature with no backward link to any requirement. Adding the safety requirement and tracing it forward to design and tests closes the forward gap, while raising the unlinked feature for a keep-or-remove decision addresses the possible orphan or gold-plating (A). (B) assuming the self-test feature is legitimate because it exists is exactly the reasoning traceability is meant to prevent; an unlinked feature must be justified or removed, not grandfathered in. (C) editing the requirements documentation and WBS dictionary first describes the items but does not establish the requirement-to-test linkage the sponsor needs, and defers the actual traceability work the matrix exists to drive. (D) a fully populated matrix is not the same as a complete one; signing off while a mandated safety requirement is untraced and an unexplained feature stands would approve a baseline known to be wrong.
