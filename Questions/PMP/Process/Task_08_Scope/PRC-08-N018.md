# Question

A predictive conference-centre refurbishment is in planning. The sponsor keeps introducing items such as "make it feel modern" and disputes later whether certain finishes were ever in scope. The project manager wants the artefact the team will point to when deciding, for a specific contested item, whether it is in scope and what would constitute acceptance of it. Which should the project manager rely on for that purpose?

## Options

- **A**: The project scope statement, which states the deliverables, their acceptance criteria, and explicit exclusions.
- **B**: The requirements documentation, which records each stakeholder requirement the refurbishment must satisfy.
- **C**: The scope management plan, which defines how scope will be defined, validated, and controlled.
- **D**: The project charter, which authorises the project and records its high-level objectives and success criteria.

## Answer

A

## Explanation

Deciding whether a specific item is in scope and what its acceptance looks like is answered by the project scope statement, which lists the deliverables, their acceptance criteria, and the exclusions. (B) is the trap: requirements documentation captures individual needs, but a recorded requirement is not the same as a committed, bounded deliverable with acceptance criteria and exclusions; many requirements are deferred or excluded when scope is defined. (C) the scope management plan describes the *process* for defining and controlling scope, not the actual deliverables and acceptance criteria for this refurbishment. (D) the charter gives only high-level objectives and success criteria, far too coarse to settle whether a particular finish is in or out.
