# Question

An agile team building a telehealth platform receives a mid-release request from a major hospital client to add a new clinical-coding standard. The product owner must decide whether and when to take it on. As the project manager coaching the team, which considerations should genuinely inform that decision? (Select all that apply.)

## Options

- **A**: The value of the new request relative to the items currently ahead of it in the backlog.
- **B**: Whether the work can be sliced into backlog items the team can complete within the normal sprint cadence.
- **C**: The impact of the new standard on the agreed release goal and any external commitments already made for this release.
- **D**: Whether accepting the request now would require displacing already-committed work in the current sprint, versus scheduling it for a future sprint.
- **E**: Whether the request should be added immediately because the client is a major account, regardless of its backlog value relative to other items.

## Answer

A, B, C, D

## Explanation

Deciding on emergent scope means weighing value, feasibility within the cadence, impact on goals and commitments, and *when* to take it on without disrupting work in flight. (A) lets the product owner reprioritise by value. (B) confirms the work can actually be sized and delivered iteratively. (C) checks alignment with the release goal and external promises. (D) is a real and correct consideration — agile welcomes change, but changing the *current* sprint's committed scope is disruptive, so timing it for an upcoming sprint is usually how the change is absorbed. (E) is the trap: stakeholder importance rightly influences how carefully the request is heard, but letting account size override value-based prioritisation outright is not how the backlog is ordered — the request still earns its place on value and fit, not on the client's size alone.
