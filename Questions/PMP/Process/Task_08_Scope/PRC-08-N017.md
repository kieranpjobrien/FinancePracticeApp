# Question

During refinement on an agile logistics-routing product, the team faces an epic-sized story: "As a dispatcher I can optimise multi-stop routes across all vehicle types, fuel models, and live traffic." It cannot fit in one sprint and the team cannot estimate it confidently. The simplest single-vehicle, no-traffic routing case is well understood, but the multi-vehicle and live-traffic dimensions carry real unknowns. How should the team take this epic forward?

## Options

- **A**: Spike the riskiest unknowns — multi-vehicle balancing and the live-traffic feed — for a sprint to retire the uncertainty, then split and estimate the epic once those answers are in.
- **B**: Break it into horizontal layers — build the data model first, then the optimisation engine, then the dispatcher UI — so each technical foundation is solid before the next is built on it.
- **C**: Split it into thin vertical slices that each route a dispatcher end-to-end, starting with the simplest single-vehicle case and adding vehicle types, fuel models, and traffic as later slices.
- **D**: Have the product owner break the epic into smaller stories along the named dimensions — vehicle types, fuel models, traffic — and bring those back to the team to estimate next refinement.

## Answer

C

## Explanation

The epic is too big to estimate or finish in a sprint, and the goal is to start delivering and learning from working software rather than building toward a distant payoff. (C) slices it vertically so the first increment routes a real dispatcher end-to-end and each later slice adds a dimension, giving demonstrable value and feedback from sprint one while the well-understood simple case carries the early work. (A) is the strongest distractor: spiking genuine unknowns is legitimate, but a whole sprint of investigation with nothing shippable defers value when an end-to-end slice of the *known* simple case could ship now and surface much of the same learning through real use. (B) horizontal layering is a common instinct and each layer feels like solid progress, yet nothing is demonstrable until the final layer lands, so it forfeits the early feedback incremental delivery exists to provide. (D) splitting is right but ownership is wrong: decomposing stories for estimability is teamwork at refinement, not something the product owner does alone and hands back, and slicing by named feature dimension still tends toward component-shaped stories rather than thin end-to-end value. The margin is fine because A, B and D are all plausible ways to break down hard work, but only C delivers working value early while still chipping away at the unknowns.
