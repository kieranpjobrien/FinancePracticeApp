# Question

On a predictive data-centre fit-out, the team has a signed scope statement and a work breakdown structure decomposed to work-package level. Estimators still return wildly divergent cost and duration figures for the same work packages, and reviewers cannot tell what each package actually includes or where one stops and the next begins. The sponsor wants the variation resolved before approving the baseline. What should the project manager do next?

## Options

- **A**: Develop the WBS dictionary so each work package has a defined scope of work, boundaries, and deliverables for estimators to price against.
- **B**: Decompose the work packages further into individual activities so estimates can be built bottom-up from smaller pieces.
- **C**: Run a three-point estimate on each work package to express the divergence as a range the sponsor can accept.
- **D**: Hold a workshop for the estimators to talk through each package and converge on a single agreed number per package.

## Answer

A

## Explanation

The estimates diverge because estimators are interpreting each work package differently; there is no shared statement of what a package contains or where its boundaries lie. The WBS dictionary supplies exactly that definition, so everyone prices the same scope. (B) decomposing into activities is part of schedule development, but smaller pieces estimated from the same ambiguous package definitions will still diverge; granularity is not the problem here. (C) three-point estimating quantifies genuine uncertainty, but it would dress up disagreement over what the work *is* as if it were risk, masking the real cause. (D) a convergence workshop may force a single number, but without a written definition the agreement is unrepeatable and the ambiguity resurfaces at the next estimate or at execution.
