# Question

An agile e-commerce team has a written definition of done that includes automated tests and an accessibility check. Stories are repeatedly marked done in the iteration review and then reopened by the product owner days later because those two checks were skipped under deadline pressure. The product owner is frustrated that "done" keeps meaning something different. At the next retrospective the team agrees the checks matter and wants to stop the reopens. Which change is most likely to actually stop them?

## Options

- **A**: Treat any unmet definition-of-done item as a blocker that keeps a story open, so a story cannot be presented or accepted as done in the review until automated tests and the accessibility check are complete.
- **B**: Move the automated-test and accessibility checks earlier by adding them to each story's acceptance criteria so they are agreed and tracked before development starts.
- **C**: Add a definition-of-ready so stories only enter the iteration once they are small enough to finish the full definition of done within the iteration.
- **D**: Have a second developer peer-review each story against the definition of done before it is demonstrated, to catch skipped tests or accessibility checks.

## Answer

A

## Explanation

The team already knows and endorses the standard; the reopens happen because "done" is being declared while the definition of done is unmet, so the durable fix is to make the unmet items actually block completion rather than to relocate, restate, or add inspection around them. Treating any unmet item as a blocker that keeps the story open (A) removes the option to declare done early, which is the precise behaviour causing the reopens. (B) putting the checks into acceptance criteria makes them visible per story, but acceptance criteria can be skipped under the same deadline pressure that is already defeating the definition of done, so nothing structurally prevents an early "done." (C) a definition-of-ready helps ensure stories are finishable, but the team's problem is not that stories are too big to finish; it is that they declare done without finishing, which readiness does not gate. (D) peer review can catch some skips, but it relies on a person consistently enforcing the standard and adds a step that itself gets compressed under pressure, whereas making the items a hard precondition removes the discretion entirely.
