# Question

On a large predictive airport-baggage-handling project, requirements have been elicited from airlines, security, ground handlers and the operator. Two months in, disputes keep erupting: stakeholders argue over whose requirements rank higher, two teams have built to different interpretations of the same need, and nobody can say which requirements a late change would affect. Each dispute is being settled ad hoc in meetings. The project manager wants to stop the recurrence rather than keep refereeing. What should the project manager do first?

## Options

- **A**: Agree and document how requirements will be prioritised, traced and changed, then apply that process to the open disputes.
- **B**: Stand up a requirements traceability matrix so any change can be traced to the requirements and components it affects.
- **C**: Run a prioritisation workshop to rank the contested requirements and resolve the current disagreements.
- **D**: Re-baseline the requirements documentation so every elicited requirement is captured in one agreed, current list.

## Answer

A

## Explanation

The disputes — over ranking, over interpretation, and over change impact — are all symptoms of a missing agreed *process* for handling requirements; settling them one meeting at a time treats each symptom while the cause recurs. (A) defines that process (how requirements are prioritised, traced and changed) and then uses it on the live issues, which is what stops the pattern. (B) a traceability matrix fixes only the "what does a change affect" symptom and presupposes agreement on how tracing will work. (C) a prioritisation workshop resolves today's ranking fight but, without an agreed method, the next contested requirement reopens it. (D) re-capturing requirements in one list addresses neither prioritisation nor change impact and leaves the interpretation disputes unresolved. The fine margin is that B, C and D each remedy one strand of the problem, whereas only A installs the governing process that the others operate within.
