# Question

A Scrum team building a logistics tracking system already refines its backlog every sprint and splits large items, yet still keeps failing to finish committed stories: once started, several prove larger and more ambiguous than they looked. Looking closer, the project manager sees the splits are made along technical layers — a "database" story, an "API" story, a "UI" story — so no single story delivers anything testable on its own, and hidden work only appears when the layers are integrated late in the sprint. The product owner brings clear priorities. What should the project manager encourage the team to do?

## Options

- **A**: Re-slice upcoming stories so each one delivers a thin end-to-end piece of working functionality, surfacing integration work inside the story rather than after it.
- **B**: Add an explicit integration story to each sprint to absorb the cross-layer work the current layered stories leave until the end.
- **C**: Tighten the definition of ready so a story is only pulled in once every layer it touches has been analysed and estimated up front.
- **D**: Have the team add a contingency buffer to each layered story's estimate to cover the integration work that keeps appearing late.

## Answer

A

## Explanation

The team is already refining and splitting, so the failure is not "split more" — it is *how* the splitting is done. Slicing by technical layer produces stories that cannot be demonstrated or tested independently and that hide the integration effort until the layers meet, which is exactly when the work blows up. (A) fixes the slicing pattern itself: vertical, end-to-end slices each deliver something testable and force integration to be confronted within the story, so size and ambiguity are exposed during refinement, not at sprint's end. (B) a dedicated integration story keeps the flawed horizontal split and simply parks the risky work in its own late, hard-to-size item — the same deferred-integration trap relocated. (C) more up-front analysis of every layer adds refinement overhead and a heavier gate, yet still ships layered slices that individually deliver nothing, so stories keep failing to finish. (D) padding estimates conceals the structural problem behind a buffer and makes the symptom predictable rather than removing it. The margin is fine because B, C and D are all real, defensible refinement responses; only A changes the axis of the split, which is the actual cause.
