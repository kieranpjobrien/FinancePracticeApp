# Question

A hybrid project delivers an office building predictively and its control software iteratively. A stakeholder requests a software behaviour change that the agile team can comfortably absorb into its next sprint. On analysis, the change would also require a different sensor layout in the building works, whose scope baseline is already approved and whose first-fix wiring starts in two weeks. How should the project manager handle the request?

## Options

- **A**: Have the team prototype the new behaviour in the next sprint to confirm it genuinely needs the revised sensor layout, then raise a building change only if the prototype proves it does.
- **B**: Take the request through integrated change control as a single coupled change, assessing the impact on both the software work and the approved building baseline together before any part of it is approved.
- **C**: Approve the software change into the backlog now since the agile stream can absorb it, and raise a separate change request against the building baseline in parallel.
- **D**: Ask whether the software could meet the stakeholder's intent within the existing sensor layout, and adopt that option if it exists, so the approved building baseline is not disturbed.

## Answer

B

## Explanation

The request looks like ordinary agile scope the team can absorb, but it reaches across the boundary into an approved predictive baseline with imminent wiring, so the two impacts are one coupled change and must be decided together. (B) does that through integrated change control, weighing the software benefit against the building rework and timing as a single decision. (C) is the strongest distractor because the agile stream genuinely can take the change, making it tempting to let it proceed and handle the building side "in parallel" — but approving one half of a coupled change before the other is assessed is precisely how the streams desynchronise, and the building work may be committed before its change is dispositioned. (A) prototyping to confirm the dependency is reasonable diligence, yet starting build-side analysis only after a sprint burns two weeks of a critical wiring lead time and still defers the integrated decision. (D) seeking a no-baseline-impact software solution is a smart thing to explore and may make the building change unnecessary, but adopting it unilaterally still resolves a cross-stream scope question outside change control and assumes it fully satisfies the stakeholder's intent. The margin is fine because A, C and D are each defensible and may even feed the decision, but only B keeps a change that spans both an evolving backlog and a frozen baseline under one controlled, jointly-assessed decision.
