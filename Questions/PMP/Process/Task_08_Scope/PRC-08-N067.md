# Question

On a predictive rail-ticketing project, the PM faces four scope situations in the same week. Match each situation on the left to the most appropriate first response on the right. Each response is used once.

## Items

- **1**: A stakeholder emails a request for a new reporting feature that is clearly outside the approved scope statement but could add real value.
- **2**: The team has quietly been building extra screens beyond the baseline over the last sprint, with no request on record, and cost is now over baseline.
- **3**: The sponsor wants assurance, before sign-off, that every approved requirement is reflected in the design and tests.
- **4**: A developer disputes that a delivered work package meets its agreed completion criteria, and the PM needs the authoritative definition of what that package must deliver.

## Matches

- **A**: Pause the unrequested work and raise it through integrated change control so its impact is assessed and a decision recorded.
- **B**: Log the request and submit it as a change request for impact analysis before any commitment is made.
- **C**: Review the requirements traceability matrix to confirm forward coverage from requirement to design and test.
- **D**: Consult the WBS dictionary entry for that work package to confirm its deliverable and acceptance criteria.

## Answer

1-B, 2-A, 3-C, 4-D

## Explanation

A new but legitimate request that is outside scope is handled by logging it and submitting a change request for analysis before committing (1-B). Work already built informally beyond the baseline is uncontrolled scope creep, so the response is to pause it and route the additions through integrated change control for impact assessment and a recorded decision (2-A). Confirming that every requirement is carried into design and tests is exactly what the requirements traceability matrix does (3-C). The authoritative description of a work package's deliverable and acceptance criteria is the WBS dictionary (4-D). The trap is 1 versus 2: both involve out-of-scope work, but an unbuilt external request enters change control as a normal change (B), whereas work already executed without authorisation must first be stopped and brought back under control (A).
