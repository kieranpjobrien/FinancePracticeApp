# Question

An agile logistics-tracking team finishes a two-week sprint: it holds its review and retrospective, ships the increment, and updates the backlog. This increment delivered the last of the high-value features the product owner had prioritised, and several stakeholders treat it as the natural end point. A new scrum master suggests the project can now be formally closed. The product backlog still holds lower-value items and the sponsor has not yet spoken about ending the funding. How should the project manager respond?

## Options

- **A**: Treat the sprint as the project close, since this iteration delivered the prioritised high-value features and the routine end-of-sprint ceremonies are complete.
- **B**: Explain that this is an iteration boundary, and project closure follows once objectives are confirmed met and the sponsor formally ends the funded initiative.
- **C**: Begin project closure activities now and run them in parallel with one or two further sprints, so administrative closeout overlaps continued delivery.
- **D**: Ask the product owner to decide whether the project is complete, since the product owner judges when the delivered value is sufficient to stop.

## Answer

B

## Explanation

Iteration-level closure (review, retrospective, increment, backlog update) recurs every sprint and is distinct from project or release closure, which occurs once objectives are confirmed achieved and the funding authority ends the initiative. (B) draws this line. (A) is tempting because the most valuable work is done, but delivering prioritised features in a sprint is still a sprint boundary, not the governance event that ends the project, and lower-value backlog and the funding decision remain open. (C) sounds efficient but starting formal closure while sprints continue means the initiative is neither genuinely closed nor cleanly governed — closure presumes delivery has stopped. (D) overstates the product owner's authority: the product owner ranks value and can recommend stopping, but ending the funded initiative is a sponsor/governance decision, not the product owner's alone.
