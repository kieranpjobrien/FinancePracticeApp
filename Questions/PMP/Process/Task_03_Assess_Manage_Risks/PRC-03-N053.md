# Question

An agile team has identified a serious technical risk: an undocumented legacy data format may make a planned migration feature far harder than estimated. The feature is scheduled for sprint 6, four sprints away, and several higher-value features sit ahead of it. The product owner wants to protect that value ordering; an architect notes the format question is the kind of thing that could be probed cheaply now or could be designed around if the team decoupled the feature from the legacy source. The project manager is deciding what to recommend. What is the best recommendation?

## Options

- **A**: Pull a small, time-boxed spike into the next sprint to probe the legacy format, so the team learns whether the estimate holds while the higher-value features still run ahead of it.
- **B**: Keep the spike sized but schedule it for sprint 5, immediately before the feature, so investigation happens against the most current understanding without disturbing earlier higher-value work at all.
- **C**: Recommend the product owner re-rank the migration feature higher so the riskiest work is confronted earlier, accepting some delay to the higher-value features in exchange for retiring the risk sooner.
- **D**: Propose a design change to decouple the feature from the legacy format now, removing the dependency that creates the risk rather than spending a sprint investigating it.

## Answer

A

## Explanation

The risk is serious, four sprints out, and the cheapest way to act on it is a small probe that does not disturb the value ordering—so investigate early while the higher-value features still run first (A). Each distractor is a defensible tactic, but each is worse on the speed-versus-disruption and reduce-versus-defer trade-offs. (B) keeps the spike small but defers it to sprint 5, which surrenders three sprints of lead time: if the spike reveals the work is far larger, the team learns it with almost no room to re-plan, which is the disruption early probing exists to avoid. (C) retires the risk earliest but pays for it by reordering a whole high-effort feature ahead of higher-value work—disproportionate when a spike buys most of the same learning for a fraction of the cost. (D) could genuinely eliminate the risk, but committing to a decoupling design before the format is even understood spends real architecture effort on a solution to a problem whose size is still unknown; you would probe first, then decide whether decoupling is warranted. The discriminator is buying maximum lead time at minimum cost and disruption, which the early small spike does best.
