# Question

A Scrum team building a payments feature identifies during backlog refinement that one upcoming story depends on a third-party fraud API that has been unreliable in two recent demos. The product owner had ranked that story low because its business value is modest. What is the most appropriate way to handle this in the next sprint planning?

## Options

- **A**: Leave the story in its low-value position; risk is the team's concern and should be tracked separately from the product backlog ordering.
- **B**: Discuss the dependency risk with the product owner so the story (or a spike to validate the API) can be considered for earlier sequencing despite its modest value.
- **C**: Add a buffer sprint at the end of the release so there is time to deal with the fraud API if it fails late.
- **D**: Escalate the unreliable API to the vendor's account manager and pause the story until the vendor guarantees uptime.

## Answer

B

## Explanation

Agile teams reorder the backlog using risk as well as value (a risk-adjusted backlog), pulling risky or uncertain items forward so failure is discovered early and cheaply, and a spike is the standard device for de-risking an unknown integration. (B) brings the risk into the product owner's prioritisation decision where sequencing actually happens. (A) wrongly separates risk from backlog ordering, which is precisely where agile addresses it. (C) an end buffer defers discovery to the most expensive moment, the opposite of fail-fast. (D) pausing for a vendor guarantee surrenders schedule control and ignores that the team can validate the API itself.
