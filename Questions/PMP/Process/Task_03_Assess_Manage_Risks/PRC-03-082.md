# Question

An agile team building a payments reconciliation feature for a fintech surfaces, during backlog refinement, that a third-party banking API the feature depends on is scheduled for a breaking version change in about three iterations; the exact cut-over date is not yet confirmed. The product owner currently has the feature ranked third by business value, behind two unrelated items. The team lead wants the team's response to fit its way of working. What is the most appropriate course of action?

## Options

- **A**: Add a spike to the next iteration to confirm the cut-over date and the size of the change, then decide on a response once the uncertainty is reduced.
- **B**: Work with the product owner to add a risk-response item to the backlog and rank it to land before the breaking change, leaving the feature itself at third.
- **C**: Have the product owner move the payments feature to the top of the backlog so the whole feature ships before the API change takes effect.
- **D**: Raise the API change to the programme risk board so it is logged on the organisational register and a contingency budget is set.

## Answer

B

## Explanation

The threat is already known and dated, so the agile response is to turn it into a backlog item the product owner ranks to complete before the change lands, while the feature keeps its value-based position; this protects the dependency without distorting the rest of the order (B). Each alternative is a reasonable instinct, which is where the discrimination lies. (A) is attractive because the date is unconfirmed, but a spike defers the response on a threat the team can already begin to plan for, and the breaking change is coming regardless of the exact day; reducing uncertainty is useful but is not the first commitment to make. (C) reprioritises the entire feature on the strength of one dependency, displacing higher-value work when only the response activity needs to precede the change. (D) escalates a risk the team can own inside its own backlog, adding governance without moving the dependency forward. The call turns on owning the risk as ranked response work now versus investigating first, reordering the whole feature, or escalating.
