# Question

A product owner and three agile teams building a streaming-media app each have a different definition of done, and the release plan assumes all features are independently shippable. During integrated planning the project manager realises this inconsistency could cause features marked "done" by one team to be unreleasable when combined. What is the most effective integrating action?

## Options

- **A**: Facilitate agreement on a shared definition of done that includes integration and release criteria, and align the release plan's "independently shippable" assumption to it.
- **B**: Have the teams keep their own definitions of done but adopt continuous integration so divergences surface in a shared build every day rather than at release.
- **C**: Schedule a hardening iteration before each release in which the teams jointly integration-test and remediate features that are not releasable when combined.
- **D**: Add release-level acceptance criteria owned by the product owner that every feature must meet before it can ship, while leaving each team's definition of done as is.

## Answer

A

## Explanation

The release plan assumes features are independently shippable, but inconsistent definitions of done mean "done" does not guarantee releasable — so the integrating fix is a shared, integration-aware definition of done that the release plan is then aligned to (A). It corrects both the divergence and the false assumption at their source. (B) gives faster feedback on integration drift but leaves the definitions themselves inconsistent, so teams still disagree on what "done" includes. (C) catches integration gaps each release but institutionalises late rework and still tolerates differing definitions, treating the symptom periodically. (D) layers a release gate on top of unaligned team definitions, adding a checkpoint rather than reconciling what each team must deliver to call work done.
