# Question

A hybrid programme delivers a customer portal in two-week sprints while a vendor builds the back-office mainframe integration under a fixed-date, predictive contract. The agile team's release plan and the vendor's milestone schedule were produced independently. During integrated planning the project manager notices the portal's payment feature is slated for the team's sprint 4, but the vendor's settlement API will not be available until two months later. What is the best action?

## Options

- **A**: Ask the agile team to build payment against a mocked API in sprint 4 so the release plan is preserved and rework is accepted later.
- **B**: Tell the vendor to accelerate the settlement API to align with sprint 4 because the agile cadence sets the programme rhythm.
- **C**: Re-sequence the product backlog so payment-dependent stories are scheduled after the vendor's API milestone, and make the cross-team dependency explicit in the integrated plan.
- **D**: Keep the two plans separate and manage the timing gap through the weekly status report so each team retains autonomy.

## Answer

C

## Explanation

Integrating planning across a hybrid programme means surfacing and sequencing cross-team dependencies so the predictive and adaptive streams actually mesh; reordering the backlog around the hard external milestone (C) does that. (A) builds throwaway work and risks integration defects to protect a plan that should adapt. (B) imposes the agile cadence on a fixed-date contract the vendor cannot simply compress, ignoring the constraint that drives the dependency. (D) treats a known hard dependency as a reporting matter rather than a planning one, which is how integration failures occur.
