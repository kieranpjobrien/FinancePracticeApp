# Question

Four agile teams are building separate microservices for a single insurance-claims platform on a synchronised two-week cadence. Each team maintains its own backlog and definition of done. During integrated planning the project manager learns that teams A and C both depend on a shared event-schema that team B owns, but team B has not prioritised it and there is no agreed view of cross-team work. What should the project manager do first?

## Options

- **A**: Bring the three product owners together to negotiate the schema's place in team B's backlog and the trade-offs it forces on B's own commitments, then record the agreement.
- **B**: Convene a cross-team planning event for all four teams to surface every dependency, agree the shared-schema delivery, and build one visible view of the next increments before committing the plan.
- **C**: Add the schema dependency to the standing scrum-of-scrums so the teams track and resolve it through the existing cross-team coordination forum.
- **D**: Have teams A and C build temporary stub implementations of the event-schema so they can proceed now and swap in team B's version when it lands.

## Answer

B

## Explanation

The problem is broader than one schema: there is no agreed view of cross-team work and the synchronised increment is about to be planned, so the integrating move is to convene all four teams to expose the full dependency web and align their increments before committing — a cross-team planning event (B). (A) fixes the single visible dependency through the right people, but a two-PO negotiation still misses the other unsurfaced cross-team work and produces no shared increment view. (C) routes the item to a forum built for tracking and unblocking day-to-day, not for jointly planning an increment, so it manages the symptom rather than integrating the plan. (D) buys local progress by creating throwaway work and integration risk, leaving the underlying lack of an agreed cross-team plan untouched.
