# Question

Over the last four releases of a payroll system, tax-calculation errors have each been logged, fixed and closed within their target dates. The errors are not identical — each affects a different calculation — and each fix has resolved the specific case it addressed. The team is meeting its issue-closure targets, but a new tax-calculation error has appeared again in the latest release. What is the best way for the project manager to manage this?

## Options

- **A**: Keep the current approach, since each error is a genuinely distinct issue and the team is resolving every one within its target.
- **B**: Investigate what the four issues have in common — design, data or process — and address the underlying source rather than only the latest case.
- **C**: Add a tax-calculation regression test suite that runs each release, so any recurrence is caught and fixed before the release ships.
- **D**: Raise the default priority of any tax-calculation issue, so each new occurrence is resolved faster and with senior attention.

## Answer

B

## Explanation

Four separate fixes have each closed cleanly, yet errors in the same area keep appearing — that pattern points to a shared underlying source the individual patches are not touching, so the action with leverage is to investigate what the issues have in common and fix the source. (A) is genuinely tempting because the metrics look healthy: every issue is closed on time. But closing symptoms on time is not the same as solving the problem, and accepting recurrence as routine lets the real defect persist. (C) is a real improvement and catches recurrences earlier, but it still detects and patches instances rather than removing whatever keeps producing them. (D) makes the reaction faster and more visible without making the errors any less likely to occur. The discrimination is between eliminating the common cause and variously tolerating, detecting or reacting faster to the symptoms.
