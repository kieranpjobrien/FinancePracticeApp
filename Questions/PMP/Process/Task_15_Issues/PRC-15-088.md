# Question

An agile team building a retail e-commerce checkout has tracked "payment gateway provider may impose a mid-project API change" on the risk register for two months. This morning the provider confirms the API change is now mandatory and takes effect in three weeks, breaking the current integration. What should the project manager do with this entry?

## Options

- **A**: Log it as a lesson learned so future teams choose a provider with a more stable API.
- **B**: Keep it on the risk register and raise its probability to certain to reflect that the change is now confirmed.
- **C**: Submit a change request to extend the timeline before recording anything further about the API change.
- **D**: Move it from the risk register into the issue log, assign an owner and a target date, and manage it to resolution.

## Answer

D

## Explanation

A risk that has materialised is now a present problem and belongs in the issue log with an owner and a target date so it can be actively worked to closure. (B) leaves a realised event on the register of things that *might* happen, which is the wrong artefact and stops it being managed as the live problem it is. (C) leaps to a schedule change before the issue is even logged or its true impact assessed, and a timeline extension may not be the answer. (A) is a closure-stage reflection that does nothing to resolve the integration break the team faces now.
