# Question

A project manager leading the installation of a new bottling line is reviewing the issue log with the team. One entry reads: "If the regional power authority proceeds with the planned substation upgrade in Q3, the site may experience scheduled outages that could delay commissioning." The team has been treating this entry as an open issue and reviewing it weekly. The project manager wants the log to reflect the correct status of each entry. How should the project manager handle this entry?

## Options

- **A**: Move it to the risk register as a threat, since the event is conditional and has not yet occurred.
- **B**: Keep it in the issue log but mark it low priority until the outages are confirmed.
- **C**: Close it in the issue log because nothing has actually happened yet.
- **D**: Escalate it to the sponsor so the commissioning date can be formally rebaselined now.

## Answer

A

## Explanation

The defining test between an issue and a risk is whether the event has occurred: this entry describes a conditional future event ("if... may... could"), which is the textbook signature of a risk, so it belongs in the risk register where it can be assessed and a response planned (A). Leaving it in the issue log, even at low priority, perpetuates the misclassification and clutters issue tracking with something that is not a current problem (B). Simply closing it discards a legitimate threat that the team should continue to monitor and plan for (C). Escalating to rebaseline the schedule presumes an impact that has not materialised and is premature given the upgrade may not even proceed (D).
