# Question

On a hybrid telecom rollout, an issue under investigation has evolved and now looks more like a scope expansion than a defect. The team has already spent two days working on it. What should the project manager do?

## Options

- **A**: Stop work on the item, raise a change request covering the newly identified scope, and process it through change control.
- **B**: Continue resolving the item as an issue to avoid losing the two days of effort already invested by the delivery team recently.
- **C**: Split the effort so scope work continues while the change request is processed in parallel by the project governance board.
- **D**: Reclassify the item as a new requirement and add it silently to the next iteration's backlog for the team to pick up.

## Answer

A

## Explanation

Integrated change control applies whenever scope shifts, even for "small" items already underway. (A) respects governance and stops unauthorised scope growth. (B) continues unapproved scope work on sunk-cost logic. (C) runs scope work without authorisation, risking rework if the change is rejected. (D) bypasses change control entirely by disguising scope as backlog refinement.
