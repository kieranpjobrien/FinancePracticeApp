# Question

An agile team building an insurance-claims platform has fixed the same defect — claim data lost when a save coincides with a session timeout — in four of the last five iterations, each time by patching the screen where it surfaced. It has now reappeared on a fifth screen, mid-iteration, with the current sprint goal already committed and the patch itself only a few hours' work. A diagnosis of the shared session-handling layer would take most of the remaining iteration. How should the team proceed?

## Options

- **A**: Open a spike to diagnose the shared session-handling layer now and size a fix for the common cause, deferring the screen patch.
- **B**: Apply the quick screen patch to protect the sprint goal, and bring the systemic cause to the team as a prioritised backlog item for planning.
- **C**: Patch the fifth screen and extend the regression suite to cover the save-during-timeout path on every screen that handles claims.
- **D**: Have the product owner accept the patched build and log the recurrence so its frequency is tracked across future iterations.

## Answer

B

## Explanation

The pattern — one failure mode resurfacing on a fifth screen — clearly points to a shared root cause, so principle-spotting alone might suggest "go fix the cause." But the team is mid-iteration with a committed goal and a root-cause diagnosis that would consume most of the remaining time, so the disciplined response is to stop the immediate bleeding with the cheap patch *and* route the systemic cause into planning as a prioritised item, where it can be sized and scheduled without sacrificing the current commitment (B). Dropping everything into a spike now (A) attacks the right target but unilaterally trades a committed sprint goal for unplanned work the product owner has not prioritised — correct intent, wrong sequencing and wrong decision-owner. Patching plus broad regression tests (C) hardens against this failure mode but still leaves the underlying defect in the session layer, deferring rather than eliminating it. Accepting the build and merely tracking frequency (D) is the weakest: it documents the recurrence without committing to remove it. The margin is between fixing the cause *immediately at the cost of the commitment* and *securing the commitment while ensuring the cause is prioritised for real* — the latter respects both the sprint and the backlog's owner.
