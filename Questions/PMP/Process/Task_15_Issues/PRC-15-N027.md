# Question

A new project manager inherits an aerospace tooling project whose issue log contains only two columns: a free-text description and a tick-box marked "done". The PM finds it impossible to tell who is handling each open issue or when it is due. Which set of attributes should the PM add to make the log usable for managing issues to closure?

## Options

- **A**: Owner, priority, target resolution date, and current status for each issue.
- **B**: The name of the person who first reported the issue and the exact time it was raised.
- **C**: A probability and impact score for each issue, mirroring the risk register.
- **D**: A running cost estimate of the total effort spent discussing each issue to date.

## Answer

A

## Explanation

To drive issues to closure the log must show who is accountable, how urgent it is, by when it should be resolved, and where it currently stands; owner, priority, target date and status cover exactly that. (B) records provenance, which is minor and does not help anyone resolve or track the work. (C) applies probability/impact scoring, which belongs to risks that have not yet occurred, not to issues that already have. (D) adds accounting overhead that tells you nothing about how to close the issue.
