# Question

On a predictive building-management-system project, an issue is logged: the installed sensors report temperature accurately but the client now says they also expected humidity readings, which the contract and signed requirements never specified. Resolving the client's expectation would require additional hardware and rework. The issue owner asks the project manager how to proceed. What is the most appropriate response?

## Options

- **A**: Resolve the issue by quietly adding the humidity sensors, since keeping the client satisfied is the project's ultimate purpose.
- **B**: Close the issue as invalid because humidity was never in the requirements, and tell the client the request is out of scope.
- **C**: Use the issue to surface the gap, then route the added humidity capability through the change control process for impact assessment and a decision.
- **D**: Log it as a risk to the project's reputation and monitor whether the client raises it again before doing anything.

## Answer

C

## Explanation

The issue has revealed a genuine expectation gap, but adding capability beyond the agreed baseline is a scope change, so it must go through change control where cost, schedule and value are assessed and an authorised decision is made. (A) bypasses change control and commits unbudgeted scope, undermining baselines and possibly the contract. (B) is technically correct that it is out of baseline but dismisses a real stakeholder concern without offering the proper change route. (D) downgrades a live, raised request to a passively monitored risk, deferring a decision the client is already expecting.
