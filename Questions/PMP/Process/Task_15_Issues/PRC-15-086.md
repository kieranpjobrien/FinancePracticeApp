# Question

On a hybrid pharmaceutical-serialisation programme, an issue where label data failed to sync to a packaging line was logged, fixed and closed last month. The same symptom has now reappeared on a different line, and a team member has already re-applied the previous fix and re-closed the issue. The project manager learns of this only at the weekly review. What should the project manager do?

## Options

- **A**: Raise a change request to standardise the fix as a permanent configuration on every packaging line.
- **B**: Accept the re-closure, since the proven fix was applied again and the symptom is gone for now.
- **C**: Reopen the matter and investigate the common cause across both occurrences before accepting that the re-applied fix has resolved it.
- **D**: Record both occurrences as a lesson learned so future programmes watch for label-sync problems.

## Answer

C

## Explanation

A defect that recurs after being "fixed and closed" signals that the earlier resolution treated a symptom, so the value lies in investigating the common root cause before trusting the repeated patch. (B) accepts a fix that has already failed to prevent recurrence once, risking a third instance at a worse moment. (A) standardises a remedy whose adequacy is unproven; locking in a symptom-level fix across all lines could entrench the underlying fault. (D) captures learning but does nothing to resolve a live, recurring problem in the current programme.
