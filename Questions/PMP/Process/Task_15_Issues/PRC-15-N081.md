# Question

On a water-treatment infrastructure project, a recurring calibration fault has come up at three consecutive weekly status meetings. A different engineer mentions it each week, the meeting agrees it "needs looking at", and the same note appears in the next set of minutes with no change in between. Several plausible causes have been floated but none confirmed, and the fault is intermittent rather than stopping work. What should the project manager do first?

## Options

- **A**: Add the fault to the issue log with a single named owner and a target resolution date.
- **B**: Commission a focused diagnostic to confirm the root cause before deciding on any response.
- **C**: Escalate the fault to the engineering manager for direction at the next steering meeting.
- **D**: Raise a change request so corrective calibration work can be funded and scheduled.

## Answer

A

## Explanation

The fault is not progressing because it has no durable home: it lives only in rotating verbal mentions and minutes, with no one accountable and no deadline. The first step is to convert it from recurring discussion into a tracked issue with one named owner and a target date (A); that single act creates the accountability and follow-up cadence under which any of the other actions can then be driven. Commissioning a diagnostic (B) is a sensible early task — but it *is* an action the owner would direct, and assigning it without first establishing ownership risks repeating the same ownerless drift that has already happened three times. Escalation (C) is premature: nothing here exceeds the PM's authority and there is no specific decision yet to put to the engineering manager. Raising a change request (D) presumes the response and its cost before the fault has even been characterised. The margin is between *establishing ownership and tracking first* and jumping to a particular response (diagnose, escalate, fund) before anyone owns the item.
