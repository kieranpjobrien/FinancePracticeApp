# Question

While resolving a logged issue about a malfunctioning report, the project manager and the user agree that the cleanest fix is to add two new fields the user has always wanted but that were never in the agreed scope. The user asks the PM to "just include them while you're in there". What should the project manager do?

## Options

- **A**: Add the two fields as part of resolving the issue, since the work is already open and the user has confirmed the need.
- **B**: Fix the malfunctioning report to restore agreed behaviour, and route the two new fields through the change control process as a separate scope request.
- **C**: Decline to touch the report at all until the user submits a formal change request covering both the fix and the new fields.
- **D**: Add the two fields but log them quietly so the change does not draw attention from the change control board.

## Answer

B

## Explanation

The malfunction is a defect to be corrected back to baseline, while the two new fields are added scope that must go through change control regardless of how convenient the timing is. (A) lets scope creep ride in under the cover of an issue fix, bypassing change control. (C) wrongly holds the legitimate defect fix hostage to a change request it does not require. (D) deliberately conceals a scope change from governance, which is a control and integrity failure.
