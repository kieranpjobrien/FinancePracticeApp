# Question

On a warehouse-automation project, the risk register has long contained an entry that a single specialist integrator the project depends on could become unavailable. This morning the integrator's firm went into administration and withdrew its staff with immediate effect, halting integration work. What should the project manager do first?

## Options

- **A**: Record the now-realised risk as an issue and trigger the response that was planned for it.
- **B**: Update the risk register entry's probability to reflect that it has occurred.
- **C**: Raise a change request to extend the project schedule to absorb the delay.
- **D**: Escalate to the sponsor for a decision on whether to continue the project.

## Answer

A

## Explanation

A risk that materialises becomes an issue: the threat the register anticipated has now happened and is blocking work, so the project manager records it as an issue and activates the contingency or fallback response already planned for it (A). Editing the risk's probability is a documentation afterthought that does nothing to address the live stoppage (B). Raising a change request to extend the schedule presumes an impact and response before the planned response has even been triggered, and may be unnecessary if the contingency restores progress (C). Escalating a continue-or-stop decision to the sponsor is premature when a response was already planned and has not yet been tried (D).
