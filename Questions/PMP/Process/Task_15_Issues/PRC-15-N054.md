# Question

On a project that closes several issues a week, a developer marks a high-priority issue as resolved and closes it in the log. The next day the business user who originally reported it says the problem is still occurring. The project manager wants a change that prevents recurrence without materially slowing the team's throughput on routine items. Which change best fits that aim?

## Options

- **A**: Move closed issues to a "pending confirmation" state and require sign-off from the originator before final closure.
- **B**: Add a verification step on closure scaled to priority — originator or independent confirmation for high-priority items, resolver self-certification for low-priority ones.
- **C**: Have the project manager review and approve every issue closure during the weekly issue-log review.
- **D**: Require resolution notes to state the test performed and the evidence the fix worked before an issue can be closed.

## Answer

B

## Explanation

The give-away the question withholds is where the verification effort should land: the failure was on a high-priority item, but applying full independent confirmation to every closure would tax a team that closes several issues a week, so risk-proportionate verification — strong confirmation where impact is high, lighter where it is low — both fixes the gap and respects throughput. (A) routes every item, trivial or critical, back through the originator, which is heavier than the stated aim wants and can stall closure when originators are slow to respond. (C) makes the PM a bottleneck on all closures and adds a reviewer who often cannot personally confirm the fix worked. (D) improves the record and is good practice, but a well-written note still reflects only the resolver's own judgement that the fix worked — the same single point that just failed.
