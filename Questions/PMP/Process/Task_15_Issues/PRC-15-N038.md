# Question

During a weekly review on an offshore wind-farm cable project, the team flags that a forecast storm window in three weeks could prevent the survey vessel from sailing, which would delay a critical cable-lay activity. A team member wants to add this to the issue log so it gets attention. How should the project manager classify and handle it?

## Options

- **A**: Record it on the risk register as a threat, with probability, impact and a response, because the event has not yet occurred.
- **B**: Record it in the issue log with an owner and target date, because it concerns a critical activity that needs active tracking now.
- **C**: Raise a change request to re-sequence the survey ahead of the storm window, so the schedule no longer exposes the critical activity to it.
- **D**: Record it in both the issue log and the risk register, so neither the issue review nor the risk review can overlook it.

## Answer

A

## Explanation

A future, uncertain event that has not happened is a risk, so it belongs on the risk register with probability, impact and a planned response, regardless of how critical the activity is. (B) is tempting because the activity is critical and "needs tracking now", but the issue log is for problems that have already materialised; logging a not-yet-occurred event there mismanages it and skips the probability/impact analysis a risk needs. (C) jumps to authorising a specific response — re-sequencing — before the threat has even been assessed and weighed against alternatives like accepting it or planning a different mitigation; the response follows risk analysis, it does not replace it. (D) double-logs one item across two registers, creating duplicate tracking and confusion over which artefact governs the response. The discrimination is recognising the event as a future uncertainty and placing it where uncertainties are analysed, not where present problems or authorised changes are managed.
