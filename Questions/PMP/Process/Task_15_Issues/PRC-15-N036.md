# Question

A serious issue has just surfaced on a public-sector grants-management project: a data-validation defect has been allowing duplicate applications through, and some have already progressed to assessment. As project manager, which actions form part of properly managing this newly raised issue in its first hours, before any fix or schedule change is decided? (Select THREE.)

## Options

- **A**: Record the issue in the issue log with a clear description, an assigned owner and a target resolution date.
- **B**: Assess the impact and urgency to determine its priority relative to other open issues.
- **C**: Identify and inform the stakeholders affected by the duplicate applications already in assessment.
- **D**: Direct the team to deploy a fix to the validation defect today so no further duplicates can enter while the issue is being worked.
- **E**: Raise a change request to re-baseline the launch date, using the team's first estimate of the rework the fix will need.

## Answer

A, B, C

## Explanation

Managing a newly raised issue starts with recording it with an owner and target (A), triaging its impact and urgency to set priority (B), and engaging the stakeholders it already affects (C) — all of which can and should happen immediately. (D) is tempting because stopping further duplicates is urgent, but pushing an unverified fix straight to a live grants system ahead of impact assessment and the project's change control risks introducing worse defects; the fix is the response, decided after triage, not a first-hour reflex. (E) commits a baseline change to a re-based launch date off a first-pass estimate, before the impact is understood or the fix scoped — premature, and it short-circuits the impact analysis that should precede any re-baseline.
