# Question

Midway through a sprint, a Kanban-based platform team finds that four of its six in-progress stories are blocked waiting on a single API change owned by an external team that will not respond for three days. The two remaining stories are nearly done. The team has capacity sitting idle. What is the most effective action to keep value flowing?

## Options

- **A**: Finish and release the two unblocked stories, then pull the next-highest-priority unblocked items from the backlog while the dependency clears.
- **B**: Have the idle members start new stories from the backlog regardless of priority so no one is without work.
- **C**: Reassign the idle developers to help the external team make the API change faster, even though it is outside the team's remit.
- **D**: Pause the sprint and wait for the API change, using the idle time for backlog refinement and documentation.

## Answer

A

## Explanation

The aim is to maximise the flow of completed value, so finishing what is closest to done and then pulling the highest-priority unblocked work keeps throughput up without piling on more WIP (A). (B) starts work without regard to priority, which can increase work in progress and deliver lower-value items first. (C) may help but oversteps the team's boundary and assumes the external team welcomes it; it is a negotiation, not the immediate way to keep this team's value flowing. (D) halts delivery entirely when valuable unblocked work is available to be pulled.
