# Question

A hybrid project integrating a third-party payments gateway is ready to release a major feature, but the gateway vendor has just announced their sandbox will be unavailable for testing for the next ten days due to their own maintenance. The release was meant to ship this week to capture a seasonal sales peak. The team has fully built and unit-tested the feature against a local mock. What is the best course of action?

## Options

- **A**: Release the feature behind a feature toggle now, keep it switched off in production, and enable it for users once integration testing against the live gateway is possible.
- **B**: Release the feature fully enabled now, relying on the local mock testing, and fix any gateway integration defects reactively if they appear in production.
- **C**: Hold the entire release for ten days until the vendor sandbox returns, then test and ship after the seasonal peak has passed.
- **D**: Switch to a different payments vendor immediately so the feature can be tested and shipped within the seasonal window.

## Answer

A

## Explanation

A feature toggle lets the team ship the surrounding release now and decouple the risky, untested integration so it is only activated after proper testing, balancing urgency with safety (A). (B) enables an unverified payments integration in production, where defects directly damage revenue and trust during the peak. (C) sacrifices the entire seasonal opportunity for a dependency that affects only one feature's final testing. (D) is a disproportionate, high-risk switch of a core vendor under time pressure, introducing far more risk than the original problem.
