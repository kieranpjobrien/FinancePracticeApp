# Question

A project manager on a hospitality-booking platform is keeping value flowing under several different execution pressures arising in the same week. Match each situation to the response that best protects the urgent delivery of value.

## Items

- **1**: A critical-path configuration task is running late. The sponsor has approved spending to recover it, and the task owner says the work can be divided among more people without losing quality.
- **2**: A search-filter improvement has been built and has passed all testing, but it is being held back so it can launch alongside the rest of its epic in a month.
- **3**: The highest-priority story cannot proceed because a payments vendor has not delivered a sandbox, and chasing the vendor will take days. Two well-understood lower-priority stories are ready to begin.
- **4**: The sponsor insists on both a fixed scope and an earlier fixed date; the team's throughput data shows the two cannot both hold.

## Matches

- **A**: Release the completed item now and continue delivering the remaining scope afterward.
- **B**: Add the approved resource to the activity to compress its duration and recover the schedule.
- **C**: Surface the conflict with the supporting data and agree with the sponsor which constraint will move.
- **D**: Start the ready lower-priority work now while continuing to work the blocker.

## Answer

1-B, 2-A, 3-D, 4-C

## Explanation

Each situation calls for a different lever, and the tempting mismatches are what make it discriminating. Item 1 is a late but divisible critical-path task with funding approved, so compressing it by adding resource — crashing — is the recovery that fits (1-B); the trap is reading "running late" and reaching for the trade-off conversation (C) when a funded recovery exists. Item 2 is finished, tested value held back to batch, so releasing it now is correct (2-A); pairing it with "start other work" (D) would miss that the value is already done and waiting. Item 3 has the top item externally blocked with ready alternatives, so delivering the next-best value while working the blocker keeps flow alive (3-D); crashing (B) cannot help when the constraint is an external dependency, not internal capacity. Item 4 is a genuinely infeasible fixed-scope, fixed-date demand, which calls for an evidence-based trade-off about which constraint gives (4-C), not silent acceptance or simply releasing whatever is ready. Cross-wiring any pair either withholds done value, throws resource at a problem resource cannot solve, or commits the team to an impossible plan.
