# Question

A start-up's agile team is under pressure from investors to demonstrate market traction quickly. The team is building a food-delivery platform with a large planned feature set, but funding depends on showing real user activity within eight weeks. Which approaches genuinely help deliver demonstrable value with urgency? (Select TWO.)

## Options

- **A**: Define a minimum viable product covering the core order-and-deliver journey and release it to a limited real user group to generate live usage data.
- **B**: Slice the highest-value end-to-end workflow thin so a working, releasable version reaches users early rather than building each layer fully first.
- **C**: Build the complete feature set to a polished standard before any release so the first impression on investors is strong.
- **D**: Lengthen each sprint to four weeks so the team can deliver larger, more complete chunks before the funding review.
- **E**: Defer all releases until internal QA has signed off the entire platform, then launch everything at once just before the deadline.

## Answer

A, B

## Explanation

Early, demonstrable value comes from releasing a thin, working slice of the core journey to real users and gathering live evidence of traction (A and B). (C) and (E) both withhold value until everything is complete, which contradicts the eight-week urgency and produces no real usage data until the last moment. (D) lengthens feedback loops and delays the working software that would prove traction, moving in the wrong direction for urgency.
