# Question

A newly formed agile team is planning how to deliver a large customer-portal project. The portal's most-used function is account self-service; several other sections are lower-traffic. The sponsor wants value realised as early as is practical and is comfortable with customers seeing the portal evolve. The team is deciding how to sequence and release the work. Which approach best supports delivering value with urgency?

## Options

- **A**: Release a working version of the high-use self-service function to customers first, then add and release the remaining sections in priority order as they are completed.
- **B**: Build a thin version of every portal section so the whole portal launches at once as a complete, if basic, experience, then deepen each section in later releases.
- **C**: Release increments to an internal user group each iteration and make the first customer-facing release once the full portal is feature-complete.
- **D**: Sequence the build from the simplest sections to the most complex, releasing each as it is finished, so the team builds delivery momentum before tackling self-service.

## Answer

A

## Explanation

Delivering value early means putting the highest-value working slice in front of real customers first; releasing the most-used self-service function realises the most benefit soonest and starts genuine feedback (A). (B) launches a complete-but-shallow portal, which spreads early effort across low-traffic sections and delays a genuinely useful version of the function customers actually want most. (C) gathers only internal feedback and withholds all customer value until the whole portal is finished, defeating the urgency objective. (D) optimises for team momentum and ease rather than value, deferring the highest-value function behind low-traffic sections.
