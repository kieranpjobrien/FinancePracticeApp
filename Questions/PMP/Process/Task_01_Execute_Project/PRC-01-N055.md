# Question

An agile team is one day from the sprint review for a banking feature. The agreed definition of done requires a security scan, which has not yet run on two otherwise-finished stories. A developer suggests marking the two stories done for the demo and running the scan early next sprint. The product owner is keen to show progress to stakeholders. What should the scrum master do?

## Options

- **A**: Keep the definition of done intact so the two stories stay "in progress" until the scan passes, and have the team demo them at review as working-but-unverified rather than as done.
- **B**: Prioritise running the security scan on the two stories within the remaining day so there is a genuine chance they pass and can be legitimately called done for the review.
- **C**: Mark the two stories done for the review but record the outstanding scan as a tracked item carried into next sprint, keeping the status visible.
- **D**: Ask the product owner whether the security scan can be moved out of the definition of done for this release so the stories qualify as done on time.

## Answer

B

## Explanation

The definition of done is the team's quality contract and a security scan on banking code is non-negotiable, but the scrum master's best move is to remove the impediment to meeting it: focus the remaining day on actually running the scan so the stories can pass and be honestly demoed as done (B). (A) correctly refuses to weaken the gate but settles for the stories not being done when there is still time to make them genuinely done, a less effective outcome than B. (C) declares unscanned banking code complete; a tracking note does not change that the status is false and the security exposure is real. (D) invites the product owner to trade away a security gate under deadline pressure, precisely where it must hold firm.
