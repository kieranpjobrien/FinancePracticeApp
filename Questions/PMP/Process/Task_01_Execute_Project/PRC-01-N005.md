# Question

An agile team has a working increment of an insurance quote engine that passes all acceptance criteria; the product owner has accepted it and wants the value and feedback now. It could go to a small pilot group of brokers immediately. The increment writes quotes into the live policy database, and the rollback path if a defect surfaced in production has not yet been built or tested. A second sprint would add that rollback path plus minor convenience features. What should the project manager advocate?

## Options

- **A**: Release the accepted increment to the pilot brokers now to start generating value and feedback, and treat any production issues reactively as they arise.
- **B**: Release to the pilot brokers now but first confirm a tested rollback path exists, so an early production defect that corrupts live policy data can be reversed.
- **C**: Hold the release until the next sprint so the increment ships with the rollback path and the convenience features together as a more complete package.
- **D**: Release now to a read-only shadow mode where brokers see quotes but nothing is written to the live database, deferring the real release until the rollback path is ready.

## Answer

B

## Explanation

Delivering an accepted increment early is the value-maximising default, but this one writes irreversibly to live policy data with no tested way back, so the margin is releasing early *responsibly*. (B) keeps the early-value intent and adds only the safeguard the risk demands — a tested rollback before real data can be corrupted — which is the smallest precondition that makes "ship now" safe. (A) ships into an irreversible live write with no recovery path and only a reactive plan, which is reckless given the blast radius. (C) bundles the rollback path with optional convenience features and defers all value a whole sprint, conflating a genuine safety prerequisite with nice-to-haves and over-delaying. (D) is a reasonable risk reduction, but shadow mode delivers no real broker value — no bound quotes, no genuine feedback on the live flow — so it postpones the very value the product owner wants when a tested rollback would let the real release proceed now.
