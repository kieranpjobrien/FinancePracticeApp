# Question

An agritech start-up is building a crop-monitoring app and wants its first release to deliver real value to farmers as early as possible while the market window is open. The product owner is scoping that first release. Which TWO principles should most guide what goes into it for early value delivery? (Select TWO.)

## Options

- **A**: Include the smallest set of capabilities that lets a farmer achieve a genuine outcome, then expand in later releases.
- **B**: Order candidate capabilities by the value they deliver relative to the effort to build them, and take the strongest from the top.
- **C**: Include every capability the launch customers named in interviews, so the first release fully reflects what they asked for.
- **D**: Scope the first release to whatever the team can comfortably finish within one iteration, and let that capacity define the contents.
- **E**: Lead with the capabilities that best demonstrate the app's technical sophistication, to differentiate it from simpler competitors.

## Answer

A, B

## Explanation

Early value comes from shipping the smallest genuinely useful product and ordering work by value against effort. (A) is the minimum-viable principle: deliver a real outcome with the least scope, then grow. (B) sequences by value-to-effort so the strongest returns surface first. (C) is the most tempting distractor because honouring customer requests sounds customer-centric, but bundling everything interviewees named inflates the first release and delays the value the earliest, smallest outcome could already deliver — desirability is not the same as the leanest valuable slice. (D) letting one iteration's capacity define the contents is backwards: it lets effort, not value, choose scope, and can ship a tidy but low-value release while the capability farmers most need waits. (E) leading with technical sophistication optimises for showing off rather than for the outcome farmers value, spending the open market window on differentiation they may not care about.
