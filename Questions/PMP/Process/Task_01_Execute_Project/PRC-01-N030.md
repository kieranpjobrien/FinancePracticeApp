# Question

A predictive aerospace ground-support tooling project is on schedule. A lead engineer has been adding extra configurability to a module that has already passed its acceptance criteria, judging from direct contact with the customer's operators that they would find it genuinely useful in service. The work was not in the baseline and is now eating into the float ahead of the next critical-path activity. What should the project manager do first?

## Options

- **A**: Redirect the engineer to the scoped critical-path work now and raise the configurability as a change request so the customer can decide whether to fund and schedule it.
- **B**: Let the engineer finish the configurability, since it comes from real operator feedback and is the kind of in-service value the tooling exists to provide.
- **C**: Allow the work to continue but cap it to the available float, stopping the moment it would touch the critical path.
- **D**: Pause the engineer and ask the customer's project office informally whether they want the enhancement before deciding how to proceed.

## Answer

A

## Explanation

The module already meets its acceptance criteria, so further effort on it is unauthorised scope that is now consuming schedule the project needs, and the disciplined response is to protect the baseline while capturing the idea properly. (A) returns the engineer to committed work and routes the enhancement through change control, letting the accountable customer decide if it is worth funding and time. (B) is the most sympathetic reading because the feedback is real, but it lets the team unilaterally add scope and spend float on value no one has authorised. (C) sounds controlled, yet float is shared schedule reserve, so burning it on unrequested work erodes the project's ability to absorb genuine problems later. (D) at least seeks customer input but does so informally, which risks an unrecorded commitment and skips the change assessment that should accompany a baseline change.
