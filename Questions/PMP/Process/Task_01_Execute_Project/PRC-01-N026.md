# Question

A medical-device companion app team is under pressure to release a glucose-trends feature this sprint, with a competitor about to launch something similar. The feature works in the demo. Its definition of done has two outstanding gates: data-accuracy validation against the device, which is non-negotiable for a health feature, and a full accessibility audit. The product owner wants the fastest path to users. The team estimates the accuracy validation can be finished in three days, while the complete accessibility audit needs another two weeks. What should the project manager recommend?

## Options

- **A**: Complete the accuracy validation in this sprint and release the feature once it passes, scheduling the full accessibility audit as a committed fast-follow with interim accessibility checks documented.
- **B**: Hold the release until both the accuracy validation and the full accessibility audit are complete, keeping the entire definition of done intact before anything ships.
- **C**: Release after the accuracy validation passes and treat the outstanding accessibility audit as a known item to be picked up whenever capacity allows.
- **D**: Split the feature so the validated trend view ships now and the richer interactive views that drive most of the accessibility load are held until the audit is done.

## Answer

A

## Explanation

The accuracy gate is a genuine patient-safety bar and cannot be skipped, but the accessibility audit is a real obligation on a different timescale, so the judgement is how to ship safely and fast without either ignoring accessibility or letting it block a validated feature for two weeks. (A) clears the safety-critical gate first, then releases with accessibility handled as a committed, dated fast-follow plus documented interim checks, which keeps urgency and accountability together. (B) is safe but treats a two-week audit as an absolute blocker on a feature that is already accuracy-validated, conceding the market for little added user protection. (C) ships after accuracy too, but downgrading accessibility to "whenever capacity allows" quietly abandons a real obligation under deadline pressure. (D) is a plausible slice, yet carving the feature to dodge the audit risks shipping a degraded trend view and still leaves the deferred parts un-audited, so it trades user value for the same unfinished work.
