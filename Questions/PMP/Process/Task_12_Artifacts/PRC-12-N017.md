# Question

A project manager is closing a completed manufacturing-line installation and wants to handle the project's artifacts responsibly at closure. Which two actions are appropriate when finalising and archiving project artifacts? (Choose two.)

## Options

- **A**: Ensure final versions of key artifacts, such as the lessons-learned report and as-built records, are completed and stored in the organisational records repository.
- **B**: Apply the organisation's retention schedule so artifacts are kept for the required period and disposed of appropriately thereafter.
- **C**: Retain every working draft, email, and superseded version indefinitely alongside the finals, so nothing is ever lost.
- **D**: Transfer the project records to the operations team's own drive and remove them from the project repository, since the project is ending.
- **E**: Strip metadata and version histories from archived documents so the archive contains only clean final files.

## Answer

A, B

## Explanation

Closing artifacts means finalising and storing the authoritative versions in a governed repository (A) and applying the retention schedule that dictates how long they are kept and when they may be disposed of (B). (C) over-retention is the opposite failure to deletion but still wrong: keeping every draft and email indefinitely defies the retention schedule, bloats the archive, and buries the authoritative finals among noise. (D) moving records to operations' local drive and clearing the project repository looks like a clean handover but pulls them out of governed, retained storage into an uncontrolled location — the finals should be archived in the records repository, with operations given access, not custody of the only copy. (E) removing version histories and metadata strips the traceability that makes an archive trustworthy and auditable.
