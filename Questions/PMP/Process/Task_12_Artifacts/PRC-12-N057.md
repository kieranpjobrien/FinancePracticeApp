# Question

A hybrid team treats a user story as complete once its code is merged. The supporting API documentation and test evidence are meant to follow "when there is time" and routinely never do, so downstream teams and auditors keep finding stories marked done with no artifacts behind them. The team is willing to fix this but does not want documentation to become a separate, deferrable chore again. What should the project manager facilitate?

## Options

- **A**: Add the required API documentation and test evidence to the team's definition of done so a story is not complete until they exist.
- **B**: Make producing the documentation and test evidence an acceptance criterion on each story that needs them, so it is part of the story's own scope.
- **C**: Add a documentation and test-evidence column to the board so a story cannot move to done until that work is visibly finished.
- **D**: Build a documentation step into the team's pull-request checklist so a merge cannot complete without the artifacts attached.

## Answer

A

## Explanation

The problem is general — it applies to every story that needs these artifacts — and the definition of done is the team-wide standard that makes "done" mean the artifacts exist, applied uniformly without re-litigating each story; that makes (A) the right facilitation. (B) per-story acceptance criteria work, but relying on remembering to add them to each story leaves gaps when someone forgets, whereas the DoD applies by default. (C) a board column enforces the same idea but is a workflow mechanic that can be marked complete superficially; it implements a DoD rather than being the standard itself. (D) a PR checklist anchors the gate at merge — but merge is exactly the point the team currently treats as "done", and a checklist item is easily ticked without the artifact genuinely existing, so it is weaker than making it part of the definition of done.
