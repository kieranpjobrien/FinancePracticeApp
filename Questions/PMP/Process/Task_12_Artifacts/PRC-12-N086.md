# Question

A project manager joins a marketing-platform project and finds the team's documents spread across personal drives, email attachments, and three chat channels. People routinely cannot find the latest version of a plan and sometimes work from a copy that turns out to be old. The PM has a short window to make a change that holds. Which step would most effectively improve how the project's artifacts are managed?

## Options

- **A**: Stand up one agreed repository as the controlled home for project artifacts and migrate the current documents into it.
- **B**: Issue a naming-and-versioning convention so every file records its version and status in the filename, wherever it lives.
- **C**: Turn on version history and check-in/check-out on the existing shared drive so the latest copy is always identifiable.
- **D**: Appoint a document controller responsible for collecting the current versions and circulating an authoritative master set.

## Answer

A

## Explanation

Every option targets the same pain — people cannot reliably find or trust the current version — but they differ on whether they fix the cause or a symptom. The cause is that there is no single controlled location where the authoritative copy lives; the artifacts are dispersed across drives, email, and chat. Establishing one repository and consolidating into it (A) removes that dispersion and is the precondition the other measures depend on. (B) a naming convention adds value, but applied across scattered locations it produces well-named copies in many places rather than one findable source. (C) version history is powerful, yet bolting it onto only the shared drive leaves everything still living in email and chat untouched, so the latest copy can still be elsewhere. (D) a document controller can help, but funnelling versions through one person recreates a manual, instantly stale master set unless there is first a single place those versions live. The discriminator is consolidation versus improving the handling of an already-fragmented set.
