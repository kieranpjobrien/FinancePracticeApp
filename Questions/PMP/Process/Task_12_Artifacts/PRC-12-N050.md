# Question

A hybrid product team keeps having incidents where stakeholders act on outdated information: a vendor quoted against a superseded requirements list, and a developer coded from an old wireframe. In both cases the current version existed in the repository, but the stakeholder had a copy obtained earlier and never realised a newer one had been issued. The project manager wants to stop people working from stale artifacts. Which two measures most directly address the cause? (Choose two.)

## Options

- **A**: Distribute artifacts by link to the controlled location rather than by attachment, so recipients always open the live current version.
- **B**: Apply a versioning scheme and archive superseded versions so an out-of-date copy is clearly identifiable as not current.
- **C**: Tighten change control so requirements and wireframes are revised less often, reducing how quickly any held copy goes stale.
- **D**: Have the team notify all stakeholders by email whenever an artifact is updated, so holders know to fetch the new version.
- **E**: Require each artifact to carry a visible "current as of" date and version on its first page.

## Answer

A, B

## Explanation

The incidents arose because people held detached copies and could not tell a newer version had superseded theirs. Distributing by link to the controlled location removes the detached copy entirely (A), and a versioning-plus-archiving scheme makes any superseded copy recognisably not current (B); together they attack the cause directly. (E) is a sound supporting practice but only helps if the holder happens to re-check the document and notice the date — it does not stop them acting on the copy in hand. (D) pushes notifications that are easily missed or buried, recreating the same "didn't realise it changed" failure. (C) slows the rate of change but does nothing to prevent the wrong version being used and merely makes the artifacts less responsive.
