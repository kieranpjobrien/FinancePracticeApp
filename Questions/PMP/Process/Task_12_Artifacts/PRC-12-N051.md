# Question

During a defence-contract audit the auditor asks the project manager to show, for one delivered component, exactly which requirement version, design revision, and test record applied at the moment it was released to production. The repository has full version history for each document individually and a complete change log, and the PM can show the latest version of each. But the PM cannot demonstrate which versions were the matched, mutually consistent set at the production-release point. What was missing from the project's artifact management?

## Options

- **A**: Baselining that captures a labelled, point-in-time snapshot of the related artifacts together as one approved configuration.
- **B**: A configuration status accounting report listing the current version of every controlled artifact.
- **C**: An integrated change control process linking each approved change to the artifacts it affected.
- **D**: A traceability matrix linking the requirement to its design revision and test record.

## Answer

A

## Explanation

The auditor needs the *coherent set* of versions as they stood at one defined moment — a production baseline — and that is exactly what baselining provides: a labelled snapshot of related artifacts approved together. Individual histories and change logs (which the PM already has) cannot reconstruct that matched state, so (A) is the gap. (B) status accounting reports the *current* versions, the very thing the PM could already show and not what reconstructs a past matched set. (C) integrated change control was evidently working — there is a complete change log — yet tracking changes individually still does not freeze a coherent multi-artifact baseline. (D) a traceability matrix shows which items *relate*, but unless it is itself baselined it reflects current links, not the version set frozen at release.
