# Question

A globally distributed engineering team works across time zones on shared design artifacts in a common repository. Two engineers edited the same drawing within the same hour and one save silently overwrote the other's changes, which were noticed missing only days later. The project manager must harden how shared artifacts are managed so concurrent work cannot silently lose changes. Which two measures most directly prevent this? (Choose two.)

## Options

- **A**: Configure the repository so a save based on an out-of-date copy is rejected, forcing the editor to reconcile against the current version before committing.
- **B**: Enable a check-out/lock mechanism so a file being edited cannot be edited concurrently by a second person until released.
- **C**: Turn on per-change version history so every edit is retained as a distinct revision and an overwritten change can be recovered afterward.
- **D**: Require an automated notification to the team whenever a shared artifact is opened for editing, so others know to wait.
- **E**: Establish a working agreement that engineers announce in the team channel which drawings they are about to edit.

## Answer

A, B

## Explanation

Both selected measures *prevent* the silent overwrite at the moment it would occur: rejecting a stale-based save forces reconciliation before any commit (A), and a check-out/lock stops a second concurrent edit from happening at all (B). (C) per-change history is valuable and enables *recovery*, but it does not stop the overwrite — the team here only noticed days later, which is exactly the after-the-fact recovery that still costs lost work. (D) and (E) are coordination signals that depend on people seeing and heeding them in time; across time zones two engineers can still edit within the same hour despite a notification or a channel post, so neither reliably prevents the clash.
