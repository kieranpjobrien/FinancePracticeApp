# Question

A project manager is setting up configuration management for the artifacts on a large infrastructure programme so that the team always works from correct, current documents. Which of the following belong in the configuration *management* approach itself, as opposed to a related but separate discipline? (Choose three.)

## Options

- **A**: A controlled repository where each artifact has a unique identifier and a version history.
- **B**: A defined process for how changes to controlled artifacts are requested, reviewed, and approved before a new version is released.
- **C**: Clear marking of superseded versions and a single identified location for the current approved version.
- **D**: A role-based access scheme deciding which stakeholder groups may read or edit which artifacts.
- **E**: A communications plan setting how often each stakeholder group is sent updates on artifact changes.

## Answer

A, B, C

## Explanation

Configuration management is about identifying and versioning each artifact (A), controlling how a change moves it to a new approved version (B), and making clear which version is current while superseded ones are marked (C) — identification, change control, and status accounting. (D) access control is a legitimate and often co-located practice, but it governs *who may see or edit* an artifact, not which version is authoritative; it is security/permissions, not configuration management, and a repository can be perfectly configuration-controlled with wide-open read access. (E) how often stakeholders are *notified* of changes is communications management; it rides on top of configuration management but is not part of keeping the artifact set controlled and current. The trap is that all five are sensible repository concerns — only three are configuration management proper.
