# Question

On a predictive bridge-replacement project, an approved change request adds a pedestrian walkway, increasing scope, cost, and the schedule. The change board has signed it off. The project manager updates the schedule in the planning tool but the team continues to report progress against the original cost and scope baselines. Variance reports now look misleadingly favourable. What should the project manager do?

## Options

- **A**: Add a footnote to the variance report explaining that the walkway change is not yet reflected in the baselines.
- **B**: Update all affected baselines (scope, cost, and schedule) to incorporate the approved change so performance is measured against the correct reference.
- **C**: Keep reporting against the original baselines until the walkway work physically begins, then switch.
- **D**: Track the walkway separately as a sub-project so the original baselines remain untouched for the rest of the bridge.

## Answer

B

## Explanation

Once a change is approved, the relevant baselines must be updated so earned-value and variance reporting compares actuals against an authorised plan; partial updates corrupt the measurement. (A) a footnote does not fix the numbers and the report still understates true performance against the agreed scope. (C) delaying the baseline update produces wrong variances in the interim and ignores that the change is already authorised. (D) splitting the walkway into a separate sub-project is an arbitrary workaround that fragments integrated control and is not how an approved change to this project's scope should be handled.
