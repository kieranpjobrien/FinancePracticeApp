# Question

Two squads contributing to the same product each keep their own copy of the definition of done in separate wiki pages. The pages have drifted apart, and an increment that one squad considered "done" was rejected at the release review because it failed the other squad's stricter quality bar. The release train engineer wants to prevent this clash recurring. What is the best action?

## Options

- **A**: Direct each squad to adopt the stricter definition of done so quality is never the lower of the two bars.
- **B**: Facilitate the squads in agreeing one shared definition of done, hold it as a single controlled artifact, and have both squads reference it.
- **C**: Add a release-level gate that re-checks every increment against a master checklist before it ships.
- **D**: Ask the product owner to specify the definition of done so both squads receive an authoritative standard.

## Answer

B

## Explanation

The failure stems from a duplicated, divergent artifact, so consolidating to a single agreed definition of done that both squads own and reference removes the drift at its source. (A) imposing the stricter bar by fiat may be impractical or unbalanced and, more importantly, still leaves two maintained copies free to diverge again. (C) a release gate catches the symptom late and adds rework without fixing the conflicting source artifacts. (D) the definition of done is a team-owned quality agreement, not the product owner's to dictate, and a top-down mandate undermines the teams' ownership of quality.
