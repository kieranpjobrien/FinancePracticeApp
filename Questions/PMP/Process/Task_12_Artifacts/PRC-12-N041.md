# Question

A hybrid software team keeps the requirements traceability matrix on a shared drive while developers track the requirements they actually build inside their issue tracker. At a sprint review the product owner finds a delivered feature that contradicts the matrix; the developers say the tracker is more current and they coded from it. The product owner and the sponsor are both in the room and want to know how this will be prevented. What should the project manager do first?

## Options

- **A**: Walk the two records side by side to find every requirement where they disagree, then correct each so both views match before deciding anything about process.
- **B**: Agree with the team and stakeholders which record is authoritative for requirements and how any second view will be kept reconciled to it.
- **C**: Have the business analyst confirm, for this feature only, which record reflects the customer's actual intent so the disputed feature can be resolved.
- **D**: Add a step to the change-control procedure requiring the matrix and the tracker to be updated together whenever a requirement changes.

## Answer

B

## Explanation

The recurring cause is that no record has been agreed as authoritative, so the first move is to establish which one governs and how the other is reconciled to it; everything else follows from that decision. (A) reconciling every entry now treats the symptom and, without an agreed authoritative source, the two records will simply diverge again. (C) settling this one feature resolves the immediate dispute but leaves the structural ambiguity that produced it. (D) mandating dual updates is a reasonable control but presumes both records are retained as peers; it should be defined *after* deciding which is authoritative, or it just keeps two competing sources in lockstep until they drift.
