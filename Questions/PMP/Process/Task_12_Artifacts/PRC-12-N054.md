# Question

On a predictive project, every time a document is revised, team members save a new file with the date appended, so each folder holds dozens of near-identical files and stakeholders sometimes open a superseded one. The organisation provides a shared drive and, separately, a document-management system with check-in/check-out and retained version history; adoption of the latter has been patchy and most members default to the shared drive. The project manager wants to stop superseded files being mistaken for the current one. What is the most effective improvement?

## Options

- **A**: Move the documents into the document-management system and have the team work through check-in/check-out so each file shows one current version with history behind it.
- **B**: Define a strict file-naming and versioning convention on the shared drive, with a maintained index page listing the current filename for each document and a nominated owner who keeps it accurate.
- **C**: Keep a single "current" copy of each document in a dedicated folder and move all dated revisions into an "archive" subfolder.
- **D**: Configure the shared drive to retain previous versions of a file automatically so an overwritten state can be recovered if the wrong one is used.

## Answer

A

## Explanation

A system that enforces one current version with history through check-in/check-out removes the proliferation at its source, so directing the team onto it rather than the shared drive is the most effective fix — (A). (B) a naming convention plus a maintained index and a named owner is a credible alternative and may be all a small project needs, but it still depends on every person following the convention perfectly and on the owner keeping the index current — the same manual discipline that is already failing here. (C) separating current from archived files helps today but, on a shared drive with no control, new dated copies accumulate again and the "current" folder can hold more than one. (D) automatic version retention aids recovery *after* someone uses the wrong file but does nothing to present a single unambiguous current version up front.
