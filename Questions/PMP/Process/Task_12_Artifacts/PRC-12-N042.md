# Question

On a regulated medical-device project a tester runs a validation protocol against design document Rev D and files a passing result. Rev E, released two days earlier, tightened the very tolerance the protocol checks. An internal auditor flags that the test ran against a superseded revision; the configuration management plan requires testing against the current released revision. The tester says the repository still showed Rev D as current when the test began. What should the project manager do?

## Options

- **A**: Re-run the protocol against Rev E and, regardless of that outcome, investigate why the repository presented a superseded revision as current so the gap is closed.
- **B**: Re-run the protocol against Rev E; if it also passes, treat the matter as closed since the product is shown to meet the current tolerance.
- **C**: Raise the mismatch as a non-conformance, quarantine the Rev D result, and have the change board confirm whether Rev E's tolerance change actually invalidates the earlier test before committing to re-test.
- **D**: Re-run the protocol against Rev E and update the configuration management plan to require testers to verify the released revision before each run.

## Answer

A

## Explanation

Two things are wrong: an invalid test result exists, and the repository misrepresented the current revision — and the second is the more serious, because if it can mislead one tester it can mislead anyone. Re-testing against Rev E *and* fixing the source of the bad revision pointer addresses both, which is why (A) is best. (B) re-tests but stops at a passing result and never asks why a superseded revision looked current, leaving the systemic fault live. (C) is sound governance and may well run in parallel, but deferring the re-test to a board determination leaves an unverified product against the current revision longer than necessary and still omits the repository fix. (D) re-tests and adds a control, but loading verification onto every tester papers over a repository that is itself presenting the wrong revision.
