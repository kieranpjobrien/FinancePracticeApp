# Question

Three sprints into a hybrid regulatory-reporting project, a regulator questions whether a delivered calculation still satisfies the control it was meant to implement. The product owner can show the calculation in the current build and the backlog item that requested it, but cannot show that the item traces to the specific regulatory requirement, nor which test demonstrated compliance. The project manager wants to close this gap going forward without abandoning the team's lightweight agile artifacts. Which artifact practice should the project manager strengthen first?

## Options

- **A**: Maintain traceability linking each governing requirement to the backlog item that delivers it and the test that demonstrates it.
- **B**: Expand the product backlog so each item records the full regulatory wording it derives from.
- **C**: Keep a decision log capturing why each calculation was implemented the way it was.
- **D**: Tighten the definition of done so no story closes without its acceptance evidence attached.

## Answer

A

## Explanation

The regulator's question is specifically about *linkage* — does this delivered item connect back to the requirement it must satisfy, and to the test that proves it — and only traceability across requirement, backlog item, and test answers that end to end, so (A) is the first thing to strengthen. (D) attaching acceptance evidence at done is valuable and closely related, but evidence on a story still does not establish that the story implements a particular governing requirement; the missing thread is the requirement-to-item link, not just the test artifact. (C) a decision log explains *why* a choice was made but does not demonstrate the item satisfies the mandated control, which is what is being challenged. (B) embedding regulatory wording in each backlog item duplicates the requirement into many places where copies drift, and still provides no link to the test that demonstrates compliance — traceability, not transcription, is what is needed.
