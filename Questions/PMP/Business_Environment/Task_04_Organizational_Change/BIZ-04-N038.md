# Question

An airline is introducing a new crew-rostering app to cabin crew, releasing it to one base at a time over several months, with each base going live roughly a fortnight after the last. The project manager wants to learn from each base in time to adjust how the next base is prepared and supported. Several monitoring options are on the table. Which will best serve that goal?

## Options

- **A**: After each base goes live, gather both adoption data and crew sentiment from that base and review it before the next base's preparation is finalised, so each rollout informs the one that follows.
- **B**: Stand up a live dashboard tracking app usage across all bases continuously, so the team can see at any moment where uptake is lagging.
- **C**: Run deep, structured interviews with a sample of crew at the first base only, on the assumption that the issues found there will recur and can be pre-empted everywhere else.
- **D**: Hold a single comprehensive review once every base is live, combining all the data and feedback into a thorough account of how the rollout went.

## Answer

A

## Explanation

The goal is to adjust each base's preparation using what the previous base revealed, and the fortnightly cadence creates exactly the window to do it; capturing both behaviour (adoption data) and experience (sentiment) per base and feeding it forward before the next base is locked is what closes that loop (A). A continuous cross-base dashboard (B) shows usage but not why crews are or are not adopting, and tracking numbers in real time is not the same as turning each base's lessons into changes for the next. Interviewing only the first base (C) assumes later bases are identical and stops learning after the start, so it misses how issues differ as the rollout moves between bases of different size and culture. A single end-of-rollout review (D) produces good learning too late to help any base but the last, which defeats the purpose of adjusting as you go.
