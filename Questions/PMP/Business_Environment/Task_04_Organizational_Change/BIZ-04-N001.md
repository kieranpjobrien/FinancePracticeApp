# Question

A hospital network is replacing paper-based theatre scheduling with a digital system across eleven surgical units. The build is nearly complete and a single network-wide go-live is set to align with a board commitment. The project manager has a training curriculum and a communications calendar ready to run identically at every unit. Anecdotally, some units are keen and others are stretched—three of them merged last year and are still settling new staff and workflows—but the project manager has nothing concrete on how each unit's adoption capacity actually differs. What should the project manager do next?

## Options

- **A**: Assess each unit's readiness—capacity, willingness, and capability to adopt—and use the findings to tailor training, sequencing, and support before committing to a rollout approach.
- **B**: Run the prepared training and communications uniformly but start with a pilot at one merged unit, then adjust the network rollout based on what the pilot reveals.
- **C**: Stand up a network of unit-level change champions to surface local concerns and carry the change peer-to-peer, then proceed with the planned go-live.
- **D**: Convert the single network-wide go-live into a phased rollout, scheduling the settled units first and the merged units last to give them more preparation time.

## Answer

A

## Explanation

Every option here is a legitimate change-management move, so the discriminator is what the project manager actually knows. The readiness across units is only *anecdotal*—there is no concrete picture of where capacity, willingness or capability differ—and supporting organisational change begins with diagnosing that current state before selecting an intervention (A). Piloting at one merged unit (B) is sound, but it commits to a specific tactic and a specific unit on assumption; a readiness assessment may show the real gaps lie elsewhere, and the pilot is something the findings would inform, not precede. Standing up change champions (C) is a strong adoption lever and may well follow, but deploying it network-wide before knowing each unit's situation spends effort uniformly on a problem the project manager has just acknowledged is *not* uniform. Phasing by an assumed settled-versus-merged split (D) bakes in a readiness ranking that has not been verified—a "settled" unit may be more resistant than a merged one—so it pre-sequences on a guess. The margin is that B, C and D each pick a reasonable response, but all three act on an unverified read of differential readiness, whereas A establishes that read first so the chosen response fits where the gaps truly are.
