# Question

An insurer is replacing its claims-handling system using an iterative delivery approach, releasing capability to one claims team every few weeks across about nine months. Each release changes how the receiving team works, though all teams will eventually use the same core workflow. Standing up a training session and tailored communications for each release is real, repeated effort, and the change lead is concerned about both cost and overloading the change team. The project manager is deciding how to structure the change support. What is the soundest approach?

## Options

- **A**: Deliver just-in-time training and team-specific communications to each team as its release lands, accepting the repeated effort as the price of teams being ready when their work actually changes.
- **B**: Run one organisation-wide training and communications event near the end, once the workflow has stabilised, so the effort is spent once on final, settled content rather than repeatedly on changing material.
- **C**: Hold a single thorough event at the start covering the full target workflow, then rely on short release notes for each team as its capability ships, minimising repeated formal sessions.
- **D**: Build one reusable training package on the common core workflow delivered just before each team's release, and wrap it with a brief team-specific briefing on what differs for them.

## Answer

D

## Explanation

The trade-off is real: per-release effort is costly, but training people long before or long after their work changes wastes it. The strongest structure resolves both — build the common-core training once and reuse it, timing each delivery to the team's release so it lands when needed, and add only a light team-specific layer for what differs (D). This captures most of the efficiency the change lead wants without sacrificing timeliness. Fully bespoke just-in-time effort for every team (A) is correctly timed but ignores that the shared core lets much of the material be reused, so it does more work than necessary. A single end-of-project event (B) spends the effort once but leaves every earlier team unsupported for months and trains them well after they needed it. A single up-front event plus release notes (C) front-loads a workflow people cannot yet apply and will have largely forgotten, and thin release notes are weak support at the moment a team's work actually changes.
