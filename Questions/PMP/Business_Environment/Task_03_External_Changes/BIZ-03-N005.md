# Question

Your team is building a paid developer tool on an agile basis. A major competitor open-sources a product that duplicates roughly half of your planned premium features and is gaining rapid community adoption. The product owner agrees the team should pivot toward the features that remain genuinely differentiated and is reworking the backlog to do so. Half-built work on the now-commoditised features sits in the current sprint, and a customer release is communicated for the end of the month. What is the most useful thing the project manager can do to support the pivot right now?

## Options

- **A**: Help the product owner pull together usage and competitive data on which planned features are still differentiated, so the reworked backlog reflects where the tool actually still wins.
- **B**: Facilitate a session for the team to decide the fate of the partly built commoditised work — finish, park, or discard — and recover that capacity for the new direction.
- **C**: Reforecast the release plan and prepare the stakeholder message so the communicated date and scope stay credible as the pivot lands.
- **D**: Set up a lightweight way to track adoption of the competitor's open-source product, so the team can see how fast the affected features are actually commoditising.

## Answer

A

## Explanation

All four are legitimate ways to support a pivot, so the question is which most directly shapes the decision the whole pivot now rests on: which features remain differentiated. Grounding the reworked backlog in usage and competitive evidence does exactly that, and everything else keys off the answer (A). Deciding the fate of the half-built work (B) is worth doing and frees capacity, but it is premature until the analysis says whether those features survive the new strategy — the team could spend a session disposing of work the data would have kept. Reforecasting the release and message (C) is necessary, but it can only be done credibly *after* the backlog firms up, so it follows the pivot rather than enabling it. Tracking competitor adoption (D) is a useful signal, yet it informs *how fast* to move more than *what to build*, and the differentiation analysis in A is the input the reworked backlog cannot be set without. The margin is A versus B: both act on the work in front of the team, but only A produces the evidence the re-prioritisation depends on, whereas B presumes that prioritisation is already settled.
