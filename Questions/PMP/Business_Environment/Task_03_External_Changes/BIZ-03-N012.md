# Question

Midway through building a web application, a widely adopted open-source framework releases a major version that would noticeably simplify several upcoming backlog items but requires migration effort the team cannot yet size with confidence. The developers are enthusiastic; the product owner is open to it but focused on committed feature delivery. The team raises it in the retrospective. As the change is debated, what is the most useful contribution the project manager can make?

## Options

- **A**: Suggest the product owner add a migration item to the backlog and rank it against other work at the next refinement.
- **B**: Suggest the team timebox a spike to size the migration effort and confirm the benefits, so the product owner can prioritise it against feature work with real information.
- **C**: Suggest the team adopt the new version in the upcoming sprint while enthusiasm is high, before momentum is lost.
- **D**: Suggest holding off until the framework version has proven stable in the wider community, to avoid migrating onto an immature release.

## Answer

B

## Explanation

The blocker to a good decision is that the effort and benefit are not yet known, so the most useful move is a timeboxed spike to produce that information for the product owner to prioritise against feature work (B). Simply adding a backlog item to rank later (A) asks the product owner to prioritise something still unsized, so it is weaker than first reducing the uncertainty. Adopting next sprint on enthusiasm (C) commits to migration before its cost is understood and could derail committed delivery. Waiting for the release to mature (D) is a legitimate consideration, but as a blanket deferral it pre-empts the assessment rather than informing it, and the spike is where that maturity concern would actually be weighed.
