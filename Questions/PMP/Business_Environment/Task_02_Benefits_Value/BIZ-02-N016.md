# Question

A product owner is ordering a backlog to maximise the value delivered within a fixed quarterly budget. Several competing items each have merit, and the product owner wants the ordering to reflect value rather than convenience. Which considerations legitimately inform a value-based ordering of the backlog? (Select all that apply.)

## Options

- **A**: The cost of delay associated with each item if its delivery slips beyond the quarter.
- **B**: The size of the benefit each item is expected to produce relative to its effort.
- **C**: Dependencies that mean one item must precede another to unlock its benefit.
- **D**: The story-point estimate of each item taken on its own, ordering the smallest first to maximise the count of items completed.
- **E**: The order in which stakeholders submitted the requests, honouring a first-in-first-out queue.

## Answer

A, B, C

## Explanation

Cost of delay (A), benefit-to-effort ratio (B), and enabling dependencies (C) are all genuine inputs to sequencing work for maximum value. Ordering by raw size to finish the most items (D) optimises throughput, not value — clearing many low-benefit items can deliver less worth than completing one high-benefit one, so size matters only *relative* to benefit, as in B. Honouring submission order (E) is a fairness convention that treats every request as equally valuable; it ignores benefit, urgency, and dependency, which is exactly what a value-based ordering must weigh. The trap in D and E is that both are real ordering heuristics teams use, but each substitutes a proxy (item count, queue position) for the value the product owner is trying to maximise.
