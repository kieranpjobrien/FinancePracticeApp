# Question

An agile team building a subscription analytics platform for an insurer has a strong product backlog and clear sprint goals, but the project manager realises the team cannot articulate how the work connects to the organisation's expected returns. The PM decides to introduce a lightweight benefits management plan. Which elements are essential for that plan to actually guide benefit realisation rather than become shelfware? (Select all that apply.)

## Options

- **A**: A description of each target benefit with its metric and the baseline it will be measured against.
- **B**: The named individual or function accountable for realising each benefit after the product is in use.
- **C**: A detailed sprint-by-sprint schedule mapping every backlog item to a delivery date.
- **D**: The timeframe over which each benefit is expected to materialise and when it will be assessed.
- **E**: The full technical architecture and data model underpinning the platform.

## Answer

A, B, D

## Explanation

A benefits management plan exists to make value measurable and owned: it needs the benefits defined with metrics and baselines (A), an accountable owner for realisation that outlasts the project (B), and a timeframe for when each benefit is expected and assessed (D) — without these three a benefit cannot be tracked or claimed. C is a delivery artefact; a fixed item-by-item schedule both contradicts the team's agile approach and describes how work is done, not what value results. E is solution documentation that belongs in technical artefacts, not in a plan about outcomes — its presence is exactly the kind of detail that turns a benefits plan into an ignored document.
