# Question

An insurer's agile team is rebuilding its claims-intake portal. The full rebuild will take about eight months. The team has organised the work so that, every two months, a self-contained slice could be released to production and begin delivering value — for example, the first slice would let customers upload documents online, removing a manual postal step. The programme sponsor, worried about change fatigue among adjusters, proposes a single big-bang release at the end instead. From a benefits-delivery standpoint, what should the PM recommend?

## Options

- **A**: Release each two-month slice as it is ready, so value is realised earlier and progressively rather than withheld until month eight.
- **B**: Adopt the single end-of-programme release, since one coordinated change minimises disruption to adjusters and simplifies training.
- **C**: Release the first slice early to prove the approach, then switch to a big-bang release for the remainder to limit change events.
- **D**: Delay all releases until at least half the portal is complete, so each release carries enough functionality to be worth the disruption.

## Answer

A

## Explanation

When work is genuinely decomposed into independently valuable slices, incremental release realises benefit earlier and starts the value clock sooner — the first slice alone removes a manual postal step from month two rather than month eight (A). B forfeits up to eight months of accruable value to avoid change disruption that can be managed through adjuster communication and training rather than by withholding working functionality. C captures only the first slice's early value and then reverts to deferring the rest, giving up most of the incremental benefit for the same change-fatigue reason that proper change management addresses. D arbitrarily withholds ready, valuable increments on a "big enough to bother" rule, delaying realised benefit for slices that already pay for themselves. The fine margin is A versus C: both ship something early, but only A sustains incremental value delivery across the whole programme instead of surrendering it after the first release.
