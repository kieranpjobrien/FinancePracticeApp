# Question

An agile team is building a citizen-facing benefits portal for a government agency. Accessibility to a published national standard is mandatory at launch. The team currently runs an accessibility audit only on the full release candidate, two days before go-live. The last release surfaced eleven accessibility defects at that audit, forcing a delayed launch. The product owner asks the project manager how to reduce the chance of this recurring. What is the most effective change?

## Options

- **A**: Add a dedicated accessibility-remediation sprint immediately before each release so defects found at the final audit have time to be fixed.
- **B**: Build the accessibility acceptance criteria into the team's definition of done so each story is verified for them before it is accepted.
- **C**: Hire a specialist accessibility tester to run the release-candidate audit earlier and more thoroughly.
- **D**: Raise a risk in the register for accessibility defects and assign the product owner as risk owner.

## Answer

B

## Explanation

The defects recur because compliance is verified only once, at the end, when remediation is most expensive and schedule-critical; folding the accessibility criteria into the definition of done moves verification to each story and stops defects accumulating to the release candidate (B). A remediation sprint (A) institutionalises late discovery and buffers it rather than preventing it, accepting the very batching that causes the problem. A specialist running the same end-of-cycle audit (C) finds defects slightly earlier but still after the work is done, leaving the feedback loop long. Logging a risk and naming an owner (D) records the concern without changing the process that produces the defects. The fine line is A versus B: both respond to the audit failures, but only B addresses when compliance is checked rather than padding time to fix what late checking reveals.
