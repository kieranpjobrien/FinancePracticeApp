# Question

A bank's mobile-app replacement project is being estimated. The sponsor pushes the project manager to commit to a fixed delivery date before the product breakdown structure is complete, arguing the activity list can be 'filled in later'. Why should the project manager resist committing to the schedule at this point?

## Options

- **A**: Because schedules can never be produced until the project is finished
- **B**: Because only the project board may set delivery dates, never the project manager
- **C**: Because product-based planning forbids any estimating before the closing stage
- **D**: Because, without the full set of defined products, the scope is not yet known and any estimate of time would be unreliable

## Answer

D

## Explanation

Scope is defined by the products to be delivered, so until the product breakdown structure is complete the full scope is unknown and any time estimate rests on an incomplete picture. (A) Schedules are produced during planning, not only at the end. (B) Setting tolerances such as time involves the board, but the issue here is the missing product definition, not who owns dates. (C) Product-based planning does not ban early estimating; it requires products to be defined first so estimates are sound.
